Cryptographic receipts · live

Every run, a signed proof.

Every POST /v1/run returns a receipt: a stable JSON object that names the recipe by its source hash, fingerprints the input and output, stamps the runtime version and time, and signs the bundle with HMAC-SHA256. Anyone can verify it. Offline. Years from now. Without us.

What it proves

Behavior provenance

Which exact recipe ran, on which exact input, produced which exact output, on which runtime, when. Not "approximately" — bit-exact, hash-pinned.

Why an LLM can't

Stochastic by design

Every Claude/GPT call is non-deterministic at the token level. Even with temperature 0, retries differ. You can't sign what you can't reproduce.

Why we can

Deterministic by definition

A recipe is pure code. fn(input) = output, always. Which means we can hash both ends and bind them to a source hash forever.

Anatomy of a receipt

{
  "spec":            "rs-1",
  "source_hash":     "8f3d…16hex",        // which code ran
  "input_hash":      "sha256/64hex",       // what you sent in
  "output_hash":     "sha256/64hex",       // what came out
  "version_id":      "ver_xyz789",        // registry pointer
  "runtime_version": "0.3.0",             // reproducer
  "issued_at":       "2026-05-06T17:42:11Z",
  "cache_hit":       false,
  "latency_us":      35,
  "hmac":            "f8a1…64hex"          // signs everything above
}

Three things receipts let you do

Proof for auditors

"Show me what ran in March"

Store receipts in your warehouse. Auditors get a tamper-evident log of every classification, route, and filter. Required under EU AI Act 2026.

Reproducible experiments

Pin the exact behavior

A receipt fixes the recipe at a hash. Six months later, you can rerun the exact same recipe and get bit-exact outputs. No "model drift" debate.

Portable trust

Verify without us

The verification algorithm is public canonical-JSON + HMAC. Build your own verifier in 50 lines. We're not in the loop after the receipt is issued.
