## P0 — ship-blocking, do this wave

**P0-1. `scripts/inject-cookbook-jsonld.cjs:42` injects the FORBIDDEN install pattern into every cookbook page's HowTo JSON-LD.**
The script writes `text: 'npm i -g @kolmogorov/kolm'` into a `<script type="application/ld+json">` tag that gets baked into every `/cookbook/*.html`. This is the exact string banned by `tests/site.test.js:63` (`FORBIDDEN_PUBLIC_PATTERNS`) and the correct replacement is documented in `scripts/fix-forbidden-claims.cjs:27-29` (`npm i -g github:sneaky-hippo/kolmogorov-stack`). Search engines + LLM crawlers eat schema.org HowTo first — this contradicts every other surface that uses the github tarball pattern. **Fix**: change line 42 to the github form. One-line change; rerun the inject script in the next deploy.

**P0-2. `public/playground.html:132` claims "live · in your browser" but the playground is hardcoded canned text.**
`gen()` at lines 236-270 returns string literals selected by sample id. `demoReceiptHash()` at 325-330 generates random hex labelled as a verified receipt. Trace messages at 350-358 print `verify HMAC chain: 7e3f...` / `signature ok` for verifications that never happened. The page lies about doing the load-bearing thing the product sells (HMAC-chained verifiable receipts). KOLM-103 in the backlog is the *real* browser playground (WASM) — until that ships, **delete the "live" eyebrow and label this clearly as a static demo**, or hit `/v1/public/run` (server-side) with the chosen sample and render the actual receipt. Honest static beats fake live for a buyer auditing the trust surface.

**P0-3. `src/auth.js:291-293` `adminApiKey()` falls back to `ks_admin_change_me` when no env var is set in non-prod.**
`isProductionRuntime()` is the only thing between this and a production-shaped environment leaking the literal string. Two failure modes: (a) a staging/preview deploy where `NODE_ENV !== 'production'` accepts a hardcoded admin key, (b) an env-var typo (`KOLM_ADMIN_KEY` vs `KOLM_ADMIN_API_KEY`) silently degrades to the fallback in prod. **Fix**: fail-closed always — return null + 503 the `/v1/admin/*` routes if no env key is set, even in dev. Print a single-line console warning at boot. Dev convenience is not worth the prod-leak risk.

## P1 — fix this wave or next

**P1-1. `public/onboarding.html:124` ships a fake Homebrew tap; line 127 "curl" tab is a copy-paste of the npm command.**
Line 124: `brew install sneaky-hippo/kolm/kolm` — no such tap exists, install will fail. Line 127: `npm i -g github:sneaky-hippo/kolmogorov-stack.sh | bash` is nonsense (`.sh | bash` chained to an npm package). **Fix**: collapse the install tabs to a single npm-via-github recipe matching `signup.html:161`. Or hide brew/curl tabs until those install paths actually exist.

**P1-2. `src/router.js:445-471` `/v1/signin` returns 501 to email+password posts with a confusing hint.**
The response says `not implemented; pass api_key to sign in`. A user who legitimately tried email+password gets a 501 with no path forward in the UI — `signup.html` only sends `api_key`, but a frontend bug or an external client posting password lands here. **Fix**: return 400 with `{ error: 'PASSWORD_AUTH_NOT_SUPPORTED', hint: 'sign in with your ks_ API key, or use Google/GitHub OAuth at /v1/oauth/google' }`. 501 is wrong (it's not "server doesn't know how" — it's "we don't accept passwords by design").

**P1-3. `public/signup.html:482-500` redundantly saves the api_key to 4 localStorage keys after the httpOnly cookie is already set.**
`persistThenGo()` writes `kolm_api_key`, `apiKey`, `recipeApiKey`, `ks_api_key` all in plain JS-readable storage. The 4-alias fallback (`KEY_NAMES` pattern in dashboard.html:197) is migration legacy. **Fix**: consolidate to one localStorage key (`ks_api_key`) and rely on the httpOnly cookie for transport. The 3 extra keys are XSS exfil surface for zero UX benefit — every page that needs the key can `fetch('/v1/account')` and use the cookie.

**P1-4. `src/router.js` is 4125 lines / 193KB; `cli/kolm.js` is 9416 lines / 427KB.**
Both files are above the threshold where new contributors stop reading and start grep-and-pray. Memory shows Wave 60i already had to fix a duplicate `/v1/admin/tenants` route exactly because the file got too big to scan. **Fix**: extract by feature seam — `routes/auth.js`, `routes/billing.js`, `routes/admin.js`, `routes/hub.js`, `routes/receipts.js`, `routes/lead.js`. CLI: `cli/cmd/{auth,build,publish,hub,verify,admin}.js` + a slim dispatcher. No behavior change, pure rearrange. This unblocks faster review on every subsequent wave.

**P1-5. `cli/kolm.js` verb-name vs behavior mismatch on `kolm verify`.**
`cli/kolm.js:3568` has a `cmdVerify` handler with its own help text, but `scripts/fix-forbidden-claims.cjs` actively rewrites `$ kolm verify` shell examples to `kolm inspect` across the public surface. So the docs say `inspect`, the CLI accepts `verify`, the help text mentions both. **Fix**: pick one. Either keep `verify` as a first-class verb and stop rewriting it, or alias `verify` to `inspect` with a one-line deprecation note in help and remove the standalone handler.

## P2 — cleanup, defensible to defer

**P2-1. `src/router.js` `/v1/account` returns `api_key` in the JSON body for cookie-authed clients to mirror to localStorage.**
This works for the 4-alias save pattern (P1-3), but defeats the httpOnly cookie's purpose — any XSS reads `/v1/account` and exfils the raw key. Once P1-3 is in, **strip `api_key` from the response body** and only return `api_key_prefix` for display.

**P2-2. `src/auth.js:41-45` `tenantKeyMatches()` still falls back to plain `api_key` column comparison.**
Migration legacy from the pre-hash era. If no tenant in the DB still has a plain `api_key` column populated (only `api_key_hash`), this branch is dead. **Fix**: confirm with a single SQL `SELECT count(*) FROM tenants WHERE api_key IS NOT NULL AND api_key_hash IS NOT NULL`, drop the fallback, save 4 lines.

**P2-3. `cli/kolm.js` EXIT enum used inconsistently.**
`cmdSignup` at 2210, `cmdLogin` at 2181 use bare `process.exit(1)` for missing-prereq and bad-args cases where `EXIT.MISSING_PREREQ` / `EXIT.BAD_ARGS` would let CI scripts branch on exit code. **Fix**: pass through the enum on the ~20 sites that currently use bare 1. Mechanical sed-able change.

**P2-4. Codebase rot at repo root.**
9× `server-*.log` files, 7× `tmp-*` directories, a literal `CUsersuserDesktopkolmogorov-stack/` directory (Windows-path-bug artifact from a broken bash call), `backups/v7.20-broken/`. None of this should be in the working tree. **Fix**: extend `.gitignore` for `server-*.log` + `tmp-*/`, `git rm -rf` the path-bug directory and the v7.20-broken backup. Doesn't ship but slows every grep.

**P2-5. `signup.html` is *also* `signin.html` via URL-aware H1 swap, but `signin.html` does not exist as a file.**
Works because vercel.json rewrites both to the same file. Fine for now, but tooling that scans for canonical sign-in surfaces (sitemap generators, link checkers) hits a single file with two URLs. **Fix**: either split into two physical files, or add a comment at `signup.html:1` documenting the dual-purpose pattern so the next contributor doesn't "fix" it.
