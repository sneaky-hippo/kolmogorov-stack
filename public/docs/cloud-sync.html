Cloud sync

Cloud sync is opt-in. By default nothing about your traffic leaves your machine. When you turn sync on you pick one of four states, and the privacy policy gates every byte that goes upstream. The sync target can be the hosted kolm.ai backend or a self-hosted bucket you control.

The 4 sync states

StateWhat gets sent
disabledNothing. The default. Sync daemon is not even running.
metadata_onlyRow count, latency p50/p95, cost totals, error rate. No message text. No prompt shas. No detector spans.
redacted_onlyRedacted message bodies (VAR_ placeholders only), redacted response bodies, schema-bound. Privacy classes set to block still block sync.
raw_enabledRaw and redacted bodies. Only available when storage mode is raw_local or raw_cloud_sync. Requires explicit per-namespace opt-in.

Per-class privacy gating

Even with cloud sync on, the privacy policy is the gatekeeper. Each detector class can have a different action for local storage and cloud sync.

{
  "default": {
    "ssn":        { "local": "redact", "sync": "block" },
    "credit_card":{ "local": "redact", "sync": "block" },
    "email":      { "local": "redact", "sync": "redact" },
    "mrn":        { "local": "redact", "sync": "block" },
    "ip_v4":      { "local": "allow",  "sync": "redact" }
  }
}

If the sync action for a class is block, any row that contained that class is held back from sync entirely, even if the local action redacted it.

Turn it on

kolm sync set redacted_only
kolm sync set disabled
kolm sync show
# state:      redacted_only
# target:     https://kolm.ai/v1/sync
# pending:    412 rows queued
# last-sync:  2026-05-18T09:02:14Z (124 rows, 312 KB)

Sync target

The default target is the hosted kolm.ai sync endpoint. Self-hosted is a single env var pointing at your own S3-compatible bucket or kolm.ai cloud relay.

export KOLM_SYNC_TARGET=s3://my-bucket/kolm-lake
kolm sync set redacted_only

Wire encryption

Sync uses TLS to the target and signs each batch with the local ed25519 key. The cloud cannot accept a batch that does not verify against the device's public key registered on first sync.

Audit what was sent

Every sync run writes a line to ~/.kolm/runtime/sync.log with batch sha, row count, class counts, and the public key fingerprint.

kolm sync audit --window 7d
# 2026-05-18T09:02 batch_sha=8f3a... rows=124 classes=email:18,phone:4,jwt:1
# 2026-05-17T09:01 batch_sha=4d12... rows=98  classes=email:14,phone:3

Pause and resume

kolm sync pause
kolm sync resume
kolm sync flush  # send everything queued now

Next steps

Pair sync with team shared namespaces if other people will read this data, and revisit privacy any time you flip a class to allow. Start from /docs/quickstart if your lake is still empty.
