What Type 2 actually examines

Type 1 and Type 2 differ in scope and effort. Type 1 is a snapshot, Type 2 is a film. The four lines below are the substance of that difference; they govern what evidence we collect, how often, and what we have to disclose.

ElementWhat the auditor verifiesCadence
Control design (Type 1)The control exists, is documented, and would work if operated as designed. The narrative, the policy, the system description match.Point-in-time
Control operation (Type 2)The auditor samples evidence over the observation window and verifies the control actually ran each time it should have. Design is necessary, operation is the bar.Continuous, 6 months minimum
Population samplingThe auditor selects roughly 10 to 25 percent of in-scope events for each control and requests the evidence record for each one. Stratified random across the window.Stratified random
Exception reportingAny control failure during the observation window is disclosed in the report, even if remediated the same day. Type 2 reports do not airbrush.Required even if remediated

How we collect Type 2 evidence today

Evidence collection is already running. The observation window starts in Q4 2026; what changes on day one is the auditor's sampling, not the instrumentation. Each line below is wired into production today and will keep emitting through the window.

  • Vanta agents on every laptop and CI runnerContinuous control monitoring across logical access, MFA enrollment, disk encryption, and patch level. Drift alerts within 24 hours.
  • HMAC-SHA-256 receipt chain on every .kolm compileProcessing-integrity evidence is automatic. Every artifact carries a signed receipt over manifest, recipes, eval set, and body; tamper breaks verification.
  • Audit log endpoint /v1/audit/logEvery privileged action is persisted, tenant-scoped, and queryable. Sampling for Type 2 reads from this endpoint; the integrity is HMAC-protected.
  • Compliance package endpoint /v1/account/compliance-packageThe auditor can pull a signed JSON bundle for any tenant on demand: controls, BAA status, subprocessor inventory, framework posture, receipt records, signed-at timestamp.
  • Quarterly access reviewEvery human and service account is reviewed against the role mapping. Orphans are dispositioned; the review minutes are kept as evidence.
  • Annual third-party penetration testHalborn, last completed Q1 2026, next engagement scheduled Q1 2027. Report shared on MNDA.
  • Background checks on workforceHireRight, conducted at hire for every employee with production access. Re-verification on role change to elevated access.
  • Vendor risk assessment per subprocessorEach subprocessor reviewed against its current security posture and SOC 2 letter. Inventory and change history at /subprocessors.

Subprocessor inheritance for Type 2

The kolm runtime is stateless. PHI never reaches Vercel, Railway, Stripe, Resend, Cloudflare, or GitHub; this is verified at /subprocessors. Only customer-opt-in inference providers (Anthropic, OpenAI, Google) ever receive customer payloads, and only with the customer's own API key.

The practical consequence: the Type 2 observation window scope is limited to the kolm compile pipeline, the audit log endpoint, and the compliance package endpoint. Subprocessor SOC 2 letters can be inherited where applicable; we publish the inheritance map in the audit room alongside the Type 2 report and update it on any subprocessor change.
