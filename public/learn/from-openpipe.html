From OpenPipe to .kolm.

Your dataset is the asset, not the fine-tune. This guide pulls your OpenPipe training data, compiles it into a signed offline .kolm, and verifies the K-score against the same held-out cases. Forty-five minutes end to end.

migration guide . v1.0 . valid for OpenPipe export format 2024+

What changes. OpenPipe serves your fine-tuned LoRA from their cloud; you pay per token and your traffic is round-tripped through their API. .kolm is a 30 KB file on your disk that runs locally, has a K-score, is HMAC-signed, and unzips into plain JavaScript recipes. You own the artifact; you can host it; you can ship it on hardware that never touches the internet.

step 1

Export the dataset.

5 min

OpenPipe stores your training data as chat-format JSONL. Pull the latest snapshot of your dataset:

  • Dashboard: Datasets → pick the dataset → Download JSONL
  • API: GET /api/v1/datasets/{id}/export with your OpenPipe API key
curl -H "Authorization: Bearer $OPENPIPE_KEY" \
  https://app.openpipe.ai/api/v1/datasets/your-dataset-id/export \
  > openpipe-export.jsonl
wc -l openpipe-export.jsonl
  4287 openpipe-export.jsonl
checkpoint the file should be valid JSONL; each line is {"messages":[...]}.
step 2

Normalize to .kolm shape.

10 min

OpenPipe emits OpenAI chat messages; kolm expects {input, output} pairs. Convert with a 12-line script. The pattern: the last assistant turn becomes output, everything before it becomes input.

cat > convert.mjs <<'EOF'
import fs from 'node:fs';
const lines = fs.readFileSync('openpipe-export.jsonl','utf8').split('\n').filter(Boolean);
const out = lines.map(line => {
  const row = JSON.parse(line);
  const msgs = row.messages || [];
  const lastAssistant = [...msgs].reverse().find(m => m.role === 'assistant');
  const prompt = msgs.filter(m => m.role !== 'assistant').map(m => m.content).join('\n');
  return JSON.stringify({ input: prompt, output: (lastAssistant && lastAssistant.content) || '' });
}).join('\n');
fs.writeFileSync('train.jsonl', out + '\n');
console.log('wrote train.jsonl with', lines.length, 'rows');
EOF
node convert.mjs
wrote train.jsonl with 4287 rows

Split 90/10 into train and held-out eval so you can measure regression honestly:

head -n 3858 train.jsonl > train-90.jsonl
tail -n 429 train.jsonl > holdout-10.jsonl

Validate the shape:

kolm seeds validate train-90.jsonl
ok. 3858 rows, shape {input, output}
checkpoint if kolm seeds validate reports parse error or CSV detected, fix the source format before continuing.
step 3

Write the spec.

10 min

OpenPipe inferred the task from your dataset. kolm wants you to state it explicitly so the K-score is honest. Start from a template that matches the shape of your OpenPipe fine-tune.

If your OpenPipe model is doing…Use template
Customer-support routing, tagging, intent classificationclassifier
Pulling structured fields from forms or documentsextractor
Long-form summary or rewritingsummarizer
PII removal, content filteringredactor
Anything elseblank
 
kolm new my-migration --from classifier
# edit my-migration.spec.json: set task, categories, gates
$EDITOR my-migration.spec.json

A minimal spec for a 5-class router (mirrors OpenPipe's chat-format training shape):

{
  "task": "route incoming customer messages to a category",
  "shape": { "input": "string", "output": { "label": "string" } },
  "labels": ["billing","bug","auth","feature_request","how_to"],
  "gates": { "K_min": 0.85, "size_max_kb": 200, "p50_ms_max": 50 }
}
step 4

Compile.

5 min

Compile against your normalized data. The compiler will fit a recipe, score it on the held-out set, and sign the artifact.

kolm compile \
  --spec my-migration.spec.json \
  --examples train-90.jsonl \
  --eval holdout-10.jsonl \
  --out my-migration.kolm
compiling . . . . . . . . . . . . . . . . . . . . . . . . . . . .
K-score for my-migration.kolm: 0.911  (gate >= 0.85 - pass)
size:        47.2 KB
p50 latency: 8 ms
verifying signature . ok
artifact written: my-migration.kolm
checkpoint if K-score is below the gate, run kolm eval my-migration.kolm --examples holdout-10.jsonl --trace to see which cases regressed; the trace lists every miss with expected vs got.
step 5

Compare to OpenPipe head-to-head.

10 min

Run both on the same held-out cases. This is the only fair benchmark.

kolm eval my-migration.kolm --examples holdout-10.jsonl --json > kolm-results.json
node compare-openpipe.mjs # your existing OpenPipe eval script
diff <(jq -r '.cases[].got' kolm-results.json) <(jq -r '.got' openpipe-results.jsonl)

step 6

Ship.

5 min

Three deploy paths, pick one:

Local CLI for batch jobs:

echo {"input":"reset my password please"} | kolm run my-migration.kolm

HTTP server as a drop-in OpenPipe API replacement:

kolm serve my-migration.kolm --port 8080
listening on http://127.0.0.1:8080
POST /v1/run    runs the artifact against {input: ...}

Hub publish for team sharing with a verifiable SHA-pinned download URL:

kolm publish my-migration.kolm --private

Things that change

  • Latency drops 10-50x. You stop paying the network round-trip.
  • Cost goes to zero per inference. The fixed cost is one kolm compile and the laptop you already own.
  • Data stops leaving your boundary. The artifact runs where you put it. No third-party log retention.
  • You get a K-score and a signature. OpenPipe gives you a fine-tune ID; kolm gives you a number you can put in a security review.

Things that stay the same

  • Your dataset is still the source of truth. You moved it, you did not rebuild it.
  • Your evaluation discipline. Same held-out set, same metric, same definition of better.
  • Your model selection. If your OpenPipe fine-tune was based on a particular open model, you can pin the same base in your kolm spec.

If something went wrong

  • OpenPipe export is OpenAI chat-completion format with tools calls: map each tool call to a JSON output field in your spec's shape.output; the included convert.mjs assumes plain text output.
  • K-score below 0.85: your spec is under-constrained or the data is noisier than expected. Try kolm tune my-migration.kolm --examples train-90.jsonl to fit recipe parameters automatically.
  • Latency higher than expected: kolm inspect my-migration.kolm shows runtime mode; switch "runtime": "local" in the spec for offline execution.
