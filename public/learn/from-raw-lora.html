From raw LoRA to .kolm.

You trained a LoRA with PEFT, Unsloth, Axolotl, or HF Trainer. The adapter is already on disk. This guide wraps that adapter, the base model reference, and your held-out data into one signed .kolm artifact with a real K-score. Thirty minutes, no retraining.

migration guide . v1.0 . PEFT 0.11+, Unsloth, Axolotl, llama.cpp/MLX exports supported

Why wrap. A raw LoRA folder is three things in different formats: weights (safetensors), config (JSON), and your eval discipline (uncodified, lives in your head). .kolm binds the three into a deterministic ZIP with a content hash, a K-score against the eval set, and an HMAC chain. Same model, same accuracy; the difference is provenance and portability.
v0.2 provenance (wave 144). The v0.2 kolm compile path emits a recipe-class artifact: deterministic compiled code, HMAC-signed manifest + receipt, optional kolm-moe-1 composition block across N expert recipes, optional bundled kolm-tokenizer-1 byte-level BPE, a KOLMIDX deterministic lookup container, and the K-score gate. Loading your adapter_model.safetensors bytes into the artifact as a populated recipes/lora.bin member (and lighting up runtime inference against the base) lands with the roadmap-tier runtime bridge. Until then, this guide gives you the manifest-and-eval wrap; the live weight-bridge half lights up once the roadmap tier ships.

step 1

Stage the adapter + held-out data.

5 min

Put everything in one folder; kolm reads from there.

mkdir staging
cp /path/to/your/lora/adapter_model.safetensors staging/
cp /path/to/your/lora/adapter_config.json staging/
cp /path/to/holdout-eval.jsonl staging/

Verify adapter_config.json tells you what you need:

cat staging/adapter_config.json | head -20
{
  "base_model_name_or_path": "meta-llama/Llama-3.1-8B-Instruct",
  "peft_type": "LORA",
  "r": 16,
  "lora_alpha": 32,
  "target_modules": ["q_proj","v_proj","k_proj","o_proj"],
  "task_type": "CAUSAL_LM"
}

The held-out file is JSONL with {input, output} or {input, expected} pairs. If your eval was in a notebook or only in your head: pause this migration and write 50-100 examples. The K-score is meaningless without a real eval set.

kolm seeds validate staging/holdout-eval.jsonl
ok. 248 rows, shape {input, output}
step 2

Write the spec.

5 min

The spec describes the task you fine-tuned for. Concrete is better than clever: state the task in one English sentence, declare input/output shape, set gates that mean something to a security reviewer.

{
  "task": "extract structured invoice fields from raw PDFs converted to text",
  "shape": {
    "input": "string",
    "output": {
      "invoice_number": "string",
      "iso_date": "string",
      "amount": "number",
      "currency": "string"
    }
  },
  "base": "meta-llama/Llama-3.1-8B-Instruct",
  "adapter": {
    "type": "lora",
    "rank": 16,
    "alpha": 32,
    "target_modules": ["q_proj","v_proj","k_proj","o_proj"]
  },
  "gates": {
    "K_min": 0.85,
    "size_max_mb": 200,
    "p50_ms_max": 250,
    "deterministic": true
  }
}

Save as staging/my-extractor.spec.json. The adapter block matches your adapter_config.json verbatim; kolm uses it to verify the wrapped LoRA loads correctly.

step 3

Wrap with kolm wrap.

5 min

kolm wrap takes a pre-trained adapter and produces a signed .kolm without re-running training. It bundles the weights, the spec, the eval set, computes the K-score against the held-out cases, and signs the receipt chain.

cd staging
kolm wrap \
  --adapter adapter_model.safetensors \
  --adapter-config adapter_config.json \
  --spec my-extractor.spec.json \
  --eval holdout-eval.jsonl \
  --out my-extractor.kolm
loading adapter (16-rank LoRA, 4 target modules) . ok
loading base ref: meta-llama/Llama-3.1-8B-Instruct
quantizing adapter to int8 . done
running eval on 248 held-out cases . . . . . . . . . done in 42s
K-score for my-extractor.kolm: 0.927  (gate >= 0.85 - pass)
size:        118.3 MB
p50 latency: 187 ms
signing manifest, recipes, evals, receipts . ok
artifact written: my-extractor.kolm
checkpoint the K-score reflects the eval set you provided. If it is lower than you expected, the eval set may have changed since training, or your validation discipline during training was looser than the K-score's. This is a feature.
step 4

Inspect + ship.

5 min 
kolm inspect my-extractor.kolm
artifact:    my-extractor.kolm
task:        extract structured invoice fields from raw PDFs converted to text
built:       2026-05-16 16:08:14Z
size:        118.3 MB
K-score:     0.927  (gate >= 0.85 - pass)
base:        meta-llama/Llama-3.1-8B-Instruct
adapter:     lora (rank=16, alpha=32)
runtime:     local
signature:   valid (hmac-sha256)
evals:       248 cases

Now ship. Three options:

# Local
kolm run my-extractor.kolm --input invoice.txt

# HTTP server (drop-in vLLM-compatible)
kolm serve my-extractor.kolm --port 8080 --vllm-compat

# Publish to the hub
kolm publish my-extractor.kolm --private

If your adapter is GGUF/GGML or MLX, not safetensors

kolm wrap also accepts:

  • GGUF: --adapter my-adapter.gguf
  • MLX: --adapter my-adapter.npz --adapter-config mlx-config.json
  • Pre-merged weights (LoRA already merged into the base): --merged --weights my-merged.safetensors ships a single-file artifact with no adapter separation.

If you also have training data and want to recompile

Skip kolm wrap; use kolm compile instead. The compiler can retrain a fresh LoRA against your data, optionally tune hyperparameters, and produce a .kolm with a higher K-score than your original. See From Predibase, step 4 for the compile path.

Verifying the bits are the same

Re-extract the wrapped LoRA and binary-diff against your original to prove the wrap was lossless:

kolm eject my-extractor.kolm
diff staging/adapter_model.safetensors my-extractor.eject/recipes/lora.safetensors
# (empty output means identical)
sha256sum staging/adapter_model.safetensors my-extractor.eject/recipes/lora.safetensors

What you get

  • One file instead of a folder.
  • A K-score against an eval set you committed to.
  • HMAC chain proving the bits have not been tampered with since wrap.
  • Reproducible verification: anyone with the file can run kolm inspect and reach the same K-score.
  • Portability: ship to a phone (CoreML/ONNX export), publish to the hub, or eject and use with any compatible runtime.
