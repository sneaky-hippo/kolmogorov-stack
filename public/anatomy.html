02 ·Anatomy

One file. Seven layers.

A .kolm is a deterministic zip with a fixed schema (RS-1). Every layer is reproducible from (task, examples, base_model, recipe_registry@version). Two compiles with the same inputs produce byte-identical artifacts.

v0.1 truth: v0 ships recipe-tier artifacts. The zip member names below are the v0.1 layout; today model.gguf holds a signed pointer record, lora.bin holds an optional behavior pack (KOLMPACK), and index.sqlite-vec holds an optional JSON lookup (KOLMIDX). Native-weight, LoRA-adapter, and sqlite-vec tiers are on the rs-2/rs-3 roadmap.

01 ·What'inside

The seven parts.

Listed in the order kolm writes them. Sizes are an example from a typical task; small artifacts run ~30 MB, large ones cap around 3 GB.

  • 01

    model.gguf v0.1 ·pointer

    In v0.1 this slot holds a signed JSON pointer to the base-model release (runtime: cloud); native INT4 weight packaging is a roadmap tier.

    target 2.1 GB
  • 02

    lora.bin v0.1 ·behavior pack

    In v0.1 this slot holds an optional behavior pack: deterministic categories and rules your task can match against. Distilled LoRA adapters land in a future tier.

    target 82 MB
  • 03

    recipes.json

    The recipe pack: deterministic-token drafts indexed by prefix-shape and embedding-shape. Drives speculative decoding for the structured-output subset.

    7.2 MB
  • 04

    index.sqlite-vec v0.1 ·lookup index

    In v0.1 this slot holds an optional JSON lookup container: keyword-to-recipe maps for fast retrieval. A sqlite-vec multimodal index lands in a future tier.

    target 11 MB
  • 05

    tests.jsonl

    This artifact's own eval suite. The K-score is computed against these cases, not against any external leaderboard. Travels with the artifact. kolm eval re-runs the embedded evals locally; no eval, no ship.

    1.4 MB
  • 06

    manifest.json

    Task description, base hash, training stats, registry version, this artifact's K-score and gate, recall model hashes. The contract for what this file is.

    12 KB
  • 07

    signature.sig

    HMAC metadata over emitted layer hashes. External anchoring is a reserved extension, not a current default.

    256 B

01b ·v0.2 additions (wave 144)

Two new optional layers.

v0.2 (wave 144) ships two new optional members the seven-layer schema didn'previously name. They appear when you opt in – kolm compile --tokenizer tokenizer.json for the first, kolm moe compose for the second – and are covered by the same HMAC ring as every other layer.

  • 08

    tokenizer.json v0.2 ·byte-level BPE

    Pure-JS byte-level BPE tokenizer following the kolm-tokenizer-1 spec. Trains via kolm tokenize train --corpus seeds.jsonl --field input, embeds via kolm compile --tokenizer tokenizer.json. The byte-level fallback guarantees UTF-8 round-trip for any input. Manifest records tokenizer.spec, tokenizer.vocab_size, and tokenizer.sha256.

    ~50–800 KB
  • 09

    moe block (training_stats.moe) v0.2 ·composite

    When you run kolm moe compose --expert a.kolm --expert b.kolm --router router.json --out composite.kolm, the resulting composite carries a training_stats.moe block: per-expert id + sha256 + recipe_id, plus the full router spec (keyword, intent_field, or first_match). The receipt chain covers this block, so the composition is tamper-evident along with everything else.

    ~2–8 KB
02 ·Three invariants

Three rules every .kolm respects.

Reproducible

Same (task, examples, base, registry@v)   - nbsp; byte-identical zip. No hidden randomness; sampling seeds are pinned in the manifest.

Self-evaluating

The artifact carries the tests it was scored against. kolm eval re-runs embedded evals and reports the current score.

Tamper-evident

HMAC metadata covers emitted layer hashes. Inspection and eval make unexpected artifact changes visible.

03 ·Walk the chain

Four rings. One signature.

The signature.sig file is the tail of a 4-ring HMAC-SHA256 chain. Each ring covers the previous ring's tag plus a fresh set of layer hashes. Tampering with one byte of any layer changes that ring's tag and cascades through every downstream ring. kolm inspect re-derives all four rings in about 80 ms and matches against signature.sig.
