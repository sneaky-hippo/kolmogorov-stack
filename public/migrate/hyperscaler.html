kolm  /  migrate  /  from hyperscaler

Distill from OpenAI or Anthropic into a local .kolm.

Hyperscaler fine-tuning (OpenAI custom models, Anthropic custom Claude, Google Vertex AI tuning, AWS Bedrock customization) keeps the tuned weights inside the vendor's infrastructure. The model exists only behind their API. That works until your deployment target moves outside the vendor's reach — air-gap, edge, offshore subsidiary, on-prem mirror. The kolm pattern flips it: use the hyperscaler API as the teacher, capture the prompt/completion pairs into seeds.jsonl, then LoRA-fine-tune a small open-weight base. The student approximates the teacher on your task; you ship a file you actually own.

Wave 188 · from hyperscaler · cross-vendor distillation migration

Teacher / student split.

OpenAI or Anthropic API (teacher)→kolm capture→seeds.jsonl→kolm distill→open-weight student .kolm

The hyperscaler stays in the loop as the highest-capability call you make per training pair, but its output flows into a file you control. The student LoRA — Qwen 0.5B, Llama 3.2 1B, Phi-3.5 mini, SmolLM2 — runs anywhere PyTorch or llama.cpp runs. Your auditor gets a signed receipt chain that names the teacher vendor, the student base, the redactor configuration, and the eval gate.

Licensing carve-out: OpenAI / Anthropic / Google terms restrict using API outputs to develop "competing" general-purpose models. A narrow-task kolm artifact (phone normalizer, denial-code classifier, sentiment tagger) is not a competing general model; the legal status of the narrow-task carve-out is unsettled in 2026. The always-safe option is --teacher=local:<open-weight-model> run on your own GPU. See RS-1 §7.7 for the licensing posture and the Wave 158 cross-vendor manifest that records which vendor saw what.

The four-step migration.

01 · capture the teacher traffic

kolm capture proxies your existing API calls.

Point your client at the kolm capture proxy; the same API key passes through. Every prompt + completion lands in a local seeds.jsonl namespace. No SDK change needed.

$ kolm capture \
    --provider anthropic \
    --as denial-classifier \
    --namespace claims

  LISTEN 127.0.0.1:7117
  capturing → ~/.kolm/captures/claims/seeds.jsonl
  forwarding → https://api.anthropic.com/v1
  threshold:  1000 pairs · required to distill
02 · reach the threshold

Run normal traffic until the namespace fills.

The proxy is transparent. Your application keeps calling the hyperscaler as before; kolm logs the (input, output) pair every time. Once the namespace clears the threshold, kolm distill unlocks.

$ kolm capture status --namespace claims
  pairs captured:  2417
  threshold:       1000  (unlocked)
  size on disk:    14.3 MB
  latest:          2026-05-17T18:42:11Z
  redactor:        PHI (Wave 157)
03 · distill into a student

kolm distill builds the open-weight LoRA.

Same teacher you already paid for, named explicitly in the manifest. Choose any of the 13 student bases in the Wave 158 catalog. Always-safe fallback: --teacher=local: for a fully on-prem build.

$ kolm distill \
    --namespace claims \
    --teacher=anthropic:claude-opus-4-7 \
    --student-base=qwen2.5-0.5b \
    --redact=phi --eval-gate=0.88

  PASS  K-score 0.911 · PHI redactor clean
  PASS  cross-vendor manifest · teacher = anthropic
  WROTE denial-classifier.kolm
04 · verify, then ship anywhere

The .kolm runs on a phone, an edge box, an air-gapped VPC.

Verifier rebuilds the receipt chain offline. No vendor API required at runtime. The deployment target that was previously impossible (residency-bound, air-gap, on-prem) is now first-class.

$ kolm verify denial-classifier.kolm
  PASS  24/24 checks · signature OK
  PASS  cross-vendor: teacher=anthropic,
        student=qwen2.5-0.5b
  PASS  receipt chain intact

# run anywhere — no vendor API at runtime
$ kolm run denial-classifier.kolm \
    --input "claim denied for code CO-45..."

What survives, what kolm adds.

Survives the migration unchanged

  • Existing hyperscaler contract. Your OpenAI / Anthropic / Google / Bedrock account stays. kolm just calls it as the teacher.
  • Prompt engineering work. The system prompts and few-shot exemplars you tuned against the teacher are the inputs to capture.
  • Eval set. Your existing eval cases forward into the K-score gate as the frozen holdout.
  • Vendor BAA (HIPAA workloads). The teacher call still flows through the vendor's BAA-covered API; kolm's PHI redactor runs before the call goes out.
  • API budget. Same per-token billing — capture phase pays for traffic you were already paying for.

kolm adds on top new

  • Portable file. The structural impossibility hyperscaler tuning can't close: weights you ship on a USB stick if you want.
  • Air-gap deploy. Runtime has zero vendor dependency. Verifier rebuilds offline.
  • Cross-vendor manifest (Wave 158). teacher_vendor, teacher_model, student_base, distillation_method — recorded so check #15 enforces it.
  • Always-safe local teacher. --teacher=local:llama-3.3-70b-instruct swaps in when the licensing carve-out makes you nervous.
  • Residency. The student weights live on your filesystem. Deploy literally anywhere.

Where staying with hyperscaler fine-tuning is the right answer.

Stay with hyperscaler tuning when your only deploy target is a hosted API and the vendor's BAA + console audit logs satisfy your auditor. Managed GPU, managed dataset upload, managed eval UX, zero ML infra to operate — that's the shortest path from "I want a tuned model" to "I have a tuned endpoint." The portability kolm adds earns nothing if the model never has to leave the vendor's cloud. The kolm envelope is engineering cost in service of a need you don't have.

Stay with hyperscaler tuning when procurement wants the Series-mega-cap vendor with enterprise contracts, dedicated TAMs, and vendor SLA. Your buyer signed off on OpenAI / Anthropic / Google as the named line item; replacing the build path with a kolm-wrapped open-weight student is a procurement re-litigation you don't want to start. The teacher-student pattern is then useful as a fallback or mirror, not a primary path.
