Add kolm to your LoRAX stack.

LoRAX (Apache-2.0, github.com/predibase/lorax) is the strongest multi-LoRA serving infrastructure in market — heterogeneous continuous batching, adapter exchange scheduling, the SGMV kernel. Nothing about your serving cluster needs to change. The migration is additive: add kolm on the build side so the LoRA bytes LoRAX serves arrive with a signed receipt chain attached, then keep serving them exactly as you do today. The signed .kolm file is the contract between the build team and the serve team. Complement, not replacement.

Wave 188 · from LoRAX · complement migration

The stack with kolm added.

Today: you train a LoRA somewhere (HuggingFace AutoTrain, a notebook, raw peft), drop the weights into LoRAX, and serve. Tomorrow: kolm sits where the training happened, captures the inputs to the build, runs the same train, and emits a signed file that contains the same LoRA bytes plus the manifest a verifier needs.

seeds.jsonl→kolm distill→signed .kolm + LoRA weights→LoRAX→OpenAI-compatible endpoint

The LoRA weights LoRAX loads are bit-identical to what they would have been without kolm. The signed file is a superset: same LoRA payload, plus the manifest the receipt chain needs. LoRAX ignores the manifest fields; kolm verify reads them when an auditor asks for proof.

The four-step migration.

01 · reshape the data

Point your existing data at seeds.jsonl.

Whatever exporter currently feeds your LoRA trainer is your migration tool. kolm reads JSONL with {"input","output","tags"}. Most teams already export this shape; if you don't, a one-liner converts.

# bring your existing pairs into the kolm canonical shape
$ jq -c '{input: .prompt, output: .completion}' \
    train.jsonl > seeds.jsonl
$ wc -l seeds.jsonl
  12847 seeds.jsonl
02 · build the signed adapter

kolm distill calls the same trainer.

The trainer call is the one you were already making (TRL / Unsloth / PEFT). kolm wraps it so the inputs hash into the receipt chain and the output gets signed.

$ kolm distill phone-normalizer \
    --teacher=anthropic:claude-opus-4-7 \
    --student-base=qwen2.5-0.5b \
    --redact=phi --eval-gate=0.85

  PASS  K-score 0.917 (R/F/E/Z)
  PASS  PHI redactor · no leakage
  WROTE phone-normalizer.kolm
03 · hand the LoRA to LoRAX

Extract the bytes LoRAX already knows.

The artifact is a tar-ish archive; the LoRA payload sits at a stable path inside. Extract it into the adapter directory your LoRAX cluster mounts.

# extract LoRA payload from the signed artifact
$ kolm extract phone-normalizer.kolm \
    --target=lora-weights \
    --out=./adapters/phone-normalizer
  shipped

# LoRAX serves it exactly as before
$ docker run --gpus all -v ./adapters:/data \
    ghcr.io/predibase/lorax:latest \
    --model-id Qwen/Qwen2.5-0.5B --adapter-source local
04 · verify the contract

kolm verify proves what the LoRA was built from.

Any party with the public key can confirm the LoRA LoRAX is serving matches what kolm signed at compile time. Receipt chain, eval gate, PHI redactor — all re-derivable offline.

$ kolm verify phone-normalizer.kolm
  PASS  24/24 checks
  PASS  Ed25519 + Sigstore validated
  PASS  receipt chain intact
  PASS  K-score gate cleared
  PASS  redactor produced no PHI leak

What survives, what kolm adds.

Survives the migration unchanged

  • LoRAX cluster. Your Helm chart, your GPU pool, your routing config, your tenant adapter directory layout. Nothing changes server-side.
  • Base model selection. Whatever Llama / Mistral / Qwen / Zephyr base you serve LoRAs against today is the --student-base you pass to kolm distill.
  • OpenAI-compatible API surface your clients call. LoRAX still exposes /v1/chat/completions; clients don't need a base-URL change.
  • Multi-tenant batching. SGMV kernel + adapter exchange scheduling continue to do their job. kolm has zero opinion about how the bytes are served.
  • LoRA weights themselves. The bytes LoRAX loads are the same bytes kolm distill wrote — the signed file wraps them, doesn't reshape them.

kolm adds on top new

  • Receipt chain · spec → seeds → split → train → recipes → evals → signatures. Auditor walks the chain offline.
  • Ed25519 + Sigstore signature. The bytes LoRAX loads can be proven to match what the build team signed.
  • PHI redactor (Wave 157) before the teacher call. 18 HIPAA Safe Harbor + 3 kolm extensions. Verifier check #14 confirms no raw PHI in the LoRA weights.
  • K-score gate on the build — compile blocks if eval is below threshold. The bytes LoRAX serves are bytes that passed a gate.
  • Drift detection (Wave 167). When the artifact ages out, the supersession chain documents the replacement.

Where LoRAX alone is the right answer.

Stay with LoRAX alone when your team controls the LoRA training pipeline, the data isn't regulated, no auditor is asking for a receipt, and you don't need the artifact to be portable beyond your serving cluster. LoRAX is excellent on its own · running it bare is the cheapest path to multi-tenant adapter serving and the production-readiness pieces kolm adds carry build-time cost (signing, attestation, redactor pre-pass). If none of those land on your roadmap, you're paying for receipts no one is going to read.

Stay with LoRAX alone if your one-and-only deploy target is the LoRAX cluster itself and you have no air-gap, no edge, no on-prem mirror requirement. The .kolm format earns its keep when the artifact has to travel. If it never leaves your hosted cluster, the portability tax is wasted.
