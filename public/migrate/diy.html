kolm  /  migrate  /  from DIY stack

Wrap your LLaMA-Factory or Axolotl pipeline with kolm.

LLaMA-Factory, Axolotl, Unsloth, TRL, PEFT — the DIY LoRA stack is what kolm builds on, not what kolm competes with. kolm is shrink-wrap around the stack you already run. The trainer call kolm emits is the same trainer call you were making by hand; the kernel optimizations Unsloth provides still apply; the YAML configs you tuned over six months keep working. What changes: the build now emits a signed artifact with a receipt chain, the eval is gated by a frozen K-score threshold, and there's an explicit drop-out path back to raw llamafactory-cli train at any time.

Wave 188 · from DIY stack · shrink-wrap migration

The stack with kolm wrapped around it.

kolm does not replace any layer below. It sits above LLaMA-Factory / Axolotl, calls the same trainers, lets Unsloth kernels keep doing what they do, and writes the build outputs into a signed envelope. The DIY stack is the foundation; kolm is the production-readiness wrap.

kolm · receipt chain + eval gate + Ed25519 + Sigstore + PHI redactor + cross-vendor manifest + drift cron

The contract: kolm never hides the underlying stack. The manifest names which backend was invoked, which base was loaded, which trainer class ran, which LoRA rank and alpha were used. kolm extract --target=training-config dumps the YAML; llamafactory-cli train --config dump.yaml rebuilds bit-identical weights. If kolm goes away, the DIY stack still runs your config.

The four-step migration.

01 · reuse your dataset

Point kolm at the JSONL you already curated.

Whatever dataset your LLaMA-Factory or Axolotl config currently reads is the kolm input. Conform to {"input","output","tags"} and you're done.

# your existing LLaMA-Factory dataset already
# lives somewhere like data/alpaca.json — reshape
$ jq -c '{input: .instruction, output: .output}' \
    data/alpaca.json > seeds.jsonl
$ wc -l seeds.jsonl
  52002 seeds.jsonl
02 · wrap the trainer call

kolm distill calls the same backend.

Pass --backend=unsloth or --backend=llamafactory and kolm shells out to the same trainer your YAML would have invoked. Deterministic flags pinned into the manifest.

$ kolm distill alpaca-style \
    --backend=unsloth \
    --student-base=llama-3.2-1b \
    --seeds seeds.jsonl \
    --eval-gate=0.85

  USING unsloth backend · SFTTrainer
    lora_rank: 16, lora_alpha: 32
  PASS  K-score 0.873
  WROTE alpaca-style.kolm
03 · inspect the wrap

Confirm kolm called the trainer you expected.

kolm inspect renders the actual backend invocation from the artifact manifest. Auditor-readable; debuggable; matches what your YAML would have produced.

$ kolm inspect alpaca-style.kolm \
    --field=training_config
  backend:     unsloth
  base_model:  meta-llama/Llama-3.2-1B
  trainer:     SFTTrainer
  lora_rank:   16
  lora_alpha:  32
  optim:       adamw_8bit
  max_steps:   1000
  seed:        42
04 · drop out anytime

kolm extract dumps a runnable YAML config.

The escape hatch is first-class. Extract the training config from the manifest, hand it to llamafactory-cli train, and you're back in raw DIY. Same flags, bit-identical weights.

$ kolm extract alpaca-style.kolm \
    --target=training-config \
    --out train.yaml
  shipped

$ llamafactory-cli train --config train.yaml
  — same backend, same flags, identical weights

What survives, what kolm adds.

Survives the migration unchanged

  • LLaMA-Factory / Axolotl as the trainer driver. kolm wraps; doesn't replace.
  • Unsloth kernels. 2-5x speedup and lower VRAM still apply because kolm shells out to the same backend.
  • TRL trainer classes — SFTTrainer, DPOTrainer, GRPOTrainer. kolm passes the trainer choice through.
  • PEFT primitives. LoRA, QLoRA, IA3, prefix tuning — whichever you used before keeps working.
  • Your YAML config. Tuned over months; the kolm spec layer is a thin envelope over it. kolm extract dumps the YAML back.

kolm adds on top new

  • Signed artifact. Ed25519 + Sigstore / Rekor. The DIY stack doesn't sign.
  • Receipt chain. 14-step provenance ledger over spec, seeds, split, train, recipes, evals, signatures.
  • K-score gate. Compile blocks if eval is below threshold — the DIY stack runs to completion regardless.
  • Cross-vendor teacher manifest. --teacher=anthropic: / openai: / local: with Wave 158 catalog tracking who saw what.
  • PHI redactor (Wave 157). Pre-teacher-call redaction; verifier check #14 confirms no raw PHI in weights.

Where raw DIY is the right answer.

Stay with raw DIY when you are a research team iterating on training methods themselves — new optimizers, new LoRA variants, custom kernels, ablations. The shortest path is the rawest one. llamafactory-cli train or a notebook gives you full control with no wrap to fight. The receipts kolm produces are valuable for production deployments; for a research run that won't ship, the wrap is overhead.

Stay with raw DIY when no one is asking for receipts. If your buyer doesn't request a provenance chain, your data isn't regulated, and your serving target is something you control end-to-end, the DIY stack is free, mature, and well-documented. The DIY stack is excellent on its own. Don't pay a wrap tax for a workflow whose value is the training experiment itself.
