What the Rubrik acquisition changed, and what stays.

June 2025: Rubrik (NYSE: RBRK) acquired Predibase. The combined product positions toward agentic AI governance — monitoring agent actions, rewinding bad state, applying Rubrik's data-security posture to LLM workloads. Predibase's hosted fine-tuning and LoRAX serving products are still shipping; the Rubrik enterprise sales motion is now wrapped around them.

The acquisition validated the category kolm sits adjacent to but did not close the build-side gap. Rubrik + Predibase governs the runtime; kolm proves the build. An agent action monitor doesn't tell you what teacher distilled the model, what data the train set held, whether the eval gate fired, or whether the bytes on disk match the bytes signed at compile time.

The two products stack cleanly: kolm produces the signed adapter, Predibase serves it on the cluster Rubrik governs. The signed file is the contract between the build artifact and the runtime governance, and your existing Rubrik procurement contract is unaffected.

The four-step migration.

01 · export your dataset

Pull your Predibase dataset into seeds.jsonl.

Predibase datasets export as JSONL with a prompt/completion shape. kolm reads the canonical {"input","output","tags"}. A one-pass rename gets you there.

# export from Predibase console or API
$ pbase dataset export --id ds_abc > predibase.jsonl

# reshape into kolm canonical
$ kolm import predibase --in predibase.jsonl \
    --out seeds.jsonl
  verify before ship · today: jq -c \
    '{input: .prompt, output: .completion}'
02 · build the signed artifact

kolm distill in place of the Predibase trainer.

Same student base, same teacher, deterministic flags pinned into the manifest. The Predibase Turbo LoRA inference path still serves the LoRA bytes downstream — only the build step moves.

$ kolm distill denial-classifier \
    --teacher=anthropic:claude-opus-4-7 \
    --student-base=qwen2.5-1.5b \
    --redact=phi --eval-gate=0.88

  PASS  K-score 0.903 · train/holdout disjoint
  PASS  PHI redactor · no leakage
  WROTE denial-classifier.kolm
03 · verify before deploying

kolm verify hands a receipt to Rubrik.

Run the verifier before the artifact leaves your build machine. The 24-check pass is what your auditor or Rubrik governance dashboard cares about.

$ kolm verify denial-classifier.kolm
  PASS  24/24 checks
  PASS  Ed25519 + Sigstore validated
  PASS  receipt chain intact
  PASS  K-score gate cleared
  PASS  redactor produced no PHI leak
04 · ship to Predibase serving

Deploy the .kolm where you used to deploy the LoRA.

Predibase serving accepts the LoRA payload. Extract it from the signed bundle, push to the same adapter endpoint, keep your existing routing config.

# extract the LoRA bytes the Predibase server expects
$ kolm extract denial-classifier.kolm \
    --target=lora-weights \
    --out=./lora
  verify before ship · today: unzip denial-classifier.kolm

# upload to Predibase via their existing CLI
$ pbase adapter push ./lora --name denial-classifier
  POSTED to Predibase

What survives, what kolm adds.

Survives the migration unchanged

  • Predibase / Rubrik contract. Your existing enterprise procurement, dedicated VPC SKU, and Rubrik governance posture stay in place. The build move is silent to procurement.
  • Turbo LoRA serving on Predibase infra. The 2-3x faster inference claim still applies because the LoRA bytes are bit-identical.
  • Base model selection · whichever Llama / Qwen / Mistral base Predibase already serves stays the --student-base.
  • Existing eval set. Predibase eval metrics forward into kolm as the frozen holdout the K-score gate measures against.
  • Rubrik agent monitoring. The agentic governance dashboard still observes runtime calls. kolm doesn't touch the runtime.

kolm adds on top new

  • Receipt chain · cryptographically verifiable build provenance the Predibase console can't render.
  • Portability. The .kolm file runs on a phone or edge box if you ever need to mirror outside Predibase.
  • PHI redactor (Wave 157) before the teacher call. Predibase's BAA covers data-in-transit; kolm's redactor proves no raw PHI is bound into the weights.
  • Cross-vendor teacher. --teacher=anthropic:, --teacher=openai:, --teacher=local: · Wave 158 manifest tracks which vendor saw what.
  • Drift detection (Wave 167). Lifecycle ladder + supersession chain even though Predibase / Rubrik don't ship one.

Where staying with Predibase end-to-end is the right answer.

Stay all-Predibase when procurement is the binding constraint, not technical receipts. The Rubrik enterprise sales motion, dedicated VPC SKU, and Series-C balance sheet are what your buyer signed off on. If your auditor is satisfied with Rubrik's governance dashboard and console audit logs, adding kolm is a build-side complexity tax for a need they don't have.

Stay all-Predibase when your workload is multi-tenant serving at scale and the question your team spends the most engineering time on is throughput, not provenance. Turbo LoRA + LoRAX on managed infra has more operational maturity in 2026 than any build-side receipt chain. Don't move the build to kolm if the build isn't where your pain lives.
