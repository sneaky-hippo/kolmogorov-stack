Install failures

command not found: kolm

The npm global bin is not on your PATH. Run npm config get prefix and add $(prefix)/bin to your shell profile, or reinstall with npm i -g github:sneaky-hippo/kolmogorov-stack after fixing the prefix.

Node version mismatch

kolm requires Node 20 or newer. Check with node -v. The CLI uses top-level await and Web Crypto APIs that older versions do not expose; you will see SyntaxError or ReferenceError: crypto is not defined if you run it on Node 18 or below. Upgrade via your version manager (nvm, fnm, volta).

EACCES on npm install -g

Do not use sudo. Set a user-owned prefix: npm config set prefix ~/.npm-global, then add ~/.npm-global/bin to PATH. Reinstall.

npm install errors during postinstall

The CLI itself is pure JS and has no native build step. If npm i -g fails on a transitive dependency, the usual cause is a stale npm cache or a partial download. Clear with npm cache clean --force and reinstall. Behind a corporate registry, point npm at it explicitly: npm config set registry https://your-registry.example.com.

Python 3.11+ required for the trainer

The local trainer at apps/trainer (used by kolm tune and by the local-cuda, local-mps, and local-mlx compute backends) requires Python 3.11 or newer. Check with python3 --version. Install via pyenv or your distribution's python3.11 package. The recipe-mode compile path does not need Python at all; this only affects adapter training.

Missing GPU drivers

local-cuda needs a working CUDA 12 driver and PyTorch built against it. local-rocm needs ROCm 6 on a supported AMD GPU. local-mps and local-mlx need macOS 13 or newer on Apple Silicon. Run kolm compute detect; backends that fail driver discovery show x with the reason. CPU-only machines fall back to local-cpu automatically, but training a real adapter on CPU is slow; see the compute backends section for cloud alternatives.

API key & auth issues

KOLM_API_KEY env var not set

The CLI reads the key from ~/.kolm/config.json first, then falls back to $KOLM_API_KEY. In CI you usually want the env var: export KOLM_API_KEY=ks_... before any kolm compile step. Persist locally with kolm login.

401 Unauthorized on /v1/jobs

The bearer token is missing, wrong, or stale. Inspect what the CLI has: cat ~/.kolm/config.json. Hit the auth probe directly:

curl -i https://kolm.ai/v1/account \
  -H "authorization: Bearer $KOLM_API_KEY"

A 200 confirms the key is live. A 401 with {"error":"invalid_key"} means it was rotated or revoked. Delete ~/.kolm/config.json and run kolm login again.

Token expired or rotated

API keys do not auto-expire, but they can be rotated from /account or POST /v1/account/rotate-key. After rotation the old key 401s immediately. Save the new value, delete the stale entry from ~/.kolm/config.json, and run kolm login again.

OAuth callback hangs at /signin

The Google or GitHub flow writes a kolm_session cookie and redirects to /account. If the redirect loops, the most common causes are (1) cookies blocked for kolm.ai, (2) a browser extension stripping the state param, or (3) clock skew over five minutes on your local machine. Clear cookies for the domain, retry in a private window, and re-check date -u against an NTP source.

402 payment required

Free tier ran out of compile minutes for the month, or the plan was downgraded. Check /account. Upgrade at /pricing or wait for the next reset.

429 rate limited on signup

Signup is gated to 10 attempts per IP per 24 hours. If you are testing from a shared egress, switch IPs or wait. This is not a bug; see /faq.

Compile failures

K-score below the 0.85 ship gate

The compile log names the failing sub-score (accuracy, size, latency, cost, coverage). Walk down the list:

Accuracy too low. Bump pair_count in your recipe spec to 24 or 32 to give the verifier more examples. Remove contradictions in your examples/ directory (two inputs that look the same with different expected outputs will crater accuracy). If accuracy plateaus, the task is probably too broad; tighten it to one judgable output.

Size or latency too high. Switch to a smaller base model. On free tier only Phi-3-mini and Qwen2.5-3B fit; Pro adds Llama-3.1-8B and Hermes-3-14B. Re-run kolm compile --base phi-3-mini.

Coverage too low. Your eval set is starved. Lower holdout_ratio from 0.30 to 0.20 so more pairs go into training, or add more examples.

To ship a prototype anyway, override the gate: kolm compile --gate 0.70. Do not do this for production.

Invalid corpus format: JSONL line errors

Capture files and example sets are line-delimited JSON. Each line must be a complete JSON object. The compile log prints JSONL parse error at line N with the offending line. Common causes: trailing commas, multi-line objects, BOM at file start. Validate with node -e "require('fs').readFileSync('examples.jsonl','utf8').split('\n').filter(Boolean).forEach((l,i)=>{try{JSON.parse(l)}catch(e){console.error('line',i+1,e.message)}})".

Out of memory on the trainer

The distill phase loads the base model into RAM (or VRAM). If you get RuntimeError: CUDA out of memory or the process is killed by OOM, try one of:

1. Drop target_size from 7B to 3B in the recipe spec.
2. Switch to a different compute backend: kolm compute pick --min-vram 40 will surface a cloud GPU with enough headroom.
3. Lower the batch size by setting KOLM_TRAIN_BATCH=2 (default is 8). Slower, but fits in less memory.

Verifier synthesis failed

Your task description does not name a verifiable property. Either add 5 to 10 paired examples that the verifier can learn from, or tighten the task to one judgable output: classification, extraction, structured output. Free-form generation needs a regex or JSON-schema check, or a paired-LM judge in the verifier slot.

Teacher API returned 4xx

Your frontier key is invalid or out of quota. kolm uses your own Anthropic / OpenAI / Google key for k-sample inference and stops if the provider rejects. Check the key against https://api.anthropic.com/v1/messages directly, then re-run.

Compile job stuck at "k-sampling"

The frontier provider is slow or rate-limiting. The job will resume; do not cancel. If it has been over 30 minutes, interrupt the running process and retry with --k 4 instead of the default 8.

Network & firewall

Corporate proxy blocking outbound

Set HTTPS_PROXY and HTTP_PROXY before any kolm command. The Node SDK and the Python trainer both honor them.

export HTTPS_PROXY=http://proxy.corp.example.com:3128
export HTTP_PROXY=http://proxy.corp.example.com:3128
export NO_PROXY=localhost,127.0.0.1

If your proxy does TLS interception, also point Node at the corporate CA: export NODE_EXTRA_CA_CERTS=/etc/ssl/corp-bundle.pem.

Airgap mode

For a permanent local-only setup, run kolm airgap enable. The CLI writes ~/.kolm/airgap.env with the four offline switches, source it from your shell rc or container entrypoint. Confirm with kolm airgap status and kolm airgap verify.

kolm airgap enable
source ~/.kolm/airgap.env
kolm airgap status

KOLM_AIRGAP=1 blocks every outbound the CLI knows about, including kolm publish. The compiled .kolm already lives at ~/.kolm/artifacts/. Copy the file to your sneakernet medium and move it manually. Full reference at /airgap.

Hugging Face offline mode

The trainer fetches base weights from Hugging Face on first use. To pre-cache them on a connected machine and run training offline, set HF_HUB_OFFLINE=1 and TRANSFORMERS_OFFLINE=1 after the cache is populated. kolm airgap enable sets both for you. Pre-cache with huggingface-cli download <org/repo> on the online machine, then copy ~/.cache/huggingface to the airgapped box.

Model cache location

The trainer's model cache lives at $HF_HOME if set, otherwise ~/.cache/huggingface on Linux/macOS and %LOCALAPPDATA%\huggingface on Windows. The compiled .kolm artifacts cache at ~/.kolm/artifacts/. Both are safe to delete; the CLI re-fetches what it needs on the next run.

Compute backend errors

kolm compute detect: "missing env var"

Cloud backends are listed as unavailable until their provider token is set. The detect output tells you which env var is missing for each backend:

kolm compute detect
  x modal           KOLM_MODAL_TOKEN not set
  x runpod          KOLM_RUNPOD_TOKEN not set
  x together        KOLM_TOGETHER_TOKEN not set
  ...

Set the token and re-run. kolm compute info <name> prints the full provider URL where you sign up for each. For a local-only pick, run kolm compute pick --airgap; it restricts the picker to on-box backends.

kolm compute test <name> exits 2

Exit code 2 means the smoke test failed. The output includes a reason field with the specific cause (missing driver, expired token, network unreachable, provider quota exhausted). Re-run with --json for machine-readable output. For cloud backends, a 401 from the provider almost always means a stale or wrong token; rotate it and retry.

CUDA not available, falls back to local-cpu

If you expected GPU training and the picker landed on local-cpu, run kolm compute pick to see why. The picker scores each available backend on S = 0.35*avail + 0.20*cost_inv + 0.15*lat_inv + 0.15*repro + 0.15*perf and prints the winning reason. Common causes: CUDA driver missing, NVIDIA GPU not visible to the user process (try nvidia-smi), or PyTorch built without CUDA support (python3 -c "import torch; print(torch.cuda.is_available())").

To force a cloud GPU instead, set the provider token and pin the backend: kolm compute use modal. Full backend list at /docs.

remote-ssh backend cannot connect

The remote-ssh backend executes the trainer on a box you own over SSH. Set KOLM_REMOTE_SSH_HOST and KOLM_REMOTE_SSH_USER, and make sure ssh user@host works without a password prompt first (key-based auth). The smoke test runs uname -a remotely; if that fails the backend test will fail with the same SSH error.

Receipt & signature verification

Receipt sha mismatch

The artifact was modified after the receipt was issued. The usual cause is that someone re-archived the .kolm zip (which re-orders entries and changes file mtimes) or edited the manifest by hand. The signed bytes are the original archive. Re-fetch from the publisher's distribution channel. Compare hashes with kolm inspect <file> and the publisher's signed hash; they must match exactly.

HMAC chain validation failed

The receipt chain ties manifest hash, layer hashes, and the per-run input/output hashes together with an HMAC-SHA256. A chain break means one of those was edited after signing. kolm inspect --explain shows which ring failed: ring 1 (model weights) means a tampered weights file, ring 4 (manifest) means metadata edits. Do not run an artifact whose chain is broken.

CID lookup 404

The artifact's CID (cidv1:sha256:<64-hex>) is not registered with the public registry. Either it was published privately, the CID was mistyped, or the publisher never ran kolm publish. Verify locally with kolm inspect <file>; the CID is computed deterministically from the manifest hashes and does not need the registry to be present.

Signature key not recognized

The artifact was signed by a key not in your trust store at ~/.kolm/trust/. Add the publisher's public key file to that directory after confirming it out-of-band, then re-run kolm inspect. For self-published artifacts, the signing key lives in ~/.kolm/config.json and is trusted by default.

Run & serve

kolm run: model file missing

The artifact zip is corrupt or truncated. unzip -l <file.kolm> should list at least the five core entries (spec.json, policy.json, verifier.js, recall/, receipt.json). Re-download from the original source.

kolm run: prompt exceeds context window

The base model has a fixed window. Trim the input, switch to a base with longer context at compile time (--base), or use the recipe-mode shortcut that routes long prompts through recall.

kolm serve --mcp: client cannot find the server

kolm serve --mcp speaks line-delimited JSON-RPC over stdio, so the client must spawn it as a subprocess. Run kolm install <harness> --apply for claude-code, cursor, continue, or cline to wire the command into the right config file. mDNS announce and .well-known/mcp.json are roadmap, not shipped today.

kolm serve --http: port in use

The optional HTTP transport binds 127.0.0.1:8765. Change with --http-port 8766. To find what is on 8765: lsof -iTCP:8765 -sTCP:LISTEN. The default stdio transport has no port.

Still stuck?

Run kolm doctor first. It checks the CLI version, the Node version, the key in ~/.kolm/config.json, the configured compute backend, and (when --http is on) the MCP port. If that comes back clean and your case is not on this page, send the output along with the failing command.
