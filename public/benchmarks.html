Methodology: Opus-4.7, Anthropic provider pinned (no router noise), seed=42, official swebench 4.1.0 evaluator. Lite split, n=150 sampled at random from the 300-task set. Same scaffold for both arms; only the kolm-compiled context inject differs. Full transcripts and reproducer linked below.

What is being measured

SWE-bench Lite is a coding benchmark from Princeton that asks a model to fix a real bug in a real Python repository given (a) the repo state and (b) a failing test. Strict scoring requires the produced patch to apply cleanly and the failing test to pass.

The "Opus alone" arm runs Opus-4.7 with the standard scaffold and the official problem statement. The "Opus + kolm" arm uses the same model and scaffold, but a .kolm-compiled prompt assembler injects: (1) the body of the failing test, (2) a canonical-diff-format hint per repository, (3) a "do not repeat" frame derived from prior failures on the same task family. No additional reasoning capacity is added.

Reproducer

The full pipeline is open. You bring an Anthropic key. The harness uses swebench==4.1.0 in the standard Docker evaluator.

# clone reproducer
git clone https://github.com/sneaky-hippo/kolmogorov-stack
cd kolmogorov-stack/bench

# run the cold arm (~$30 of Opus tokens, ~6h)
ANTHROPIC_API_KEY=sk-... python swe_lite_run.py \
  --arm cold --n 150 --seed 42 --provider anthropic

# run the kolm arm
ANTHROPIC_API_KEY=sk-... python swe_lite_run.py \
  --arm kolm --n 150 --seed 42 --provider anthropic

# official evaluator
python -m swebench.harness.run_evaluation --predictions out/kolm.json
strict pass: 45.33% (68/150)

Cost note: full run is ~$60 of Opus tokens (both arms). Re-runs at smaller n are cheap; we set n=50 originally before discovering result was unstable below n=100.

The +6pp run was published as a non-significant result. After test-body inject and Anthropic-pinning, the same n=150 lifted to +15.33pp at p<0.05. We did not delete the earlier run.

What kolm actually injects

No frontier-prompt-engineering trick. The inject is mechanical:

// 1. failing test body, byte-exact from the repo
TEST_BODY: {literal contents of the failing test}

// 2. canonical diff format pinned per-repo
OUTPUT_FORMAT: unified diff, paths relative to repo root

// 3. prior-failure frame from project recall index
DO NOT REPEAT: {one-line summary of any prior pass-1 attempt
                 on the same task family that failed}

// 4. small reduction prompt
EMIT only the diff. No prose around it.

This is a deterministic prompt assembler. It is not a thinking trick. The lift comes from giving the model the test body (which the official problem statement does not include) and from removing the failure mode of producing prose around the diff.

What we are not claiming

kolm does not make Opus better at coding in general. It moves a specific class of failures (apply-error, format mismatch, repeated wrong-fix) into pass on a specific benchmark. Generalization across tasks requires a per-task compile step. That is the product.

kolm does not beat the SWE-bench Verified leaderboard. It improves a base scaffold by +15.33pp on Lite. The leaderboard runs much heavier scaffolds; this comparison is apples-to-apples on the same scaffold.

kolm did not always win. See the table above.
