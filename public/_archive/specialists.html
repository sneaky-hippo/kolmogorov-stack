For the 20% a Recipe can't catch — a Specialist.

Recipes replace the boring 80% of your AI calls with deterministic JS. The 20% that need real fuzzy reasoning get a Specialist: a small fine-tuned LoRA, trained on the synthetic labels your Recipe just generated, that you own and run locally. Free per call. Forever.

The compounding loop — in plain English

Each layer feeds the next. The same 8 examples become a function, then synthetic data, then a model. You never write a training script.

01 · PASTE

4–8 examples

The same labeled examples that build a Recipe. No corpus, no training set — just the inputs and the outputs you'd accept.

02 · SYNTHESIZE

Recipe runs first

A deterministic JS classifier compiles in 3 seconds. It's the labeler-of-labelers — the thing that produces training data on demand.

03 · AUTO-LABEL

100k rows, free

Recipe runs over a public corpus (HuggingFace, your data, or a URL) and labels every row at $0/call. Now you have a real training set.

04 · TRAIN

1.5–3B LoRA

We fine-tune a Qwen/Llama/SmolLM LoRA on the synthetic labels. You download the .safetensors and run it locally, free, forever.

▲ The training stack is the same LoRA pipeline we already shipped for personal-LoRA training at REM Labs (CPU + GPU bf16, full chat-template flow). Specialists is that pipeline pointed at synthetic labels instead of personal data.

When the autopilot picks a Specialist over a Recipe

The orchestrator decides per cluster. Both paths use the same examples. Most teams end up with both layers running.

RECIPE WINS

Sharp, structured tasks

  • Spam / phishing flags — clear lexical patterns
  • Email / phone / URL extraction — regexes already win
  • Format / schema validation — deterministic by definition
  • Language detection — small lookup beats a model
  • Boolean policy gates — "does this string contain X"
SPECIALIST WINS

Fuzzy, semantic tasks

  • Sentiment beyond positive/negative (sarcasm, tone)
  • Intent routing across overlapping categories
  • Domain-specific summarization (legal, medical)
  • Open-ended classification (50+ categories, fuzzy boundaries)
  • Anything where Recipe fails the ≥95% verification gate

What the trainer actually does

No black box. The training stack is open-spec: you can audit it, reproduce it, or run it yourself.

# under the hood, when you click "Train Specialist"

$ recipe specialists train cpt_classify_issue \
    --base   Qwen/Qwen3-1.5B \
    --corpus huggingface:amazon_reviews_multi \
    --rank   16 \
    --epochs 3

[01/04] label-corpus      100,000 rows  ·  recipe runtime  ·  35µs/row  ·  3.5s total
[02/04] prepare-dataset   98,341 high-confidence rows kept (1,659 ambiguous filtered)
[03/04] lora-train        Qwen3-1.5B + LoRA r=16  ·  3 epochs  ·  ~42 min on 1×A10G
[04/04] eval & package    val acc 0.946 · held-out 0.927 · 87 MB safetensors

The API (preview)

# 1. you already have a Recipe in your registry
RECIPE_ID="cpt_classify_issue_type_…"

# 2. point at a corpus and train a Specialist
curl -X POST https://kolmogorov-stack-production.up.railway.app/v1/specialists/train \
  -H "Authorization: Bearer $KS" -H "Content-Type: application/json" \
  -d '{
    "name": "issue-type-specialist",
    "recipe_id": "'$RECIPE_ID'",
    "corpus": { "type": "huggingface", "name": "amazon_reviews_multi", "split": "train", "limit": 100000 },
    "base_model": "Qwen/Qwen3-1.5B",
    "rank": 16
  }'

# → { specialist_id: "spc_…", status: "queued", est_minutes: 47 }

# 3. download the .safetensors when training is done
curl https://kolmogorov-stack-production.up.railway.app/v1/specialists/spc_…/weights \
  -H "Authorization: Bearer $KS" -o issue-type.safetensors

# 4. or hit the hosted endpoint
curl -X POST .../v1/specialists/spc_…/run \
  -H "Authorization: Bearer $KS" -H "Content-Type: application/json" \
  -d '{ "input": "deploy crashes since friday" }'
# → { output: "bug", latency_ms: 38, model: "Qwen3-1.5B+lora" }
