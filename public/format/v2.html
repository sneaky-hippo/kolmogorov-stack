kolm  /  format  /  .kolm v1 to v2 transition

The .kolm file is the portable artifact every other surface in kolm refers to.

Every other surface on this site — the verifier, the drift detector, the K-score calculator, the compare matrix, the receipt chain — operates on a single underlying object: the .kolm file. Format v1 was a JSON-only manifest that worked for pure-rule recipes but could not carry compiled binaries. Format v2 is a deterministic binary bundle (ZIP with epoch-0 mtimes and pinned compression) that holds the canonical manifest.json, the receipt chain, the signature cascade, and the compiled WASM, GGUF, Rust, and C payloads alongside one another. This page documents the bundle layout, the manifest top-level keys, backwards compatibility against v1 readers, and the opt-in migration path.

The v1 to v2 transition.

v1 was the pre-RS-1 layout: a single artifact.json blob with the manifest fields, the receipt chain, and the signatures inlined at the top level. It worked for pure JavaScript rule recipes because nothing else needed to ride in the artifact. v2 promoted the file to a deterministic ZIP container so the artifact can carry compiled binaries (WASM, GGUF, Rust, C) and a sub-directory of canonical slot entries (manifest.json, receipt.json, signatures) the verifier can stream individually instead of parsing one giant JSON blob.

concernformat v1format v2
container single artifact.json blob; the entire artifact is one JSON object on disk. deterministic ZIP bundle; canonical slot entries are individual files inside the archive (RS-1 v2.1 §3).
compiled binaries no · binaries cannot embed cleanly in JSON without base64 inflation. yes · native bytes ride alongside the manifest. Targets: WASM, GGUF, Rust, C.
receipt chain location inlined under top-level receipt key in the same JSON blob. separate entry at meta/receipt.json; the verifier streams it without parsing the manifest.
signature location inlined under top-level signature, signature_ed25519, signature_sigstore keys. aggregated into meta/signatures.json; the manifest itself still mirrors the canonical signature fields so RS-1 §3.1 stays stable.
forwards compatibility v1 readers fail loudly on v2 (hex magic-byte check); they cannot silently misread. v2 readers fail-open on v1 input with a format_version_old warn-level log entry; legacy artifacts continue to verify.

The v1 to v2 transition is additive at the bundle layer and stable at the manifest layer. Every canonical manifest field RS-1 §3.1 documents (schema, version, name, artifact_class, base_model, tier, k_score, compiled_at, signature, artifact_hash_input) has the same shape and meaning in both formats. What changed is the on-disk container around the manifest, not the contents the verifier validates.

The v2 bundle structure.

The v2 bundle is a deterministic ZIP with epoch-0 mtimes and store/deflate compression so byte-identical inputs produce byte-identical artifacts. The verifier never extracts to the filesystem; it streams entries by name. Mandatory slot entries live under meta/; recipe payload lives under recipes/<recipe_id>/; the input corpus and the eval pack live under seeds/ and evals/ respectively. Any class-dependent payload (model weights, native binaries, retrieval index) is namespaced beneath its owning recipe so a multi-recipe bundle stays unambiguous.

.kolm/
  meta/manifest.json          # canonical-JSON manifest, root of trust (RS-1 v2.1 §3.1)
  meta/receipt.json           # build-time receipt chain: spec → seeds → recipes → evals → export
  meta/signatures.json        # HMAC + Ed25519 + optional sigstore bundle (RS-1 §5.1)

  recipes/<recipe_id>/
    source.js                  # the compiled recipe source (with metadata)
    compiled/                  # present only when the recipe class admits native compilation
      target.wasm              # WebAssembly artifact (browser + server runtime)
      target.gguf              # quantized weights for distilled_model recipes
      target.rs.bin            # native Rust binary for compiled_rule recipes (RS-1 §7.5)
      target.c.bin             # native C binary for compiled_rule recipes (RS-1 §7.5)

  seeds/seeds.jsonl          # tenant-captured IO corpus (RS-1 §4)
  seeds/holdout.jsonl        # deterministic 80/20 split, never seen by the recipe at train time

  evals/eval_spec.json       # declarative eval pack (comparators, judge id, gate threshold)
  evals/eval_results.json    # scored results that produced the manifest k_score field
  • Class-dependent payloads (model.gguf, lora.bin, native .bin outputs, optional index.sqlite-vec for retrieval recipes) follow the RS-1 v2.1 §3 mandatory-vs-optional split. The bundle layout above re-namespaces them under recipes/<recipe_id>/compiled/ so multi-recipe bundles stay unambiguous; single-recipe bundles can flatten by convention but the verifier accepts both paths.
  • Pretokenize provenance (pretokenize/) and MoE composition provenance (moe/) extend the same pattern when the manifest opts in (RS-1 §7.3 + §7.4). They are sibling top-level entries inside the bundle, never nested under meta/.
  • The CID is cidv1:sha256: followed by hex(SHA-256(canonical-JSON(manifest.artifact_hash_input))) per RS-1 §3. The CID is computed over the canonical manifest input regardless of which bundle layout produced the bytes.

The manifest.json top-level schema.

The manifest is the root of trust. Every other entry in the bundle hashes into the manifest's artifact_hash_input; the receipt chain ends with the manifest hash; every signature in meta/signatures.json covers a canonical-JSON serialization of the manifest body. The top-level keys below match the RS-1 v2.1 §3.1 contract; the canonical-JSON rule is sorted keys recursively, no insignificant whitespace, UTF-8, integers as integers, so byte-identical input produces byte-identical output across the JavaScript, Python, and Rust reference implementations.

format_version selects the bundle layout ("v1" or "v2"); kolm_version is the toolchain that produced it. license records the SPDX identifier the artifact ships under. signature_alg encodes the cascade: 'sigstore+ed25519+hmac-sha256' (full), 'ed25519+hmac-sha256', or 'hmac-sha256' per RS-1 §5.1. artifact_hash_input is the canonical-hash bundle every signature covers; any tamper drifts the artifact hash and breaks every downstream signature including the Rekor anchor.

The full canonical key set lives in the RS-1 v2.1 spec at §3.1 Canonical manifest fields. The JSON Schema source-of-truth is published at kolm.ai/docs/manifest-v0.1.json and the verifier cross-checks every required key as part of the 24-check cascade.

Backwards compatibility · readers across the version boundary.

Every .kolm file leads with a four-byte magic header. v1 artifacts start with 0x7B 0x22 0x73 0x63 (the ASCII bytes for {"sc, the opening of a canonical JSON object whose first key is schema). v2 bundles start with the ZIP local file header magic 0x50 0x4B 0x03 0x04. Readers branch on the hex magic-byte check before allocating any further state.

A v2 reader on a v1 file detects the JSON magic, parses the artifact as a top-level JSON object, emits a format_version_old warn-level log entry, and continues verification using the v1 codepath. Verification passes when the v1 signatures cover the v1 canonical body. This is the fail-open branch.

A v1 reader on a v2 file detects the ZIP magic, recognizes the bytes are not a JSON document, and refuses to parse rather than risk silently treating raw archive bytes as a malformed manifest. The v1 reader prints format_version_unsupported · expected v1 JSON, saw ZIP local file header and exits non-zero. No silent corruption: a v1 reader knows it cannot read a v2 file.

The version namespace for the on-disk container (format_version) is independent of the RS-1 spec revision (the manifest version field, currently "2.0") and independent of the toolchain version (kolm_version). A v1 reader can verify a v1-format artifact built by a recent toolchain; a v2 reader can verify a v2-format artifact built by an older toolchain. The three version axes evolve on independent cadences.

Why a binary bundle, not just JSON.

Promoting the container from one JSON blob to a binary bundle was a deliberate tradeoff. JSON was simpler to inspect with a text editor; the binary bundle costs the verifier a streaming-archive dependency. Four reasons made the tradeoff worth taking.

  • Compiled binaries are bytes, not strings. A 15 KB native Rust binary (RS-1 §7.5) inflates to 20 KB of base64 inside JSON, plus the cost of double-decoding to verify bin_hash. A 500 MB GGUF model would balloon to roughly 670 MB of base64 and force the verifier to allocate the whole string before validating the first byte. Storing the binary as a raw archive entry preserves byte-exact identity without an encoding round-trip.
  • GGUF and WASM each can be 100 MB or more. Distilled student weights routinely cross the 100 MB threshold; large MoE bundles cross 500 MB. A single-object JSON parse forces the parser to materialize the whole document; a streaming-archive reader walks entries one at a time and validates each entry's SHA-256 against the manifest claim without holding the full payload in memory.
  • Streaming parse for large packs. The verifier hashes each entry in the bundle independently and compares against manifest.artifact_hash_input. Streaming the entries lets a verifier reject a tampered seeds.jsonl after reading 4 KB even when the bundle is 500 MB; in v1 the entire JSON would have to be loaded before seeds_hash could be cross-checked.
  • Byte-exact reproducibility for the receipt-chain integrity hash. The receipt chain ends with the artifact hash; the artifact hash is sensitive to the byte order entries appear in the bundle. The deterministic ZIP layout (epoch-0 mtimes, sorted entries, pinned compression) guarantees byte-for-byte reproducibility across builders. Two independent rebuilds from the same canonical inputs produce two artifacts whose SHA-256 matches; the verifier can rebuild from sources and compare the bytes without reading the manifest at all.

The CLI flow, end to end.

Four commands compose the format-aware build, inspect, verify, and downgrade flow. Items wrapped in an amber verify before ship pill name a flag or sub-flag that is on the roadmap but not yet wired in cli/kolm.js; the underlying verb already exists and the flag is the additive piece. The bundle layout itself is already the production default.

01 · compile

kolm compile

Builds a .kolm bundle from a spec. The --format=v2 flag verify before ship selects the bundle layout explicitly; v2 is the default when the flag is omitted and a v2-capable toolchain is detected.

$ kolm compile acme.spec.json
  spec → seeds → split → train
  → recipes → evals → signatures

  artifact: acme.kolm  (1.4 MB)
  format:   v2  (default)
  hash:     de78f0a1b234c567…
02 · inspect

kolm inspect

Reads the manifest and prints the canonical key set, the receipt chain, and the signature cascade. The --format-version flag verify before ship prints just the bundle layout discriminator (v1 or v2) so a CI step can branch on the bundle layout without parsing the manifest.

$ kolm inspect acme.kolm
  schema:         rs-1
  version:        2.0
  format_version: v2
  kolm_version:   0.1
  artifact_class: rule
  k_score:        0.917
03 · verify

kolm verify

Runs the 24-check verifier cascade. The --format-strict flag verify before ship refuses to fail-open on a v1 artifact; useful when a regulated tenant has standardized on v2 and wants the CI step to fail loud on any legacy bundle sneaking through.

$ kolm verify acme.kolm
  Bundle layout (v2)             pass
  Canonical manifest fields      pass
  Receipt chain integrity        pass
  HMAC + Ed25519 signature       pass
  All 24 checks                  pass
04 · export

kolm export

Re-serializes a bundle into an alternate target shape (binder PDF, tar.gz, etc). The --downgrade=v1 flag verify before ship emits a v1-format artifact for tenants whose verifier hasn't been upgraded; the downgrade is rejected if the source bundle contains any compiled binary that cannot round-trip into the v1 JSON-only layout.

$ kolm export acme.kolm --downgrade=v1
  source:    acme.kolm (v2, 1.4 MB)
  target:    acme-legacy.kolm (v1, 0.9 MB)
  warn:      compiled_targets dropped
             v1 cannot carry native bytes.

The base verbs (kolm compile, kolm inspect, kolm verify, kolm export) are all currently wired in cli/kolm.js. The format-aware flags above are the additive piece that lands with the W187 surface; until the corresponding wave ships, treat the amber-pilled flags as roadmap and use the default codepath (v2 in, v2 out) which the existing CLI already implements.

Migrating a v1 artifact to v2.

Existing v1 .kolm files remain valid forever. v2 readers fail-open on v1 input and the canonical manifest contract is stable across the version boundary, so a v1 artifact still verifies, still surfaces a K-score, still proves a receipt chain. Migration is opt-in: a tenant runs the migrate verb when they want the bundle benefits (streaming verify, compiled-binary support, downstream re-attestation against the latest toolchain).

$ kolm migrate acme-legacy.kolm --format-v2      # verify before ship
  source:    acme-legacy.kolm (v1, 0.9 MB)
  target:    acme-legacy-v2.kolm (v2, 0.9 MB)

  receipt chain re-anchored against v2 layout
  signatures re-emitted by the local signing key
  original v1 artifact left untouched on disk

  migration is additive; the v1 file still verifies.

The kolm migrate verb is not yet wired in cli/kolm.js (verify before ship); the migration path above describes the planned shape. Until the verb ships, a tenant who wants v2 bytes can recompile from the original spec with the current default codepath, which emits v2 directly.
