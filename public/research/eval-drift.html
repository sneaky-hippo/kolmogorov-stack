Eval drift detection

A frozen evaluation set is only useful while it stays out of the training distribution. As more data is captured, mined, or synthesized, the holdout can leak in by accident: near-duplicates, paraphrased prompts, or the same task in a new template. Eval drift detection is the set of statistical tests that catch this before the gate publishes a number that has no meaning.

What it is

Three failure modes are conflated under the heading "drift". The first is contamination: a training-corpus row that is byte-identical or near-duplicate of a holdout row, usually from a sloppy synthesis pipeline. The second is distributional overlap: the holdout was sampled from the same generator as the train set, so the gate measures interpolation rather than generalization. The third is covariate shift: the production prompt distribution has moved, so even an honest gate reports a number that does not predict downstream behavior. The literature covers all three: Carlini et al. (2022) on memorization, Gao et al. (2021) on the Pile's contamination audits, Magar and Schwartz (2022) on what train-test overlap actually buys, and the classical Sugiyama et al. (2007) framework for covariate-shift detection via density-ratio estimation.

How kolm uses it

Three checks are wired into the gate stage in v0.2.6. First, byte-level near-duplicate detection on every holdout row against the training corpus using MinHash with a Jaccard threshold of 0.85. Second, n-gram overlap (n=13) flagged at the per-prompt level, mirroring the OpenAI evals contamination heuristic. Third, an embedding-space KL estimate between the training and holdout embedding distributions; if the estimate falls below a configurable threshold the gate refuses to run and emits a holdout_drift error asking for fresh seeds. The density-ratio formulation (Sugiyama) is on the v0.3 roadmap; for now the KL surrogate is sufficient for the typical 8k-row holdout sizes kolm buyers ship. The receipt records the three drift statistics so the auditor can re-compute them.
