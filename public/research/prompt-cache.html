Prompt caching

Two unrelated ideas share the name "prompt cache". One reuses the model's KV state when consecutive requests share a prefix; the provider charges roughly ten percent of the normal input rate for cached tokens. The other matches semantically equivalent prompts in a vector index so the model never runs at all. Both matter to a capture pipeline that hits the same teacher many times.

What it is

KV-cache prefix sharing is a server-side optimization. When two requests start with the same tokens (a system prompt, a long document, a few-shot block), the attention key-value tensors for that prefix can be computed once and reused. vLLM's PagedAttention (Kwon et al. 2023) and SGLang's RadixAttention (Zheng et al. 2024) generalized the idea across requests at the engine level. Anthropic, OpenAI, and DeepSeek expose it as a per-message cache_control hint with a published discount (cached input is ~10 percent of full input). Semantic prompt caching is a client-side optimization: embed the prompt, look it up in a vector index of past prompts, and if the cosine distance to the nearest is below a threshold, return the cached response. GPTCache (Bang 2023) is the reference open-source implementation. Semantic caches trade exactness for hit rate; on production traces with repeated questions, hit rates of 20 to 60 percent are common, but every false positive returns the wrong answer.

How kolm uses it

KV-prefix caching is wired into the distillation stage. The synthesis pipeline batches teacher calls by stable prompt prefix (system prompt + recipe-fixed few-shot block) and emits the cache-control hint, so the typical distillation run pays the ten-percent rate for the prefix on every call after the first. The receipt records cache_hit_rate and cache_savings_usd so the buyer can verify the bill. Semantic prompt caching is not implemented in kolm v0.2.6; it conflicts with the capture-loop's job of recording every request as a training example. Buyers who want semantic caching for runtime cost reduction can layer GPTCache or LiteLLM in front of the kolm runtime; the recipe is documented in /docs/optimizer. A first-party semantic cache that respects per-tenant isolation and emits cache-hit events to the capture log is planned for v0.3.
