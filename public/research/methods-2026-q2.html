This is a synthesis of the 2026-Q2 frontier in post-training methods, framed against what the existing kolm pipeline already does. The intent is to make Kolm’s grounding legible: every claim below is anchored to a published paper, a vendor blog, or a >5k-star OSS repo, and the closing section separates what the shipped src/tune.js, src/synthesis.js, src/assistant.js, and src/recipe-class.js modules currently do from three modules (distill-onpolicy.js, distill-preference.js, spec-decode.js) that are roadmap-only and not yet shipped. Items the underlying research dump marked as needing a second-source confirmation carry a verify before ship tag below.

1. Distillation methods — state of the art.

1.1 Pure knowledge distillation (logit / hidden-state matching).

The classical Hinton-style soft-target distillation (minimise KL between teacher and student logits at temperature T) remains the workhorse, but 2025 work has shifted toward sparse, adaptive, and feature-level variants.

  • Sparse Logit Sampling (Bhardwaj et al., 2025, arXiv:2503.16870). Importance-sampling-based sparsification gives unbiased estimates of the teacher distribution and drops logit-cache storage by ~10× without measurable quality loss. Teacher logits can be precomputed once and reused across student runs.
  • AdaKD — Token-Adaptive Knowledge Distillation (arXiv:2510.11615). Loss-driven adaptive token focusing concentrates distillation pressure on the small tail of hard tokens that carry the signal.
  • Flex-KD (arXiv:2507.10155). Distils between teacher and student with differing hidden sizes; final-layer-only distillation suffices, dodging CKA’s all-layer requirement.
  • Probe-KD. Train an MLP probe on teacher hidden states across all layers, then use the probe’s soft predictions to supervise the student via KL. Cheaper than full feature distillation.
  • DistilBERT / TinyBERT logit + hidden-state legacy combination remains the standard baseline in 2025 surveys (arXiv:2503.12067, arXiv:2504.14772).

1.2 LoRA fine-tuning on teacher outputs — the kolm default path.

Teacher (frontier API or large open-weight) generates (prompt, response) pairs from the train split; a small base is LoRA-fine-tuned on those pairs; the adapter is quantised and exported. The seminal paper is QLoRA (Dettmers et al., NeurIPS 2023, arXiv:2305.14314) — 4-bit NF4 base + 16-bit LoRA adapters lets a 65B model fine-tune on a single 48 GB GPU without loss vs. full 16-bit fine-tuning.

The current best-practice stack pairs QLoRA with the Unsloth runtime (github.com/unslothai/unsloth, >25k stars), which delivers ~2× faster training and ~70 % less VRAM than vanilla PEFT through hand-written Triton kernels. Thinking Machines Lab’s “LoRA Without Regret” blog (Oct 2025) showed empirically that LoRA matches full fine-tuning on reasoning/math benchmarks when (i) rank is sufficient for the task complexity and (ii) the dataset is in-distribution for the base.

1.3 Preference distillation: DPO / SimPO / KTO / SLiC / ORPO.

A 2025 post-training stack typically chains SFT → preference step rather than picking one preference method.

  • DPO (Rafailov et al., 2023, arXiv:2305.18290) — single logistic loss over (chosen, rejected) replaces PPO + reward model. Universal baseline; OpenAI, Microsoft Azure, and Together all expose DPO endpoints.
  • SimPO (Meng et al., NeurIPS 2024, arXiv:2405.14734). Removes the reference log-ratio term, length-normalises the implicit reward, and outperforms DPO by up to 6.4 pts on AlpacaEval 2 and 7.5 pts on Arena-Hard. princeton-nlp/SimPO >1.5k stars.
  • KTO — Kahneman-Tversky Optimization (Ethayarajh et al., 2024, arXiv:2402.01306). Works on binary good/bad labels instead of paired preferences, so it consumes thumb-up/thumb-down logs directly.
  • SLiC (Zhao et al., 2023). Margin-based ranking loss; superseded in 2025 benchmarks.
  • ORPO — Odds-Ratio Preference Optimization (Hong et al., 2024, arXiv:2403.07691). Fuses SFT and preference in one pass; no reference model.
  • TVKD — Teacher Value-based KD (Kwon et al., 2025, arXiv:2509.16965). Injects soft reward labels from a teacher’s value function as an auxiliary term on top of DPO.

Modern consensus: SimPO for stability under noisy labels, ORPO when you can fuse SFT, KTO when you only have binary signal, DPO as the universal default polish step.

1.4 Reasoning distillation (Quiet-STaR, CoT, agent distillation).

  • Quiet-STaR (Zelikman et al., 2024, arXiv:2403.09629, ezelikman/quiet-star). LMs learn to generate token-by-token rationales that improve next-token prediction. After Quiet-STaR continued-pretraining, GSM8K rose 5.9 → 10.9 and CommonsenseQA 36.3 → 47.2 zero-shot.
  • Fast Quiet-STaR (EMNLP 2025 Findings). Same inference cost as base, but stronger downstream — thinking without thought tokens.
  • Agent Distillation (NeurIPS 2025 poster, Distilling LLM Agent into Small Models with Retrieval and Code Tools). 0.5B / 1.5B / 3B sLMs match next-tier 1.5B / 3B / 7B CoT-distilled baselines when distilled with the full agentic loop (retrieval + tools), not just rationales.
  • Chain-of-Thought Curriculum Distillation (ISAIE 2025). Step-by-step CoT supervision ordered by difficulty.

1.5 Speculative decoding distillation (inference-time speedup).

  • Medusa (Cai et al., 2024). Multiple LM heads predict tokens 1..k in parallel; base unchanged. Each head is essentially a single-layer distilled predictor.
  • EAGLE / EAGLE-2 / EAGLE-3 (Li et al., arXiv:2401.15077 and follow-ups). One-layer transformer draft model that concatenates input embedding with feature vectors and reuses the target LM head; EAGLE-3 fuses hidden states from multiple intermediate layers. 2–6× inference speedup with no accuracy loss; KL between target and draft is the standard objective. EAGLE-3 currently leads MT-Bench / HumanEval / GSM8K for spec-decoding on production-class models.
  • Variational Speculative Decoding (2025) verify before ship. Reframes draft training from token-likelihood to sequence-acceptance; reports beating EAGLE-3. Single recent preprint; not promoted as a kolm claim.

1.6 On-policy distillation (the late-2025 direction).

Thinking Machines Lab’s On-Policy Distillation (Oct 2025) is the most-cited new direction in late-2025 distillation work. Student samples its own trajectories on-policy, and the teacher provides dense per-token reverse-KL supervision on those trajectories. Information-theoretically distillation teaches O(N) bits per episode versus RL’s O(1), so on-policy distillation gets RL’s exposure-bias mitigation at a fraction of the data cost. Reverse KL is mode-seeking — it locks onto the teacher’s specific behaviour rather than smearing across distributions. A Black-Box On-Policy Distillation adaptation from Microsoft is at arXiv:2511.10643.

1.7 Mixed objectives — the dominant 2025 recipe.

Empirically dominant pipeline in 2025: SFT on teacher-generated demonstrations → DPO/SimPO on a preference set sampled from the same teacher. This is the recipe behind Tulu 3, SmolLM2-Instruct, and Llama-3.2-Instruct. Doing DPO without an SFT warm-up usually under-performs; OpenAI’s DPO cookbook explicitly recommends SFT-first.

2. Sample efficiency — how much data is enough?

2.1 LoRA on a small specialised task.

  • 500–2,000 well-crafted pairs is the modal recommendation across 2025 practitioner guides (Databricks, Truefoundry, Unsloth docs). Quality and formatting consistency dominate raw count.
  • LIMA (Zhou et al., 2023, arXiv:2305.11206). Demonstrated that 1,000 curated SFT examples on Llama-1-65B produced responses preferred to or equivalent to GPT-4 in 43 % of human-evaluated pairs, and 58 % vs. Bard. Authoritative anchor for the “1k is often enough” claim.
  • For sharply scoped tasks (regex-like normalisers, classifier heads, single-format extractors), 100–500 pairs often suffice; sub-100 examples enter few-shot territory and should be served via in-context prompting rather than fine-tuning.

2.2 Full SFT for a general chat assistant.

  • OpenHermes 2.5 (Teknium, 2024): ~1M GPT-4-generated multi-turn examples. Anchor for a “well-rounded assistant from scratch” size.
  • UltraChat (Ding et al., THUNLP, thunlp/UltraChat): ~1.5M multi-round dialogues.
  • Tulu 3 SFT mix (Allen AI, 2025, allenai/tulu-3-datasets): ~939k examples drawing from WildChat, Persona-IF, etc.
  • Practical floor for a generalist assistant base is ~100k–1M instructions; below ~50k, expect noticeable persona/format brittleness.

2.3 Preference data for DPO / SimPO.

  • UltraFeedback (Cui et al., 2023, OpenBMB/UltraFeedback): ~64k prompts with 4-aspect preference annotations across UltraChat / ShareGPT / Evol-Instruct / TruthfulQA / FalseQA / FLAN. De-facto open baseline.
  • Tulu 3 preference mix: on-policy completions generated by the model being trained; significantly outperforms UltraFeedback-alone, especially at 70B scale (per Tulu 3 paper, Allen AI 2025).
  • OpenAI’s official DPO cookbook walks through a 500-pair demonstration — enough for narrow behavioural shaping (response-format preferences, single-axis tone changes). For general alignment expect 10k–60k pairs.

2.4 Reasoning / CoT distillation.

  • Agent Distillation (NeurIPS 2025): a 0.5B sLM closes a 1B–7B gap with ~10k–50k rationale-augmented trajectories from a stronger teacher.
  • Quiet-STaR’s continued pretraining used OpenWebMath + web text at the scale of millions of tokens — the rationale-pretraining regime, not the few-thousand-pair fine-tune regime.

5. Training-data creation — reputable open methods.

5.1 Self-Instruct → Stanford Alpaca → Evol-Instruct.

  • Self-Instruct (Wang et al., 2022/2023, arXiv:2212.10560, yizhongw/self-instruct, >4k stars). Bootstrap from ~175 hand-written seed instructions → N-shot prompt (N ≈ 8) the LLM to synthesise new instructions, filter for diversity (Rouge-L < 0.7) and quality, iterate. Stanford Alpaca scaled this recipe to 52k instructions via text-davinci-003.
  • WizardLM / Evol-Instruct (Xu et al., ICLR 2024, arXiv:2304.12244, nlpxucan/WizardLM, >8k stars). Iteratively evolves instructions via five In-depth ops (add constraints / deepen / concretize / increase reasoning steps / complicate input) plus In-breadth. WizardLM beat ChatGPT by +7.9 pts win-rate on hard prompts.
  • WizardCoder and WizardMath (ICLR 2024) apply the same evolution to coding and math domains.

5.2 Magpie — seed-free synthesis (ICLR 2025).

Magpie (Xu et al., arXiv:2406.08464, ICLR 2025, magpie-align/magpie). Exploits the chat template of an aligned LLM: feed the pre-fill template alone, and the aligned LLM hallucinates both a user query and a response. No seed instructions, no Evol-Instruct chain. Llama-3-8B-Base fine-tuned on Magpie data outperforms Evol-Instruct- and UltraChat-derived datasets and approaches Llama-3-8B-Instruct quality with a much smaller dataset. t-SNE shows Magpie covers Alpaca + Evol-Instruct + UltraChat combined.

5.3 Synthetic data via teacher API.

OpenAI / Anthropic / Google offer the obvious teacher endpoints. Pricing matters: Claude Opus 4.7 input $5 / output $25 per MTok; Sonnet 4.6 $3/$15; Haiku 4.5 $1/$5; batch processing is 50 % cheaper across all models; prompt caching cuts cached input by 90 %.

Practical “right number of seeds”:

  • Self-Instruct anchor: 175 seeds → 52k synthesised (Alpaca).
  • For a narrow kolm task, 20–100 seeds + Evol-Instruct or Magpie → 1k–10k pairs is the modal recipe.
  • For a generalist assistant, expect a teacher bill scaling with output tokens: 1M-token output run at Sonnet rates ≈ $15. Always use batch + caching.

Amazon Bedrock now exposes Claude 3 Haiku model distillation with Sonnet 3.5 as teacher; achieves Sonnet-like accuracy on specific tasks at Haiku price/speed.

5.4 Constitutional AI and refusal training.

Constitutional AI (Bai et al., Anthropic, 2022, arXiv:2212.08073). Use a written constitution to (a) generate critique prompts that produce revised model outputs, then (b) generate AI-feedback preference labels for RLHF / RLAIF.

Constitutional Classifiers and Constitutional Classifiers++ (Anthropic, Feb 2025 and follow-up). Train input/output classifiers on synthetic constitution-violation data. Deployment on Claude Sonnet 4.5 hit 0.05 % refusal rate on harmless queries — 87 % drop from the original. Reference: anthropic.com/research/constitutional-classifiers.

For kolm: a tenant-supplied constitution → synthetic refusal corpus → small classifier head is a credible add-on for the redactor companion modules.

5.5 Persona-driven generation.

Scaling Synthetic Data Creation with 1,000,000,000 Personas (Chan et al., Tencent AI Lab, 2024, arXiv:2406.20094, tencent-ailab/persona-hub). Persona Hub — 1B personas curated via Text-to-Persona (“who would read/write/like this?”) + Persona-to-Persona (interpersonal-relationship expansion). Inject one persona per generation prompt to dramatically diversify outputs. Applications: math/logic problems, instructions, knowledge-rich text, NPC dialogue, tool/function calls.

This is the cheapest way to diversify a small seed set without manual curation; combine with Magpie or Evol-Instruct to get topic coverage and difficulty range simultaneously.

5.6 Anchor open datasets (cite-able pedigree).

  • OpenHermes-2.5 — Teknium / Nous (~1M GPT-4 multi-turn).
  • UltraChat — THUNLP (1.5M multi-round).
  • UltraFeedback — OpenBMB (64k 4-aspect preference).
  • Tulu 3 mix — Allen AI (SFT + preference + safety + IF; ~939k SFT, on-policy DPO).
  • Magpie-Pro / Magpie-Air — Magpie-Align (seed-free; multiple sizes from 100k to 1M).

6. Natural language → recipe — state of the art.

6.1 Function calling / tool use for structured output.

Anthropic, OpenAI, Google, Mistral, Cohere, and most open Instruct models now support tool / function calling natively. Authoritative benchmark: Berkeley Function Calling Leaderboard (BFCL) V4 (ICML 2025, gorilla.cs.berkeley.edu/leaderboard.html). Real-world question-function-answer pairs across Python / Java / JavaScript / REST; AST-based scoring. As of mid-2026: Claude Opus 4.1 at 70.36 % (2nd), Claude Sonnet 4 at 70.29 % (3rd), GPT-5 at 59.22 % (7th).

For kolm nl "make me a recipe…" the cleanest path is: define a JSON schema for RecipeSpec (positives, negatives, output_spec, recipe_class, hints) → invoke teacher with tool-use forcing that schema → validate locally → hand to src/synthesis.js.

6.2 Constrained decoding.

  • XGrammar (mlc-ai/xgrammar). Default structured-generation backend for vLLM, SGLang, and TensorRT-LLM since March 2026. Under 40 µs/token, near-zero JSON overhead. verify before ship vendor docs updated frequently.
  • llguidance (Microsoft, guidance-ai/llguidance). Rust Earley parser at ~50 µs/token; OpenAI publicly credited llguidance for the foundational work underpinning their May-2025 Structured Outputs.
  • Outlines. Compiles JSON schemas to FSM index for O(1) valid-token lookup. JSONSchemaBench flagged complex-schema compile times of 40 s–10 min; best for simple, hand-curated schemas, weaker for arbitrary user input.
  • lm-format-enforcer (Gat, noamgat/lm-format-enforcer). Permits LLM to control whitespace, optional fields, field ordering — more permissive than rigid grammar enforcers.
  • JSON mode / response_format — OpenAI (Aug 2024), Anthropic (Nov 2025), Google, Mistral. The cheapest path when you only need valid JSON, not a specific schema.

For kolm: use vendor-native function-calling for teachers we can target (Anthropic / OpenAI), with XGrammar or llguidance as the local fallback when the user requested an open-weight teacher.

6.3 DSL synthesis (NL → executable program).

The “Code as Policies” / “ProgPrompt” line of work (2022–2023) demonstrated that LLMs can translate NL into executable DSL when given (i) DSL syntax in-context, (ii) ~5–15 input/output exemplars, (iii) system instructions. The 2024–2025 follow-ups — DSL Assistant (arXiv:2408.09766), Natural Language Commanding via Program Synthesis (Gandhi, Microsoft, arXiv:2306.03460), Towards Formal Verification of LLM-Generated Code from Natural Language Prompts (arXiv:2507.13290) — treat the DSL as a typed schema and the LLM as the parser. Recurring success recipe: keep the DSL tight, expose ~10 in-context examples, run a syntactic validator (the AST check kolm already has in src/synthesis.js), and bounce on first parse failure.

6.4 Verifier loops (self-debug, AlphaCodium).

AlphaCodium (Ridnik et al., Codium AI, Jan 2024, arXiv:2401.08500, Codium-ai/AlphaCodium). Two-phase flow:

  • Pre-processing — model summarises goal/inputs/outputs/rules as bullets, generates and ranks candidate solutions.
  • Iteration — generate candidate, execute against public + AI-generated tests, refine based on failure mode.

GPT-4 pass@5 rose from 19 % (single-prompt) → 44 % (AlphaCodium). Crucially, only ~100 LLM calls per problem vs. AlphaCode’s ~1M. The lesson kolm steals: synthesising tests is easier than synthesising correct code; use the verifier-rejection signal as the gradient. Maps cleanly onto the existing verifier loop in src/synthesis.js.

8. What ships in kolm today vs. roadmap.

The methods above are not all live in the kolm pipeline. The split below is read directly off the repository as of 2026-05-17. Files in the shipped column already implement the pattern named; files in the roadmap column do not yet exist in src/ and are not yet shipped.

shipped · src/tune.js

src/tune.js

Skeleton-LoRA → evolving-local-model pipeline. The trainer spawned by scripts/tune-step.py standardises on QLoRA via Unsloth (4-bit NF4 base + 16-bit LoRA adapters). Quantisation export defaults to GGUF Q5_K_M with llama.cpp imatrix calibration; AWQ-INT4 (vLLM / Marlin) offered as a server-target variant. PEFT-format adapter contract documented in kolm tune init.

shipped · src/synthesis.js

src/synthesis.js

Recipe-source generation, including teacher-emitted JS. AlphaCodium-style verifier-rejection loop with AST validation is live. Teacher-output generation slot for JSON-schema function-calling (Anthropic / OpenAI) and XGrammar / llguidance fallback for open-weight teachers. Magpie-style seed-free generation and Persona-Hub injection are prompt-engineering extensions that land here.

shipped · src/assistant.js

src/assistant.js

NL → intent → tenant action. The rule-based intent classifier (regex-based detectIntent()) is the default. The nl_to_recipe intent emits a RecipeSpec JSON via teacher function-calling, reached only when the rule-based classifier returns help, preserving the no-telemetry guarantee for existing intents.

shipped · src/recipe-class.js

src/recipe-class.js

The four-class taxonomy (rule, synthesized_rule, compiled_rule, distilled_model) and inferral rules. The data-volume rule-of-thumb table in §7 is surfaced via CLASS_DESCRIPTIONS so kolm recipe-classes and the docs web page can render it.

roadmap · not yet shipped

src/distill-onpolicy.js

Not yet shipped. Thinking Machines on-policy distillation: student samples its own trajectories, teacher returns log-probs, reverse-KL gradient computed locally. Requires a teacher endpoint that exposes log-probs (open-weight teachers via the local runtime; Anthropic / OpenAI do not currently). Sequences with src/tune.js so the trainer can switch between off-policy SFT and on-policy distill.

roadmap · not yet shipped

src/distill-preference.js

Not yet shipped. Implements DPO + SimPO as orchestration on top of the existing tune loop; reuses the K-score gate. KTO as the binary-feedback variant for thumbs-up / thumbs-down captures from ~/.kolm/tune/<artifact>/captures.jsonl.

roadmap · not yet shipped

src/spec-decode.js

Not yet shipped. Optional EAGLE-style speculative-decoding pair shipped alongside the artifact: draft head trained on the same teacher outputs, 2–6× latency win at inference. Pure inference optimisation; orthogonal to the training pipeline.

Deferred — not on the roadmap.

  • SpQR integration — niche; llama.cpp story unclear in 2025.
  • Variational Speculative Decoding — single recent preprint verify before ship.
  • Quiet-STaR rationale-pretraining — compute-prohibitive for tenant-scale runs; viable only for fleet-wide base updates.
