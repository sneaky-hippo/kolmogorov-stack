The K-score: a formal definition.

A composite gate is a calibration choice, not a universal claim. This whitepaper defines the K-score as it ships in RS-1, derives the weights from the failure distribution we have observed, ablates each axis, compares to LMSYS Arena and SWE-bench, and lays out the threat model. One number, five inputs, one gate at 0.85.

Abstract.

The K-score is a single scalar in [0, 1] that decides whether a compiled .kolm artifact ships. It is computed at compile time, embedded in the manifest, signed into the receipt body, and verifiable by any third party who can read the artifact. The score is a weighted aggregate of five normalized axes: Accuracy, Size, Latency, Cost, and coVerage. The default ship gate is 0.85; artifacts that score below 0.85 do not emit, are not signed, and produce a diagnostic JSON instead.

This whitepaper formalizes the score, derives the weights from the empirical compile failure distribution across the first 1,247 compiles in development, ablates each axis to show its load-bearing contribution, compares the K-score gate against three existing evaluation regimes (LMSYS Arena, SWE-bench, OpenAI Evals) on calibration semantics and reproducibility, lays out a threat model for adversarial inputs, and previews the v2 calibration roadmap including held-out accuracy R, fairness F, energy E, and drift Z.

The K-score is not a universal quality claim. It is the smallest budget constraint we are willing to assert at compile time, plus a signed audit trail that the constraint was honored. Quality across heterogeneous task distributions remains an open problem; the K-score does not solve it. It does close the gap between "we trained a model" and "we shipped an artifact that meets a known constraint."

Formal definition.

1. The composite

Let A, S, L, C, V each be normalized to [0, 1]. The K-score v1 (specification tag k-score-1) is defined as:

K = 0.40 · A
  + 0.15 · S
  + 0.15 · L
  + 0.15 · C
  + 0.15 · V

The composite is rounded to four decimal places and emitted as a IEEE 754 double. The weights sum to 1.0 by construction. The composite is convex: a perfect score on any single axis cannot compensate for a zero on another.

2. Accuracy A

Bare top-1 accuracy on declared eval positives. For a classification artifact, this is the fraction of held-out cases where the artifact emitted the labeled output. For a generative artifact, this is the fraction where an oracle judge (a stronger model or a pinned rubric) marks the output as semantically equivalent to the gold answer. The eval set is synthesized by the compile pipeline from the training data using held-out splitting; the held-out set is sealed before compile begins and is not visible to the recipe synthesizer.

For sub-population fairness, see axis F in v2 (section 8).

3. Size S

A monotonic decreasing function of artifact size in kilobytes:

S = max(0, min(1, 1 - log2(kb) / 30))

This places a 1 KB artifact at S = 1.0, a 1 MB artifact at S = 0.67, a 1 GB artifact at S = 0.33, and a 1 TB artifact at S = 0. The base-2 log gives roughly equal weight to each order-of-magnitude jump in size, which matches the rough engineering intuition that going from 1 MB to 10 MB matters as much as going from 1 GB to 10 GB.

4. Latency L

A monotonic decreasing function of p50 latency in microseconds:

L = 1 / (1 + lat_us / 100000)

A 10 ms response lands at L = 0.91; a 100 ms response at L = 0.50; a 1 s response at L = 0.09. The 100 ms knee is the conversational interface threshold: below 100 ms the response feels instantaneous, above 100 ms the user starts to notice latency. Latency is measured on a fixed reference machine declared in the recipe; relative latency on the verifier's machine may differ.

5. Cost C

A monotonic decreasing function of per-call cost in USD:

C = 1 / (1 + cost_usd · 1000)

A free local artifact lands at C = 1.0. A $0.001 per call frontier proxy artifact lands at C = 0.50. A $0.01 per call artifact lands at C = 0.09. The $0.001 knee matches the per-call price that consumer-grade SaaS apps can amortize without margin erosion.

6. Coverage V

Fraction of declared eval case classes that the artifact actually exercised:

V = exercised_classes / declared_classes

The coverage axis exists to catch a specific failure mode: an artifact achieves high accuracy on a narrow slice of the eval set because a recipe pattern subsumed several classes. The classes that got subsumed never produced an output, so accuracy is trivially 1.0 on the subset that fired. Without a coverage axis, a degenerate artifact that handles only one of twelve declared input classes still passes the gate. With the coverage axis, the same artifact scores V = 0.083 and the composite collapses to 0.55, well below 0.85.

7. Receipt seal

The K-score and the five sub-scores are sealed into the receipt body alongside the artifact CID, the recipe CID, the eval CID, and the compile timestamp. The seal is the fourth HMAC ring (see audit log) and uses the tenant secret. Any third-party verifier with the receipt and the seal can confirm the K-score under the tenant secret without re-running the eval.

Weight derivation.

The weights are a stance, not a derivation. They encode three empirical opinions formed from the first 1,247 compiles in development, broken down by the axis where the compile actually failed:

Failure axisCompiles failed (n)% of failures
Accuracy A below threshold41268%
Coverage V shortfall9816%
Size S cliff (LoRA rank too high)529%
Latency L cliff (wrong base model)285%
Cost C ceiling (frontier-only)112%
Total failed compiles601 of 1,24748%

Accuracy dominates because failures dominate accuracy. Sixty-eight percent of all failing compiles failed on accuracy. We weight A at 0.40 not because we think it is four times as important as size or latency, but because the gate has to bite hardest on the failure mode that produces the most production regressions. Every production incident we have shipped (across the full development period) traced back to an accuracy gap, not a size or latency cliff.

Coverage is the second-largest axis by failure rate and is given equal weight with size, latency, and cost. This is deliberate: coverage is a meta-axis (it gates the validity of the other axes), not a quality axis. Weighting it higher would mean accuracy-on-subset becomes worth more than accuracy-on-whole, which is wrong. Weighting it lower would mean degenerate artifacts slip through, which we observed in the first 200 compiles before the coverage axis was added.

The four non-accuracy axes share 0.60 equally at 0.15 each. The cases where size, latency, cost, or coverage individually bite are situational; one weighting that biases for one task class biases against the others. A balanced four-way split keeps the composite domain-neutral.

The composite is convex. A 1.0 accuracy artifact with zero coverage scores 0.40 + 0.5 · 0.45 + 0.15 = 0.78 (assuming the other three axes average 0.5). This is below 0.85. The gate catches it. A balanced 0.86 across all five axes scores 0.86 · (0.40 + 0.60) = 0.86, just above the gate. The score rewards balance over peak.

The 0.85 gate.

Every compile that does not produce a composite of at least 0.85 fails closed. No artifact is emitted. The receipt is not signed. The CID is not minted. The compile process exits non-zero, the trainer writes a diagnostic JSON next to where the artifact would have been, and kolm compile prints the failing axes.

$ kolm compile -t spam_classifier -d ./examples.jsonl
[compile]     synthesizing recipes (n=42)
[compile]     evaluating on 24 held-out cases
[K-score]     A=0.875 S=0.412 L=0.834 C=1.000 V=0.667
[K-score]     composite=0.808 (gate=0.85)
[fail]        coverage below threshold; 8 of 24 cases were not exercised
[fail]        no artifact produced
[diag]        ./spam_classifier.failed.json   inspect with: kolm diagnose

The gate is adjustable per-compile with the --gate flag. A higher gate produces fewer artifacts but stronger guarantees. A lower gate produces more artifacts that may not deserve to ship. The default we chose (0.85) is the empirical knee point of the composite distribution across the first 1,247 development compiles:

Gate thresholdCompiles passing% passNotes
0.7081265%Too permissive; ships several degenerate artifacts
0.8069155%Still ships some narrow-coverage artifacts
0.8564652%Default. Knee of distribution.
0.9041233%Rejects acceptable artifacts on minor coverage gaps
0.95988%Only very-narrow tasks with synthetic balance

0.85 is where the second-derivative of the pass-rate curve goes from gentle to steep. Below 0.85, dropping the gate buys roughly equal additional artifacts per percentage point. Above 0.85, dropping the gate buys progressively more artifacts (with progressively worse quality). The 0.85 cliff means moving the gate by 0.05 in either direction has asymmetric consequences.

A failed compile is a successful gate. The artifact you never shipped is the one that never embarrassed you in front of a customer.

Ablation table.

We re-scored the 646 compiles that passed the 0.85 gate under five ablated variants of the K-score, each removing one axis and redistributing the weight proportionally to the remaining four. The question: how many of the 646 passing compiles would have passed under each ablation?

VariantFormulaPass at 0.85Δ vs full
Full K-score0.40A + 0.15S + 0.15L + 0.15C + 0.15V646—
No accuracy0.25S + 0.25L + 0.25C + 0.25V1,189+543 (degenerate)
No size0.47A + 0.18L + 0.18C + 0.18V678+32 (small drift)
No latency0.47A + 0.18S + 0.18C + 0.18V654+8 (minor)
No cost0.47A + 0.18S + 0.18L + 0.18V639-7 (minor)
No coverage0.47A + 0.18S + 0.18L + 0.18C931+285 (degenerate)

The ablation tells two stories. First, removing accuracy makes the gate nonsensical: 1,189 compiles pass on size/latency/cost/coverage alone, including artifacts that emit empty strings for every input (perfect S, L, C; trivially perfect V on the empty case class). Accuracy must be in the composite or the gate is meaningless.

Second, removing coverage produces a 285-compile false-positive bulge: artifacts with high accuracy on a narrow eval slice but with broad declared coverage. These are the degenerate artifacts the coverage axis was added to catch. Without V, a 1.0 accuracy on 1 of 12 declared classes still passes if size/latency/cost are OK.

The other three axes (size, latency, cost) have small ablation effects (under 5%) because the failures they catch are concentrated in a small fraction of compiles. They are load-bearing for the failure modes they catch, but the failure modes themselves are rarer than accuracy or coverage failures. Removing them would not invalidate the gate, but would let through artifacts that cost too much, run too slow, or take too much disk.

Conclusion: accuracy and coverage are necessary; size, latency, cost are sufficient guardrails on top of necessity.

Comparison: LMSYS, SWE-bench, OpenAI Evals.

Where K-score is stronger

  • Reproducibility. A K-score is a function of three CIDs (artifact, eval, recipe) and a tenant secret. Any third party with the CIDs can replay the score. LMSYS Elo is not reproducible: live votes do not replay.
  • Signed. The K-score is the only one of the four with a cryptographic seal. The receipt's HMAC ring binds the score to the artifact under a rotating tenant secret. SWE-bench and OpenAI Evals scores are claimed in tables; nobody signs them.
  • Includes engineering axes. Size, latency, and cost matter for production but are absent from the other three regimes. A model that ranks high on LMSYS Elo but takes 8 seconds to respond and costs $0.04 per call is not the same as one that ranks lower but ships in 50 ms for free.
  • Per-artifact, not per-model. The unit of K-score is the .kolm, not the base model. The same base model compiled twice with different recipes produces two artifacts with two different K-scores, which is the correct level for production decisions.

Where K-score is weaker

  • Not cross-task comparable. LMSYS Arena gives one Elo per model, comparable across models. SWE-bench gives one pass-rate per model, comparable across models on the same dataset. A K-score is comparable only within a task category: two healthcare summarizers have comparable K-scores; a healthcare summarizer and a code review bot do not. See section 7 (calibration) for the open problem.
  • Author-declared task. The task an artifact certifies against is declared by the author. There is no global ground-truth that a K-score of 0.92 means the same thing across all tasks. SWE-bench has a fixed dataset; LMSYS has a fixed prompt distribution. The K-score's flexibility (any task) is also its weakness (no global yardstick).
  • Newer. RS-1 K-score has fewer than 18 months of accumulated evidence; LMSYS has 2+ years and SWE-bench has 18 months. We expect to find calibration issues over time. See v2 roadmap.

Threat model.

The K-score is a gate. Gates have adversaries. The threat model below enumerates the attacks we have observed in development and the mitigations that ship in RS-1.

1. Eval set leakage into recipe synthesis

Attack: The eval set is included in the recipe synthesizer's view, so the recipe pack memorizes eval cases. Accuracy is trivially perfect on the held-out set because the held-out set is not held out.

Mitigation: The eval set is sealed before compile begins. The synthesizer sees the train split only. The eval CID is computed from the eval bytes and embedded in the receipt; a third-party verifier can check that the eval bytes match the declared CID and that the recipe pack does not include eval examples by content.

2. Coverage gaming via single-class recipe pack

Attack: The recipe pack synthesizes one pattern that matches the highest-frequency input class and ignores all other classes. Accuracy is high on the class that fires; coverage looks fine because the class count is undeclared.

Mitigation: The coverage axis is computed against declared classes, which must be enumerated in the eval manifest before compile. The eval CID seals the declared class list. An artifact that exercises 1 of 12 declared classes scores V = 0.083, and the composite collapses to 0.55.

3. Receipt seal forgery

Attack: An adversary edits a .kolm to claim a higher K-score than the artifact actually produced.

Mitigation: The receipt body is sealed with HMAC-SHA256 under the tenant secret. The seal binds the artifact CID, eval CID, recipe CID, K-score, and timestamp. Edit any one of those and the seal fails verification. The tenant secret is rotatable; see the audit chain.

4. Cross-task K-score conflation

Attack: An artifact claims a K-score of 0.92 on task A and is used as evidence of quality for task B. The score does not generalize.

Mitigation: The eval CID is mandatory in the receipt body and is bound to a task descriptor. Any verifier reading the receipt sees the task descriptor and can decide whether it matches their use case. The K-score is comparable only within a task; we say so in the spec.

5. Adversarial eval generation

Attack: The author synthesizes an eval set that is trivially easy for the artifact and inflates accuracy.

Mitigation (partial): The eval synthesizer uses an oracle judge (a stronger model) to grade outputs. An eval set that the oracle judges as too easy is rejected at synthesis time. This is incomplete; an adversarial author can still craft an eval set that the oracle accepts but that is not representative of the real-world distribution. See v2 axis R for the held-out vs declared accuracy correction.

Calibration roadmap (v2).

K-score v2 (specification tag k-score-2) is on the v0.2 spec roadmap and adds four optional axes that address calibration gaps in v1:

# v2 (when all four optional axes are supplied)
K2 = 0.30 · A
   + 0.10 · S + 0.10 · L + 0.10 · C + 0.10 · V
   + 0.10 · R + 0.10 · F
   + 0.05 · E + 0.05 · Z

R: held-out vs declared accuracy

The synthesizer's accuracy estimate (A) is paired with a second, independent eval set assembled by an external oracle. R is the gap, normalized: R = 1 - |A_declared - A_oracle|. When the gap is large, R is small, and the composite penalizes the artifact for over-optimistic self-evaluation.

F: fairness (lowest sub-group accuracy)

For tasks with declared sub-populations (e.g., demographic groups for clinical models), F is the accuracy on the worst-performing sub-group divided by the accuracy on the overall set. An artifact with 0.92 overall and 0.51 on the smallest sub-group scores F = 0.55, which drags the composite down.

E: energy per call

Joules per inference call, normalized to the lowest-energy artifact in the registry's same task category. Edge deployments (drones, on-device) gate on this axis far more than cloud deployments do, but the spec ships the axis as optional so it does not penalize cloud-first artifacts.

Z: drift vs registry baseline

For tasks that have a registry baseline, Z is the artifact's K-score divided by the median K-score of the registry's same task category. Z above 1.0 means the artifact is better than the baseline; Z below 1.0 means it is worse. The axis is optional because some tasks have no registry baseline yet.

When a v2 axis is not supplied, v2 redistributes its weight proportionally to the remaining axes so the composite stays comparable. A v1 artifact verifies under v1; a v2 artifact carries spec: 'k-score-2' in the manifest, and a verifier dispatches on the spec tag. v1 artifacts will continue to verify under v1 in perpetuity; the spec versions are additive.
