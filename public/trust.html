How verification works

What kolm signs. What tampering looks like. What verification proves.

The chain

manifest.json   = { artifact_id, base_model_hash, recipe_pack_hash, k_score, build_time }
manifest_hash   = sha256(manifest.json)
receipt[0]      = HMAC(secret, manifest_hash)
receipt[i]      = HMAC(secret, receipt[i-1] || input_hash[i] || output_hash[i])
signature       = HMAC(secret, receipt[N])

Each receipt chains to the previous. Insert, reorder, or remove any output: every later receipt invalidates. Auditors check the tail signature against the published artifact_id and HMAC reference. Walk the four rings interactively →

Verify a receipt yourself

Public endpoint, no auth. POSTs to /v1/receipts/verify; broken rings come back named, e.g. chain[3] hmac mismatch. kolm verify uses the same endpoint.

receipt JSON
accepts v0.1, drive-by, or rs-1 legacy

Programmatic:

curl -s -X POST https://kolm.ai/v1/receipts/verify \
  -H "Content-Type: application/json" \
  -d '{"receipt": {"kolm_version":"0.1", "artifact_hash":"...", "chain":[...], "signature":"..."}}'

Optional Sigstore Rekor anchor

Receipts can carry an anchors array. Kinds: http (Sigstore Rekor), arweave, btc-op-return, ots. v0.1 runtimes ignore the field if absent. Schema: receipt-v0.1.json.

What this defends against

  • Output tampering. Chain breaks on rewrite.
  • Model swap. Manifest hash differs; gate fails.
  • Behavior drift. K-score recomputed per release; drifted artifact does not promote.
  • Input replay. Receipts embed input hash; different inputs, different chain.
  • Auditor fraud. Chain verifiable from the public anchor; no auditor trust required.

What this does not defend against

  • Malicious training data. Poisoned --examples poison the artifact. Verification proves provenance, not correctness.
  • K-score gaming. Verifier sharing data with what it scores leaves K uninformative. Hold out the test source.
  • Compromised signing secret. Standard HMAC threat model. Enterprise rotates per-tenant.

K-score, worked

Promotion gate at every release. Five components, fixed weights, single scalar. Below the gate, no promote.

K = 0.40·A + 0.15·S + 0.15·L + 0.15·C + 0.15·V    where K, A, S, L, C, V ∈ [0, 1]
varcomponentweightsignal
AAccuracy0.40pass rate on held-out eval slice
SSize0.151 − (artifact_bytes / size_envelope), clipped [0,1]
LLatency0.15p50_target / observed_p50, clipped [0,1]
CCost0.15$/call_target / observed_$/call, clipped [0,1]
VCoverage0.15fraction of intended task slice exercised by eval

A passing release

From kolm bench:

A = 0.92    accuracy   :: 138/150 held-out tasks pass
S = 0.85    size       :: 3.4 GB inside a 5.0 GB envelope
L = 0.90    latency    :: p50 220 ms vs 200 ms target
C = 0.88    cost       :: $0.0011 / call vs $0.0010 target
V = 0.92    coverage   :: 92% of the named task slice exercised

K = 0.40·0.92 + 0.15·0.85 + 0.15·0.90 + 0.15·0.88 + 0.15·0.92
  = 0.368 + 0.1275 + 0.135 + 0.132 + 0.138
  = 0.9005

gate = 0.85    K ≥ gate    promote

A drifted release

Accuracy slipped on math-style; S, L, C, V unchanged:

A = 0.78    accuracy   :: 117/150 — drift on math-style tasks
S = 0.85
L = 0.90
C = 0.88
V = 0.92

K = 0.40·0.78 + 0.15·0.85 + 0.15·0.90 + 0.15·0.88 + 0.15·0.92
  = 0.312 + 0.1275 + 0.135 + 0.132 + 0.138
  = 0.8445

gate = 0.85    K < gate    DO NOT PROMOTE

Two relaxed gates. Safety-critical: K ≥ 0.95. Prototype: K ≥ 0.70 with kolm.unstable tag. Public default: 0.85. Gate, inputs, and held-out eval set are part of the receipt chain; any component measured against training is named in the receipt.
