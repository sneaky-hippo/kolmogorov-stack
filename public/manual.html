01Architecture

kolm is one product with one entry point: kolm compile. Beneath the entry point, four engines do the work. Above the entry point, three surfaces (CLI, Cloud, Mobile) run the same artifact. Same file. Same K-score. Same receipt chain.

Four engines

Distill. Verified inference at scale. K-samples the frontier model you bring, runs a deterministic verifier synthesized from your examples, produces high-quality labels. The quality moat.

Decompose. Recipe Speculative Decoding. Extracts the deterministic-token subset of observed model behavior into a registry-indexed draft pack. The compute moat.

Recall. Multimodal sqlite-vec index. Embeds and indexes text, image, audio, video, PDF; ships inside the artifact; grounds every Distill call. The data moat.

Run. Local execution. Loads model + LoRA + recipe pack + sqlite-vec index, composes them at every token. The delivery moat.

Orchestrator

The kolm compile orchestrator runs the four engines in sequence, with caching against a tenant-scoped cache and the public registry. A typical compile:

# input
task     "reply to support tickets in our voice"
examples ./tickets.jsonl     # 1,284 rows
base     qwen2.5-3b
gate     0.70

# pipeline
› synthesize verifier  # from examples
› distill              # k=8 samples × 1284 rows × frontier teacher
› decompose            # mine 442 deterministic drafts
› train lora           # 3 epochs · 82 MB delta
› evaluate             # held-out 412 rows · K-score 0.94
› package              # zip + signature
› anchor               # HMAC chain → public registry

# output
support-triage.kolm   38 MB   K=0.94   gate 0.70 · ship

02K-score

One number. Below the gate, the artifact does not ship. The K-score is a harmonic average across five components: accuracy on the held-out set, footprint, latency, cost-per-call, and coverage of structured outputs.

Formula

# weights default to 1.0 across every component K = 5 / ( 1/A + 1/F + 1/L + 1/C + 1/V ) A accuracy held-out pass-rate, 0..1 F footprint 1 − (artifact_bytes / 4 GiB), clamped 0..1 L latency 1 − (p95_ms / 2000), clamped 0..1 C cost 1 − (cents_per_1k_calls / 100), clamped 0..1 V coverage recipe-pack hit-rate on held-out, 0..1

Harmonic mean punishes outliers. A single weak component drags K down sharply. That is intentional: a model that nails accuracy at 8 GiB and 5-second p95 should not ship just because it answers well sometimes.

Ship gate

The default gate is 0.70. Set with --gate 0.85 for a stricter ship. CI integrations fail the build if K is below the gate; kolm compile exits non-zero. The gate is recorded in the manifest, so a downstream consumer can verify the artifact was held to a known bar.

03.kolm anatomy

A .kolm file is a deterministically-ordered zip. Inside: model, LoRA, recipes, recall index, tests, manifest, signature. Every layer is content-addressed by SHA-256.

04Receipts

Every artifact carries an HMAC chain over every layer. The chain is anchored to the public registry. On-chain anchoring is available for Enterprise. Tampering with any layer breaks the chain; verification is fast and offline.

HMAC chain

r0 = HMAC(secret, sha256(manifest.json))
r1 = HMAC(secret, r0 || sha256(model.gguf))
r2 = HMAC(secret, r1 || sha256(lora.bin))
r3 = HMAC(secret, r2 || sha256(recipes.json))
r4 = HMAC(secret, r3 || sha256(index.sqlite-vec))
r5 = HMAC(secret, r4 || sha256(tests.jsonl))

signature.sig = r5

The secret is per-tenant. Verification is one HMAC chain re-derive against the layer hashes in the manifest, plus a single registry lookup to confirm r5 appears at the registered position.

Registry anchor

kolm anchor writes the signature into the public registry. The registry is append-only; entries cannot be rewritten. Each anchor records the artifact hash, the tenant pseudonym, and the timestamp. /atlas is the public browser.

Enterprise tenants can additionally anchor on-chain (Bitcoin OP_RETURN or Arweave). The on-chain reference is added to a secondary signature field, so verification stays fast for the common case.

05Registry

The public registry is two things: an append-only log of artifact anchors, and an MIT-licensed pack of deterministic draft tokens auto-extracted from every compile that opted in. Future compiles get cheaper as the registry grows.

Pull the current pack:

curl -fsSL https://kolm.ai/v1/registry/export > drafts.parquet
# pre-shaped for llama.cpp and vLLM speculative-decode callbacks

06Serve · MCP

Any artifact can be exposed as a Model Context Protocol server. Drop into Claude, Cursor, Zed, Continue, or any MCP-aware agent.

$ kolm serve ./support.kolm --mcp
› mcp · ws://localhost:8765
› stdio · enabled
› http  · http://localhost:8765/v1/run
› tools exposed: run, explain, cite, k_score

The serving binary is the same one used at compile time. No model server, no GPU, no Docker.

07On-device runtime

The runtime is llama.cpp + sqlite-vec + a thin native shell. Three flavors:

  • Native (macOS/Linux/Windows). kolm run ships with llama.cpp, sqlite-vec, and the recipe-pack draft callback.
  • Web (PWA). wllama.wasm + sqlite-vec.wasm, served by the kolm app at /mobile.
  • Mobile (iOS/Android). Executorch bindings for Neural Engine and NPU. Same artifact loads.

Coldstart on a 2024-era phone for a 3B INT4 artifact: ~1.8 seconds to first token. Subsequent tokens are draft-accelerated by the recipe pack at the rates measured in /k-score.

08CLI reference

Every kolm subcommand. kolm --help prints the same list with current flags.

  • kolm login — sign in. Opens the browser; pastes your API key.
  • kolm compile <task> — build a .kolm artifact.
  • kolm run <file> <input> — run the artifact locally.
  • kolm serve <file> [--mcp] — expose the artifact as MCP/HTTP/stdio.
  • kolm inspect <file> — manifest, K-score, signature chain.
  • kolm verify <file> — verify the receipt chain against the registry.
  • kolm anchor <file> — write the signature to the registry.
  • kolm diff <a> <b> — per-component K-score and layer-hash diff.
  • kolm config — teacher provider, base model, gate defaults.
  • kolm account <export|delete|cancel> — lifecycle.
