kolm has one current production surface: the Node CLI/API path that compiles, signs, verifies, runs recipe-mode artifacts, and serves them over MCP. Cloud/mobile/native model surfaces are target bridges, not shipped runtime claims.

kolm is one product with one entry point: kolm compile. Beneath the entry point, four engines do the work. Above the entry point, three surfaces (CLI, Cloud, Mobile) run the same artifact. Same file. Same K-score. Same receipt chain.

Distill. Target bridge: k-sample frontier labels and LoRA training. Current public product should treat LoRA output as roadmap until adapters are real and benchmarked.

Decompose. Current surface: deterministic recipe-mode behavior bound into the signed artifact. This is the compiler-cache wedge.

Recall. Target bridge: sqlite-vec corpus indexes bundled in the artifact. Current public artifacts reserve the slot unless a deployment bridge populates it.

Run. Current surface: Node CLI recipe-mode run plus MCP serving. Loading model weights, applying LoRA, and mounting sqlite-vec are runtime bridge work.

The kolm compile orchestrator should be read as a staged roadmap. A current preview compile looks closer to this:

# input
task     "reply to support tickets in our voice"
examples ./tickets.jsonl     # 1,284 rows
base     qwen2.5-3b
gate     0.70

# pipeline
› build recipe artifact # current public path
› bind evals           # manifest + eval hash
› reserve lora slot    # placeholder until bridge ships
› evaluate             # release evidence
› evaluate             # held-out 412 rows · K-score 0.94
› receipt              # HMAC chain, anchor extension empty
› receipt              # HMAC chain, anchors empty

support-triage.kolm   signed zip   recipe tier   verify before deploy
support-triage.kolm 38 MB K=0.94 gate 0.70 · ship

A release score is useful only when it points to eval evidence and benchmarks. The current product can expose the gate shape; model latency, size, and coverage claims need named-device runs.

One number. Below the gate, the artifact does not ship. The K-score is a harmonic average across five components: accuracy on the held-out set, footprint, latency, cost-per-call, and coverage of structured outputs.

Formula

# weights default to 1.0 across every component K = 5 / ( 1/A + 1/F + 1/L + 1/C + 1/V ) A accuracy held-out pass-rate, 0..1 F footprint 1 − (artifact_bytes / 4 GiB), clamped 0..1 L latency 1 − (p95_ms / 2000), clamped 0..1 C cost 1 − (cents_per_1k_calls / 100), clamped 0..1 V coverage recipe-pack hit-rate on held-out, 0..1

Harmonic mean punishes outliers. A single weak component drags K down sharply. That is intentional: a model that nails accuracy at 8 GiB and 5-second p95 should not ship just because it answers well sometimes.

The default gate and CI behavior are product targets unless the installed CLI exposes those flags. Treat the manifest and receipt evidence as the current source of truth.

The default gate is 0.70. Set with --gate 0.85 for a stricter ship. CI integrations fail the build if K is below the gate; kolm compile exits non-zero. The gate is recorded in the manifest, so a downstream consumer can verify the artifact was held to a known bar.

A .kolm file is a deterministically ordered zip. Today: model pointer, placeholder LoRA, recipes, placeholder recall index, evals, manifest, signature, and receipt. Real model/index bytes are bridge outputs.

A .kolm file is a deterministically-ordered zip. Inside: model, LoRA, recipes, recall index, tests, manifest, signature. Every layer is content-addressed by SHA-256.

Every artifact carries an HMAC chain over the fields the current builder emits. External anchoring is roadmap. Tampering with covered bytes breaks local verification.

Every artifact carries an HMAC chain over every layer. The chain is bound in the artifact receipt. External anchoring is roadmap. Tampering with any layer breaks the chain; verification is fast and offline.

HMAC chain

r1 = HMAC(secret, r0 || sha256(model pointer)) r2 = HMAC(secret, r1 || sha256(lora placeholder)) r2 = HMAC(secret, r1 || sha256(lora.bin)) r4 = HMAC(secret, r3 || sha256(index placeholder)) r4 = HMAC(secret, r3 || sha256(index.sqlite-vec)) r5 = HMAC(secret, r4 || sha256(tests.jsonl)) signature.sig = r5

The secret is per-tenant. Verification is one HMAC chain re-derive against the layer hashes in the manifest and receipt body. A registry transparency log is a future hardening step, not required for today's local verification.

Anchor extension

The current receipt schema reserves anchor fields for future registry or third-party roots. Today's artifact builder emits HMAC-bound receipts; it does not publish a public transparency log.

External ledger anchoring remains a roadmap integration. Treat retained receipts and exported manifest hashes as the current evidence package.

05Registry

The registry is a versioned catalog for artifacts and deterministic draft packs. It is not yet a public transparency log.

Pull the current pack:

curl -fsSL https://kolm.ai/v1/registry/export > drafts.parquet
# pre-shaped for llama.cpp and vLLM speculative-decode callbacks

The current CLI can serve artifacts from ~/.kolm/artifacts over MCP for local agent calls. Use --http only when you want the MCP server to expose an HTTP transport as well.

$ kolm serve --mcp --http --port 8765
ok serving ~/.kolm/artifacts as local tools

No model server is required for current recipe-mode serving. Native model execution, populated vector indexes, and mobile/web bridges still depend on runtime bridge work and published benchmarks.

Current runtime: Node CLI on macOS, Linux, and Windows. Target runtimes: native desktop model loading, web, iOS, and Android after packages and benchmarks exist.

Three runtime tiers:

  • Current. kolm run, kolm eval, kolm inspect, and kolm serve --mcp for recipe-mode artifacts.
  • Roadmap. Web runtime packages with explicit memory/cache checks.
  • Roadmap. iOS/Android bindings with named-device latency, memory, battery, and store-policy evidence.

Do not quote mobile cold-start or token-rate numbers until they come from the public benchmark matrix.

08CLI reference

Every current kolm subcommand. kolm --help prints the same list with current flags.

  • kolm login — configure your API key for the CLI.
  • kolm config [base|api_key] [value] — inspect or set endpoint and API key config.
  • kolm compile <task> — build a .kolm artifact.
  • kolm run <file> <input> — run the artifact locally.
  • kolm inspect <file> — print manifest, recipes, K-score, and signature metadata.
  • kolm eval <file> — re-run embedded evals.
  • kolm score <file> — print the K-score summary.
  • kolm serve --mcp [--http] [--port] — expose ~/.kolm/artifacts as MCP tools.
  • kolm publish <file> — placeholder; public gallery publishing is not implemented.
  • kolm version — print the CLI version.
