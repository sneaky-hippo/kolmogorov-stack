For any task with verifiable structure — code, math, structured extraction, parsing, planning steps with checks — sampling k candidates from a frontier model and accepting the first that passes a deterministic verifier yields accuracy ≥ 1 − (1 − p·v)k. Generator-Verifier asymmetry has been a textbook fact since Karp 1972 and Levin 1973; it has not shipped at the LLM API surface because no one had a portable, deterministic, signed verifier. Recipe is that verifier.

The math, in three lines

Let p = single-shot accuracy of a frontier generator on a verifiable task, and v = soundness of the verifier (probability it accepts a wrong answer is <1−v). For structured verifiers (test cases, type checks, invariant checks), v ≈ 1.

Drawing k independent samples and accepting any that the verifier accepts:

P(at least one correct accepted) ≥ 1 − (1 − p·v)k

Why didn't this ship at the frontier already?

The missing ingredient

A portable, deterministic, signed verifier.

Generator-Verifier asymmetry is classical CS. The blocker was always: what runs the verifier? A test harness on someone's laptop is not a verifier you can ship. A fine-tuned classifier model is not deterministic. A regex is not auditable. Recipe is the missing component — deterministic JS / WASM, source-hashed, HMAC-signed, runs anywhere, verifies the same on every box on earth.

Why now

Two preconditions just landed.

(1) Frontier models hit p > 0.85 on HumanEval/MBPP/GSM8K — the math now bites at k≤16. (2) Recipe synthesis is reliable — we can compile a verifier from 4–8 examples in seconds. Before either landed, the asymmetry was a blackboard insight; with both, it's a shippable primitive.

What this implies for benchmarks

Recipe is a SOTA amplifier. For any frontier model M with single-shot accuracy p on a verifiable benchmark, the pair (M + Recipe verifier) monotonically beats M alone. The size of the lift depends on (1 − p) and the verifier soundness v.

  • HumanEval — per-problem test cases. v≈1. Lift is dramatic at any p < 0.99.
  • MBPP — same shape. Same lift.
  • SWE-bench Verified / Lite — patch + tests. Verifier is the test runner. Lift compounds with retry budget.
  • GSM8K — numerical answer. Verifier is exact-match parser. Lift bounded but nonzero.
  • MMLU — multiple choice. Verifier is choice extraction. Lift bounded; this is not where Verified Inference shines.

The honest framing: Verified Inference closes the gap between capability and reliability. It does not invent capability. If a model genuinely cannot solve a problem (small p), no k rescues it. But for the long tail where p ∈ [0.7, 0.95] — where most production AI lives — Verified Inference is the difference between "good demo" and "shippable system."
