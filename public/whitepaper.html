§0 Abstract

This document specifies RS-1, the wire format and protocol for .kolm artifacts. A .kolm is a single-file, deterministic, signed bundle: base model, personal LoRA adapter, recipe pack of deterministic-token drafts, multimodal recall index, eval suite, manifest with cryptographic anchors to a public registry. The compiler is kolm; the pipeline has eight normative stages (gather -spec -synthesize -k-sample -verify -fit -observe -sign).

RS-1 satisfies three goals: portability (runs on commodity CPUs, consumer GPUs, mobile NPUs, no network), verifiability (third parties byte-reproduce from inputs; consumers verify output lineage offline), compositionality (artifacts import other artifacts; receipt chain survives).

§1 Introduction

Current AI deployment forces three choices: (a) closed remote model: opacity, latency, per-token cost; (b) self-host open-weight: poor task-specific quality; (c) fine-tune open-weight: fragile, undocumented, unverifiable.

RS-1 is a fourth option: compile a task into a portable artifact. The compiler distills frontier behavior into an open-weight base + adapter + draft pack, signs every layer, emits a single file. Owned by the operator, runs offline, proves its own provenance.

Implementations MAY add capabilities, but MUST NOT rename fields, change zip layer ordering, or break receipt verification against registry.kolm.ai.

Status of this memo. RS-1 published under MIT (code) and CC-BY-4.0 (text). Errata, additions, test suite at github.com/sneaky-hippo/kolmogorov-stack. Spec follows SemVer; backward-incompatible changes increment major (RS-1 → RS-2) and require a registry epoch transition.

§3 Artifact format

A .kolm is a ZIP archive (PKZip 2.0 / DEFLATE) with fixed inner layout. Entries MUST appear in canonical order with deterministic options (mtime 2020-01-01T00:00:00Z, no extra fields, UTF-8 names, no Zip64 unless a layer exceeds 4 GiB).

v0.1 implementation status. Today, model.gguf contains a signed JSON pointer to the resolved base model (runtime "cloud"), lora.bin contains an optional KOLMPACK behavior pack, and index.sqlite-vec contains an optional KOLMIDX JSON lookup container. Native quantized weights, GGUF-LoRA adapters, and sqlite-vec recall databases are roadmap tiers (rs-2 model-adapter, rs-3 native-runtime). The table below describes the rs-1 format contract. Runtimes MUST accept either contents.
#PathRequiredPurpose
01manifest.jsonyesSource of truth for every layer's hash, size, role. Read first by every runtime.
02signature.sigyesHMAC-SHA256 chain root. 256-byte binary. See §8 and Appendix B.
03model.ggufyesBase model in GGUF [GGUF-v3]. Quantization MUST match manifest.base_model.quantization.
04lora.binconditionalPersonal adapter (GGUF-LoRA). REQUIRED if any stage produced training labels (§5.6).
05recipes.jsonyesRecipe pack: deterministic-token drafts by prefix-shape and embedding-shape (§7.5).
06index.sqlite-vecconditionalMultimodal recall index. REQUIRED if corpus supplied.
07tests.jsonlyesEval suite the artifact passed at compile. Used for K-score recompute (§6.3).
08verifiers.jsonyesVerifier definitions referenced by tests.jsonl (§7).
09provenance/OPTIONALAudit material: prompts, k-sample logs, label-acceptance traces. Operators MAY strip without breaking signature verification; signature commits to a stripped-tree hash.
Layer ordering invariant. The first 4 KiB MUST contain the local file header for manifest.json. Runtimes MAY read only the first 4 KiB to detect format/version before streaming.

Field stability. Appendix A names are REQUIRED at documented paths. Extension fields MAY be added under a top-level x_ prefix (e.g. x_acme_metadata); they MUST NOT be assumed present by others.

§5 Compile pipeline

Eight stages in order. Stages 4 (recall), 5 (k-sample), 6 (fit) MAY be elided when inputs are absent. Stages 1, 2, 3, 7, 8 are REQUIRED.

5.1 Stage 1: Gather

Inputs: task description, optional examples (jsonl), optional corpus. Compiler MUST canonicalize to UTF-8, strip BOMs, record SHA-256 as task.input_hash.

5.2 Stage 2: Spec

Synthesize structured task spec: input shape, output shape, desiderata, edge cases. Hashed to task.intent_hash; semantically equivalent descriptions SHOULD produce the same hash up to documented normalization (whitespace, casing, punctuation).

5.3 Stage 3: Synthesize verifier

Emit verifiers (§7). At least one MUST be deterministic (schema, regex, function); classifier verifiers MAY only appear in composite mode where a deterministic verifier dominates.

5.4 Stage 4: K-sample

Query frontier k times per input (default k=5, range 1-12) and apply the verifier. First passing output becomes the label. If none pass, the example is frontier.unverified and excluded from training (FM-001, §10.3).

5.5 Stage 5: Verify

Independent verifier pass over accepted pairs detects label drift. Failing pairs dropped. Acceptance rate logged in provenance/k-sample.log.

5.6 Stage 6: Fit

Fit a LoRA adapter on verified pairs against the base. Defaults in §A.adapter. INT4 + bf16 mixed-precision MAY be used when VRAM allows; full-precision CPU is permitted but not RECOMMENDED.

5.7 Stage 7: Observe

Run fitted artifact against tests.jsonl, compute K-score (§6). Recipe candidates from accepted outputs are proposed to the registry; those passing gated synthesis enter future epochs.

If the K-score gate fails, the compiler MUST NOT emit a signed artifact and SHOULD emit a diagnostic bundle under ./build/. See FM-001 (§10.3).

5.8 Stage 8: Sign

Compute layer hashes, build manifest, compute signature chain (§8), write artifact, post anchor to registry. After ack the signature is anchored; consumers verify offline against a cached epoch root.

§6 K-score

Composite quality metric. Single number in [0, 100], three subcomponents.

6.1 Definition

T = task accuracy (proportion of verifier-passing outputs). C = calibration: 1 - mean absolute deviation between stated confidence and observed accuracy on a 10-bucket reliability diagram. L = latency score: step function over p50 (<10 ms = 100, <50 ms = 95, <250 ms = 85, <1 s = 70, <5 s = 50, - s = 0). Then:

// composite K-score
K = round( 100 · ( 0.60·T + 0.25·C + 0.15·L ) , 1 )

Weights (0.60, 0.25, 0.15) are normative for K-score 1.0; MAY be overridden by a task profile (latency-critical, calibration-critical) with name and weights in manifest.k_score.profile.

6.2 Gate semantics

GateConditionBehavior
passedK ≥ floor and T ≥ 0.85Artifact emitted.
warnedfloor > K ≥ floor - 0.05 or 0.85 > T ≥ 0.75Emitted with warning. Operators should inspect the eval suite.
failedK < floor - 0.05 or T < 0.75Artifact MUST NOT be emitted. Compile exits non-zero. See FM-001.

6.3 Recompute protocol

Any party MAY recompute: (1) load tests.jsonl and verifiers.json; (2) run inference on each test input; (3) apply the verifier; (4) recompute T, C, L per §6.1. Divergence beyond ±0.5 K-points SHOULD be reported as tampering.

§7 Verifier types

A verifier is a deterministic decision procedure (input, candidate_output) → boolean. Five normative types:

TypeFormDeterministic?Use
schemaJSON Schema 2020-12 over outputyesStructured outputs (extraction, function-calls).
regexRE2 regex over outputyesFormat constraints (dates, IDs, labels).
functionPure JS/Python evaluator by SHA-256yesComputable predicates (parser, math, code-runs).
classifierBundled small model (< 200 MB), calibrated thresholdno, but stableSubjective tasks (style, tone, harmlessness). MUST pair with a deterministic gate in a composite.
compositeBoolean combination of the abovefollows leavesReal tasks. AND/OR, documented short-circuit order.

Verifiers are referenced from tests.jsonl by ID; bodies in verifiers.json, content-addressed. Renaming invalidates the receipt chain.

§8 Receipt chain

HMAC-SHA256 over a canonical statement of an inference event. Emitted at every kolm run, chained to the artifact signature, which anchors to the public registry.

8.1 Statement form

// canonical receipt statement (CSON, sorted keys)
{
  "v": "rs-1-receipts/1.0.0",
  "artifact": "kolm:8b73e9...02",
  "input_hash": "sha256:7c1af2...",
  "output_hash": "sha256:e84b91...",
  "runtime": { "name": "kolm-run", "version": "6.5.0", "host": "linux-x86_64" },
  "observed_at": "2026-05-08T14:32:11Z",
  "k_score_passed": true
}

8.2 Chain construction

Receipt body is HMAC-SHA256'd under a per-artifact key derived from the artifact signature and the operator tenant secret. MAC is appended for the wire-format receipt.

signature.sig is itself an HMAC over the canonical concatenation of every manifest layer hash, under the registry epoch key. Anchor posts go into a Merkle tree; daily root is published at registry.kolm.ai/epoch/<date>/root.json.

8.3 Offline verification

Verifier needs: (1) receipt body + MAC, (2) artifact signature.sig + manifest, (3) registry epoch root for the anchor day. Every chain link reconstructs without network.

Tenant secret rotation. Old-secret receipts remain verifiable against the previous epoch snapshot. New receipts use the new secret. See FM-202.

§9 Reproducibility

Byte-identical reproducibility. Given the same (task, examples, corpus, base_model, recipe_registry@epoch), two independent compiles MUST produce the identical zip, not only the same K-score.

Mechanism:

  • K-sample ties broken by lexicographic comparison of canonical outputs, not wall-clock.
  • LoRA training uses fixed seeds from task.intent_hash; data shuffle is a deterministic permutation of (intent_hash, example_index).
  • ZIP entries: fixed mtime, no extra fields, stable per-file order (§3).
  • FP determinism: GPU via cuBLAS workspace limits + deterministic-cudnn; CPU via single-threaded BLAS for the final norm-and-clip pass.

A frontier model non-deterministic at temperature 0 (e.g. provider-side batching) is the only legitimate source of irreproducibility. The compiler caches first-compile k-sample winners in provenance/k-sample.log; replays use the cache when inputs match.

§10 Threat model & failure modes

10.1 In scope

  • Tampered artifacts (modified weights, swapped LoRA, edited evals).
  • Forged receipts (claiming an output came from a given artifact).
  • Registry-side recipe corruption (malicious draft biases inference).
  • Frontier-model failure during compile (rate limit, content policy, downtime).
  • Runtime failure on the consumer device (missing CPU instructions, OOM).

10.2 Out of scope

  • Side-channel attacks on the operator's machine.
  • Plaintext exfiltration before compile (operator owns inputs and transport).
  • Inverting an artifact to reconstruct training data. A .kolm stores neither raw examples nor raw corpus; only embedded chunks and verifier-passing labels.

10.3 Failure modes (FM)

Twelve normative failure modes, each with an exit code, HTTP status (when surfaced via API), and recovery action. Representative slice:

CodePhaseSeverityAction
FM-001compilefailK-score below floor; diagnostic bundle, exit 65.
FM-002compiledegradedCorpus partially unreadable; proceed, warn, list skipped.
FM-003compiledegradedFrontier quota exceeded; fall back to cached winners; mark recompile_recommended.
FM-101runtimefailSignature invalid; refuse, exit 70.
FM-103runtimefailOOM. Return 503, suggest smaller quantization.
FM-201registrydegradedRegistry down; verify against last-cached epoch root.
FM-301authfailBearer invalid. Return 401.

§11 Conformance

The operational definition is the public test suite at github.com/sneaky-hippo/kolmogorov-stack/tree/main/conformance. This section is the human-readable summary.

11.1 Compiler conformance

  • MUST implement stages 1, 2, 3, 7, 8 (§5).
  • MUST emit byte-identical artifacts for identical inputs (§9).
  • MUST NOT emit a signed artifact if the K-score gate fails (§6.2).
  • MUST include at least one deterministic verifier per task (§7).
  • MUST post anchor records to the registry on success.

11.2 Runtime conformance

  • MUST verify signature.sig before executing any layer.
  • MUST emit a receipt per inference event (§8).
  • MUST refuse artifacts whose rs major exceeds runtime support.
  • SHOULD support offline operation. Receipt verification and inference MUST work without network when the epoch root is locally cached.
  • MAY implement recipe-driven speculative decoding (§7.5); affects latency, not correctness.

11.3 Naming

Conformant implementations MAY use the name kolm, extension .kolm, MIME application/vnd.kolm.rs1+zip. kolm/Kolmogorov are trademarks of Kolmogorov, Inc.; permitted for conformant implementations and editorial reference. RS-1, K-score, recipe pack are public terms.

§B Appendix B: signature wire format

signature.sig is a 256-byte binary blob:

// signature.sig - RS-1 1.0.0
offset  size  field
0       4     magic                "KOLM"
4       2     version              0x0100  (= 1.0)
6       2     reserved             0x0000
8       32    manifest_sha256
40      32    layers_concat_sha256 // sha256 of sorted (path, sha256) tuples
72      32    epoch_root           // registry root at anchor time
104     32    anchor_record_id     // id assigned by registry post
136     32    hmac                 // hmac-sha256 over bytes 0..135 under epoch key
168     88    reserved (zeros)
256     EOF

Verification: compute HMAC-SHA256(epoch_key, bytes[0..135]) and compare to bytes[136..167]. The epoch key is published at registry.kolm.ai/epoch/<anchor_date>/key.json and signed by the registry's long-term Ed25519 key.

End of specification. Errata and conformance vectors at github.com/sneaky-hippo/kolmogorov-stack. Maintainers: spec@kolm.ai.
