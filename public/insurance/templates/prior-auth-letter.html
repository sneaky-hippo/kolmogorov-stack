Prior-authorization letter generator

A kolm template that drafts a clinically-grounded prior-authorization request letter from a planned-service code + member context + medical history. The model is fine-tuned to satisfy the standard "medical necessity" four-criterion framework (1: appropriate diagnosis; 2: evidence-based treatment guideline; 3: prior failed conservative therapy; 4: clinical urgency). Output is structured: { letter_body, criteria_satisfied: { ... }, supporting_evidence_citations[], submission_route: "278|portal|fax", confidence }. Class: model-class (class 3).

Input & output spec

One request per planned service. The recipe ingests the planned-service context, the member's diagnostic and treatment history, and the payer's policy index, then emits a letter whose four medical-necessity criteria are independently scored. Missing-evidence criteria are flagged in criteria_satisfied as false rather than fabricated.

input

{
  "planned_service": {
    "cpt":         ["27447"],      // total knee arthroplasty
    "icd10":       ["M17.11"],
    "planned_dos": "2026-06-15"
  },
  "member": {
    "age":                 68,
    "dx_history":          ["M17.11", "E11.9"],
    "treatment_history":   [
      "trial_nsaid_12wk",
      "trial_physical_therapy_8wk",
      "trial_intraarticular_steroid"
    ],
    "plan_code":           "PPO-Gold-2026",
    "member_id_redacted":  "<hash>"
  },
  "payer": {
    "policy_index": {
      "plan_doc_ref": "PPO-Gold-2026 §5.4 (joint arthroplasty)",
      "mcg_ref":      "MCG 25th ed. SG-OR-TKA"
    },
    "prior_auth_form_url": "https://payer.example.com/forms/pa-tka.pdf"
  }
}

output

{
  "letter_subject": "Prior authorization request: CPT 27447 (TKA) for member <hash>",
  "letter_body":    "<full PA letter, ~500 words>",
  "criteria_satisfied": {
    "diagnosis":      true,
    "guideline":      true,
    "prior_therapy":  true,
    "urgency":        false    // elective, no urgency claimed
  },
  "supporting_evidence_citations": [
    {
      "source":      "AAOS Clinical Practice Guideline",
      "section":     "TKA 2023 §3.2",
      "anchor_text": "TKA is indicated for severe OA refractory to conservative management."
    }
  ],
  "submission_route": "278",
  "confidence":       0.94
}
Honest criteria reporting. If the member's record does not support a criterion (e.g. no documented prior failed therapy), the model is trained to set the corresponding criteria_satisfied[*] to false rather than draft a paragraph that asserts it. Downstream automation can branch on the boolean: a letter with criteria_satisfied.prior_therapy === false is routed to the clinical reviewer before submission.

Four-criterion medical-necessity framework

The recipe is fine-tuned against the four-criterion framework used by CMS NCD/LCD coverage decisions and most state insurance regulators. Each criterion is a binary in the output; the letter body is conditioned on which criteria the input supports.

criterion 1
appropriate_diagnosis
criterion 2
evidence_based_guideline
criterion 3
prior_failed_conservative_therapy
criterion 4
clinical_urgency

Evidence sources are pulled in this priority order: national specialty society guideline (AAOS, ACC, ASCO, etc.) → USPSTF recommendation → CMS NCD/LCD → payer published policy → peer-reviewed journal. The priority order is enforced at compile time in pack.rules.evidence_sources_priority.

K-score axes for a grounded letter

Same axis set as the denial-appeal-letter sibling. Groundedness measures the fraction of supporting_evidence_citations that resolve to a policy_index entry; an ungrounded citation fails the recipe.

accuracy
schema validity + criteria coverage
weight 0.35
size
model footprint vs target_size
weight 0.10
latency
p95 generation latency on target hardware
weight 0.15
cost
per-letter token cost on target hardware
weight 0.10
coverage
holdout cases passing the schema check
weight 0.10
groundedness
fraction of citations anchored in policy_index
weight 0.20

Gate: composite K-score must be ≥ 0.92 on the holdout. The groundedness axis alone has a floor of 0.98.

4-ring receipt schema

Every generated letter ships a receipt your auditor can re-verify offline. The execution ring carries the criteria-satisfied vector so the auditor can confirm post-hoc that the recipe was honest about which criteria the input actually supported.

ring 01 · pre-image
planned_cpt + member_hash + dx_hash
tenant-rotated HMAC
ring 02 · derived
policy_index_hash + plan_doc_ref
bound to recipe_version
ring 03 · execution
criteria_vector + citations_hash + groundedness
judge_id signed
ring 04 · seal
SHA-256(ring1..3) · verify endpoint
offline-verifiable

Verify command: kolm verify --hash <receipt_hash>. Runs locally against the seal, no network required.

Submission route

BAA fast-track

Health plans on a BAA see kolm production-ready in two business days. The customer-hosted bridge runs in the tenant network boundary; PHI in the member context (member id, DOB, dx history, treatment history) is hashed inside the bridge and never crosses our infrastructure; the audit log is yours to keep.

how it works

# 1. Sign the BAA (legal-team turnaround)
[ DocuSign · 48 hours typical · mutual NDA in same envelope ]

# 2. Provision the bridge in your VPC
$ kolm bridge install \
    --provider aws \
    --region us-east-1 \
    --baa <baa-id>

# 3. Compile this template against your seed letters + policy index
$ kolm compile --spec prior-auth-letter.spec.json \
    --examples ./your-anonymized-pa-letters.jsonl \
    --policy-index ./your-policy-index.json \
    --gate 0.92 \
    --out prior-auth-letter.kolm

# 4. Generate one letter
$ kolm run prior-auth-letter.kolm \
    --input pa-request.json \
    --emit pa-letter.json
