EDI 278 services review transformer

A kolm template that ingests EDI 278 services-review transactions (5010A1) - health-services review requests + responses (prior authorizations, referrals, admissions, certifications, extensions). Distinguishes 278 request (information-source UMO) from 278 response. Pure rule-class; deterministic; pairs cleanly with the existing prior-auth-review template (which adds the policy-mapped recommendation on top).

Input & output spec

Input is raw EDI 278 X12 text. Output is a normalized JSON object whose direction field is auto-detected: a 278 request carries UM + service-detail loops, a 278 response carries the HCR (health-care-services-review) decision segment. Downstream prior-auth tooling can rely on the fixed shape without re-parsing X12.

Companion template. The 278 transformer normalizes the wire format; prior-auth-review adds the policy-mapped recommendation on top. Pipe edi-278 output into prior-auth-review for a structured cert / partial / deny answer with criteria references; serialize prior-auth-review output back into a 278 response via the HCR-emitting half of this same recipe.

Request vs response

A 278 transaction is either an information-source request from a requester (provider, clinic) to a UMO (utilization-management organization), or a response from the UMO carrying the certification decision. The recipe auto-detects which one it is received by the presence of the HCR segment.

request · no HCR segment

# 278 request shape
direction        = "request"
umo              = information-source NM1*X3
requester        = NM1*1P
service.UM       = requested service category
service.HSD      = requested units + dates
attached_paperwork = optional PWK
response         = absent

response · HCR present

# 278 response shape
direction         = "response"
response.decision = HCR01 mapped to certified / denied / pended / partial / modified
response.certification_number = REF*BB
response.action_code          = HCR01 raw (A1..A6)
response.certified_units      = HSD certified
response.valid_from / valid_to = DTP*348 / DTP*349
response.decision_reason       = HCR03

4-ring receipt schema

Every transform ships a receipt your auditor can re-verify offline. Because this is a rule-class recipe the execution ring carries a deterministic hash of the parsed segments rather than a model judge id; the seal is identical to the model-class templates so downstream verifier code does not branch on recipe class.

ring 01 · pre-image
isa_control_num + bht_reference + member_hash
tenant-rotated HMAC
ring 02 · derived
um01_dict_hash + hcr01_dict_hash + loop_topology_hash
bound to recipe_version
ring 03 · execution
parsed_segments_hash + direction_flag + decision_code
deterministic, no judge
ring 04 · seal
SHA-256(ring1..3) · verify endpoint
offline-verifiable

Verify command: kolm verify --hash <receipt_hash>. Runs locally against the seal, no network required.

BAA fast-track

Health plans on a BAA see kolm production-ready in two business days. The customer-hosted bridge runs in the tenant network boundary; PHI in the 278 envelope (subscriber and patient identifiers, DOB, gender) is hashed inside the bridge and never crosses our infrastructure; the audit log is yours to keep.

how it works

# 1. Sign the BAA (legal-team turnaround)
[ DocuSign · 48 hours typical · mutual NDA in same envelope ]

# 2. Provision the bridge in your VPC
$ kolm bridge install \
    --provider aws \
    --region us-east-1 \
    --baa <baa-id>

# 3. Compile this template (rule-class, no seed examples required)
$ kolm compile --spec edi-278.spec.json \
    --gate 0.93 \
    --out edi-278.kolm

# 4. Stream production 278 traffic through it
$ kolm run edi-278.kolm \
    --input services-review.278 \
    --emit normalized.jsonl
