Provider NPI directory lookup

A kolm template that resolves a provider NPI to a normalized directory record + validates the NPI checksum (Luhn-mod-10) + the entity-type code (1: individual provider / 2: organization). The lookup itself queries the NPPES NPI Registry through a pluggable tenant-side resolver, the template ships the parser + validator + the schema; tenants wire to NPPES or their internal provider master. Class: rule (class 0).

Input & output spec

Input is a 10-digit NPI and an optional resolver hint. Output is the validated record plus a boolean for the Luhn-mod-10 checksum and the resolved entity-type code from the NPPES schema.

input

{
  "npi":      "1234567893",
  "resolver": "nppes_v2"
}

output

{
  "npi":             "1234567893",
  "valid_checksum":  true,
  "entity_type":     "individual",
  "resolver_used":   "nppes_v2",
  "record": {
    "name":           "<tenant-supplied>",
    "addresses":      [],
    "taxonomy_codes": [],
    "identifiers":    []
  }
}

Luhn-mod-10 checksum (CMS spec)

CMS specifies the NPI check digit as a Luhn-mod-10 over the first 9 digits with a constant 5-digit prefix prepended before the computation. Pinning the prefix in the rule library lets QA re-verify any check without re-reading the CMS specification.

prefix (CMS-pinned)
80840
algorithm
Luhn-mod-10
entity-type 1
individual provider
entity-type 2
organization

NPPES public API: https://npiregistry.cms.hhs.gov/api/?version=2.1. Tenants may swap the resolver to their internal provider master without changing the template; the output schema is identical regardless of resolver.

4-ring receipt schema

Every NPI lookup ships a receipt your auditor can re-verify offline. The chain proves which NPI was queried, which resolver responded, the record returned, and the seal.

ring 01 · pre-image
npi + resolver + request_timestamp
tenant-rotated HMAC
ring 02 · derived
luhn_prefix_table + resolver_url_hash
bound to recipe_version
ring 03 · execution
valid_checksum + entity_type + record_digest
deterministic, no judge
ring 04 · seal
SHA-256(ring1..3) · verify endpoint
offline-verifiable

Verify command: kolm verify --hash <receipt_hash>. Runs locally against the seal, no network required.

BAA fast-track

Health plans on a BAA see kolm production-ready in two business days. The customer-hosted bridge runs in the tenant network boundary; provider PII is processed there and never crosses our infrastructure; the audit log is yours to keep.

how it works

# 1. Sign the BAA (legal-team turnaround)
[ DocuSign · 48 hours typical · mutual NDA in same envelope ]

# 2. Provision the bridge in your VPC
$ kolm bridge install \
    --provider aws \
    --region us-east-1 \
    --baa <baa-id>

# 3. Compile this template against the rule library
$ kolm compile --spec npi-directory.spec.json \
    --gate 0.95 \
    --out npi-directory.kolm

# 4. Run a batch of NPIs through the resolver
$ kolm run npi-directory.kolm \
    --input provider-npis.jsonl \
    --emit npi-records.jsonl
