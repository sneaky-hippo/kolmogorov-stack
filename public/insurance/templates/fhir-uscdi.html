FHIR R4 USCDI v3 normalizer

A kolm template that ingests FHIR R4 bundles (Patient + Encounter + Condition + Observation + MedicationRequest + Procedure + Immunization + AllergyIntolerance + DocumentReference + DiagnosticReport + CarePlan) and emits normalized USCDI v3 JSON shaped for analytics. Pure rule-class — no inference required for the structural mapping. USCDI v3 (the July 2022 ONC data class set) plus the v4 candidate fields are recognized; v4-only fields are passed through with a _uscdi_class: "v4_candidate" tag so downstream consumers can opt in.

Input & output spec

Input is a FHIR R4 Bundle (the wire format every certified EHR is required to expose). Output is one normalized USCDI v3 JSON object with each data class collapsed to a flat, analytics-friendly array; coding-system priority is enforced so consumers do not have to chase concept maps.

FHIR resource → USCDI v3 data class map

Eleven FHIR resourceType values populate the USCDI v3 data class buckets that ONC published in July 2022. The map is exhaustive for v3; v4 candidate fields ride through with a _uscdi_class: "v4_candidate" tag so downstream consumers can opt in without breaking v3 contracts.

Patient
patient_demographics
Encounter
encounter_information
Condition
problems
Observation
laboratory | vital_signs | smoking_status | assessment_and_plan_of_treatment
MedicationRequest / MedicationStatement
medications
Procedure
procedures
Immunization
immunizations
AllergyIntolerance
allergies_and_intolerances
DocumentReference
clinical_notes
DiagnosticReport
diagnostic_imaging | laboratory | pathology
CarePlan
care_team_member_information | goals | assessment_and_plan_of_treatment

Coding-system priority is fixed: SNOMED CT before ICD-10-CM for problems; LOINC for observations and diagnostic reports; RxNorm before NDC for medications; CVX for immunizations; SNOMED CT before CPT for procedures. The full priority table lives in pack.rules.coding_system_priority.

4-ring receipt schema

Every normalization ships a receipt your auditor can re-verify offline. The chain proves which bundle was ingested, which version of the resource-to-USCDI map was applied, what the output digest was, and that the PHI boundary held.

ring 01 · pre-image
bundle_hash + entry_count + resourceType counts
tenant-rotated HMAC
ring 02 · derived
resource_to_uscdi_class version + coding_system_priority version
bound to recipe_version
ring 03 · execution
output_digest + redaction_count + class_buckets_populated
judge_id signed
ring 04 · seal
SHA-256(ring1..3) · verify endpoint
offline-verifiable

Verify command: kolm verify --hash <receipt_hash>. Runs locally against the seal, no network required.

BAA fast-track

Health plans on a BAA see kolm production-ready in two business days. The customer-hosted bridge runs in the tenant network boundary; FHIR Bundles are processed there and never cross our infrastructure; the audit log is yours to keep.

how it works

# 1. Sign the BAA (legal-team turnaround)
[ DocuSign · 48 hours typical · mutual NDA in same envelope ]

# 2. Provision the bridge in your VPC
$ kolm bridge install \
    --provider aws \
    --region us-east-1 \
    --baa <baa-id>

# 3. Compile this template (rule-class, no seed examples needed)
$ kolm compile --spec fhir-uscdi.spec.json \
    --gate 0.94 \
    --out fhir-uscdi.kolm

# 4. Pipe FHIR R4 Bundles through it
$ kolm run fhir-uscdi.kolm \
    --input ehr-export.bundle.json \
    --emit uscdi-v3.jsonl
