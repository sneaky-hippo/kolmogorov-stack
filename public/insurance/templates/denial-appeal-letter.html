Denial-code → appeal letter generator

A kolm template that drafts a complete, factually-grounded appeal letter from a denial code + claim context + member context. The model is fine-tuned to cite the policy section being appealed, the medical-necessity criteria the original denial missed, the patient-specific facts, and the regulatory hook (state-level external review timing). Output is structured: { letter_body, citations, regulatory_basis, recommended_attachments[], confidence }. Class: model-class (class 3). The model never invents citations — the citation list is grounded in the lib.pack.policy_index passed at compile time.

Input & output spec

One request per appeal. The denial context (CARC code + reason text) is paired with the claim, the member, and the tenant-supplied policy index. Every citation in the output is anchored back to an entry the tenant gave the recipe at compile time; the model cannot invent a source.

input

{
  "denial_code":        "50",             // CARC 50
  "denial_reason_text": "These are non-covered services because this is not deemed a medical necessity by the payer.",
  "claim": {
    "cpt":               ["70553"],
    "icd10":             ["G43.909", "R51"],
    "dos":               "2026-04-12",
    "billed":            1840.00,
    "member_id_redacted": "<hash>"
  },
  "member": {
    "age":                41,
    "plan_code":          "PPO-Gold-2026",
    "prior_auth_history": ["PA-2026-001"]
  },
  "policy_index": {
    "plan_doc_ref":       "PPO-Gold-2026 §4.7 (advanced imaging)",
    "mcg_ref":            "MCG 25th ed. ACG-A-0193 (MRI brain)",
    "state":              "CA"
  }
}

output

{

Grounded citations. The recipe takes a tenant-supplied policy_index at compile time (your plan documents, your MCG/InterQual licenses, your medical-policy bulletins). At inference the model is constrained to populate citations[*].source from that index; if no entry supports the argument, the model is trained to drop the argument rather than fabricate a citation. The same constraint applies to regulatory_basis.statute — pulled from pack.rules.regulatory_basis_by_state.

K-score axes for a grounded letter

Letter generation cannot be scored on exact-match accuracy — two factually-equivalent appeal letters can use different prose. The K-score weights instead reward grounded citations, schema validity, and end-to-end coverage, with a groundedness axis specific to this recipe class.

accuracy
schema validity + criteria coverage
weight 0.35
size
model footprint vs target_size
weight 0.10
latency
p95 generation latency on target hardware
weight 0.15
cost
per-letter token cost on target hardware
weight 0.10
coverage
holdout cases passing the schema check
weight 0.10
groundedness
fraction of citations anchored in policy_index
weight 0.20

Gate: composite K-score must be ≥ 0.92 on the holdout. Tighter would be punitive for natural-language quality; looser would ship ungrounded prose. Groundedness is computed as the fraction of citations in the letter that resolve to an entry in the supplied policy_index; the score floor for the groundedness axis alone is 0.98 (i.e. at most one ungrounded citation per fifty).

4-ring receipt schema

Every generated letter ships a receipt your auditor can re-verify offline. Because this is a model-class recipe the execution ring carries the judge id and the groundedness score; the seal is identical to the rule-class templates so downstream verifier code does not branch on recipe class.

ring 01 · pre-image
denial_code + claim_hash + member_hash
tenant-rotated HMAC
ring 02 · derived
policy_index_hash + plan_doc_ref + state
bound to recipe_version
ring 03 · execution
letter_hash + citations_hash + groundedness
judge_id signed
ring 04 · seal
SHA-256(ring1..3) · verify endpoint
offline-verifiable

Verify command: kolm verify --hash <receipt_hash>. Runs locally against the seal, no network required.

BAA fast-track

Health plans on a BAA see kolm production-ready in two business days. The customer-hosted bridge runs in the tenant network boundary; PHI in the denial context (member id, DOB if present, free-text reason) is hashed inside the bridge and never crosses our infrastructure; the audit log is yours to keep.

how it works

# 1. Sign the BAA (legal-team turnaround)
[ DocuSign · 48 hours typical · mutual NDA in same envelope ]

# 2. Provision the bridge in your VPC
$ kolm bridge install \
    --provider aws \
    --region us-east-1 \
    --baa <baa-id>

# 3. Compile this template against your seed letters + policy index
$ kolm compile --spec denial-appeal-letter.spec.json \
    --examples ./your-anonymized-appeals.jsonl \
    --policy-index ./your-policy-index.json \
    --gate 0.92 \
    --out denial-appeal.kolm

# 4. Generate one letter
$ kolm run denial-appeal.kolm \
    --input denial-context.json \
    --emit appeal-letter.json
