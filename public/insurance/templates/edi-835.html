EDI 835 remittance transformer

A kolm template that ingests EDI 835 ERA (Electronic Remittance Advice) X12 transactions (5010A1) and emits normalized payment JSON ready for posting to your AR system. Balance-checks the TS3/TS2/PLB segments against the BPR total. Pure rule-class · fully deterministic, no model inference.

Segments parsed

The transformer recognizes the 5010A1 ERA segments that drive payment posting. The ISA/GS/ST envelopes are parsed for routing but their values are not load-bearing for balance.

BPR
payment amount, method (ACH / CHK / NON), payment date hooks
TRN
reassociation trace number for matching to the EFT
DTM
qualifier 405 - production date of the payment
N1 PR / N1 PE
payer and payee identification with NPI (XX) and tax id (FI)
LX
header-level grouping segment, retained for parity with the 5010 loop hierarchy
CLP
claim payment - control number, status code, charge, paid, patient responsibility
CAS
claim adjustment - group code (CO / OA / PI / PR), reason code, amount
SVC
service-line payment detail, optional, attached to the open CLP
PLB
provider-level adjustment - withholds, refunds, capitation
SE / GE / IEA
trailer counts checked against parsed segment count

Input & output spec

Input is the raw X12 text of an EDI 835 ERA transaction. Output is normalized JSON with a fixed shape so your AR posting code does not need to re-parse anything.

Balance-check semantics

Payment posting is unforgiving: a one-cent variance between the BPR declared total and the sum of CLP payments plus PLB adjustments breaks AR reconciliation. The transformer computes the balance inline and returns a boolean a posting job can fail-fast on.

the equation

# declared total is BPR02
declared_total = BPR.02

# computed total is sum of CLP04 minus sum of PLB amounts
# (PLB amounts are signed in the X12 spec; refunds reduce
# the EFT and capitation withholds reduce paid claims)
computed_total = sum(claim.total_paid for claim in claims)
              - sum(plb.amount for plb in provider_adjustments)

# tolerance is one cent to absorb FP rounding
balanced = abs(computed_total - declared_total) < 0.01

CLP02 status codes (real X12 5010)

"1"  -> Processed as Primary
"2"  -> Processed as Secondary
"3"  -> Processed as Tertiary
"4"  -> Denied
"19" -> Processed as Primary Forwarded
"20" -> Processed as Secondary Forwarded
"22" -> Reversal of Previous Payment
"23" -> Not Our Claim, Forwarded

# CAS01 adjustment groups
CO -> Contractual Obligation
OA -> Other Adjustment
PI -> Payer Initiated Reductions
PR -> Patient Responsibility

Why rule class

EDI 835 is a fully specified deterministic grammar. There is no language to interpret, no clinical judgement, no policy reasoning. A model would be slower, more expensive, and would introduce non-determinism into a payment-posting path that legal and compliance have already locked. This template is class 0 - rule - by design and by spec.
