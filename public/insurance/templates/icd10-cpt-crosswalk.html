ICD-10 ↔ CPT crosswalk lookup

A kolm template that resolves bidirectional ICD-10-CM ↔ CPT relationships for claims, prior-auth, and policy-mapping use cases. Two query modes: (1) given an ICD-10 diagnosis, what CPT procedures are typically associated (medically appropriate)? (2) given a CPT procedure, what ICD-10 diagnoses justify it under common payer policy? The crosswalk is a curated subset (top ~500 ICD-10 ↔ ~2000 CPT) shipped in pack.rules; tenants can extend with their plan-specific overrides. Class: rule (class 0).

Input & output spec

Input picks a direction (icd_to_cpt or cpt_to_icd) and supplies the seed code. Output is the matches list with per-match strength (primary / secondary / optional) and the basis (CMS LCD, AMA CPT guidelines, or specialty society guidance).

input · icd_to_cpt

{
  "mode":  "icd_to_cpt",
  "icd10": "I10"
}

output · icd_to_cpt

{
  "mode":  "icd_to_cpt",
  "query": "I10",
  "matches": [
    { "code": "99213", "description": "Office visit, established, low complexity",      "strength": "primary",   "basis": "ama_cpt_guidelines" },
    { "code": "99214", "description": "Office visit, established, moderate complexity", "strength": "primary",   "basis": "ama_cpt_guidelines" },
    { "code": "93000", "description": "Electrocardiogram, complete",                    "strength": "secondary", "basis": "ama_cpt_guidelines" }
  ]
}

Match strength tiers

Every crosswalk entry carries a strength label so claims and prior-auth pipelines can be tuned to the plan's appetite for borderline-medically-appropriate edges. The basis field is provenance: which authority backs the edge.

primary
core medically-indicated pairing
secondary
commonly associated under typical workup
optional
supported under specific clinical scenarios

Tenants extend the seed crosswalk with their own LCD/NCD-mapped overrides via params.pack_overrides.icd_to_cpt_seed at compile time. The override map merges over the shipped seed; per-edge audit ships the basis so a downstream reviewer can re-derive the recommendation.

4-ring receipt schema

Every crosswalk lookup ships a receipt your auditor can re-verify offline. The chain proves which code was queried, which crosswalk version was applied, the matches returned, and the seal.

ring 01 · pre-image
mode + query_code + crosswalk_version
tenant-rotated HMAC
ring 02 · derived
seed_table_hash + override_layer_hash
bound to recipe_version
ring 03 · execution
matches[] + strength + basis
deterministic, no judge
ring 04 · seal
SHA-256(ring1..3) · verify endpoint
offline-verifiable

Verify command: kolm verify --hash <receipt_hash>. Runs locally against the seal, no network required.

BAA fast-track

Health plans on a BAA see kolm production-ready in two business days. The customer-hosted bridge runs in the tenant network boundary; PHI in claim payloads is processed there and never crosses our infrastructure; the audit log is yours to keep.

how it works

# 1. Sign the BAA (legal-team turnaround)
[ DocuSign · 48 hours typical · mutual NDA in same envelope ]

# 2. Provision the bridge in your VPC
$ kolm bridge install \
    --provider aws \
    --region us-east-1 \
    --baa <baa-id>

# 3. Compile this template against the crosswalk
$ kolm compile --spec icd10-cpt-crosswalk.spec.json \
    --gate 0.93 \
    --out icd10-cpt-crosswalk.kolm

# 4. Run a batch of code lookups
$ kolm run icd10-cpt-crosswalk.kolm \
    --input lookups.jsonl \
    --emit crosswalk-matches.jsonl
