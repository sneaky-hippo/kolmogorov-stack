BYOC deploy

Deploy a signed .kolm to your own infrastructure. kolm.ai mints the deploy manifest and records the attestation; you keep the keys and the network path. For payload-blind operation, pick a Trusted Execution Environment target. Business plan and up.

Read the docs

What BYOC is

Your .kolm artifact, running on cloud infrastructure you own. kolm.ai compiles the artifact, signs the deploy manifest, and pins the post-boot attestation. The artifact pulls down to your AWS, GCP, Azure, or Fly account and runs there. kolm.ai never holds the runtime, never sees request payloads on TEE targets, and never receives a copy of the artifact from your cloud.

This is the deploy mode for teams who need data residency, who answer to a regulator that requires the inference path to stay inside their own VPC, or who want to keep cloud lock-in optional. The artifact format is identical across local, BYOC, and airgap; the same signed binary runs in all three places.

For deployments that should never leave a single machine, see /airgap. For a hosted inference path that kolm.ai operates, see /cloud.

Five deploy modes

Pick one when you issue a script
Mode Target TEE Attestation format
fly Fly Machines, your Fly org No Docker image sha256:…
aws-nitro AWS Nitro Enclaves, your AWS account Yes COSE_Sign1 over CBOR · PCR0 = SHA-384 of EIF
gcp-cvm GCP Confidential VM, your GCP project Yes SEV-SNP 1184-byte report or Intel TDX quote
azure-cvm Azure Confidential VM, your Azure subscription Yes Microsoft Azure Attestation JWT, or raw SEV-SNP / TDX
docker Any Docker host you operate No Docker image sha256:…

TEE-backed targets keep request payloads encrypted in memory and produce a hardware-signed measurement of the running code. Non-TEE targets are still private from kolm.ai at the network level (we do not receive runtime traffic), but the host operator can read process memory.

TOFU verification

BYOC uses trust-on-first-use for attestation. The first attestation a deployment posts is pinned as the expected measurement; every later attestation must match or the deployment is flagged.

  • Parsers extract the workload measurement, the vendor claims, and the signing cert chain from the raw attestation payload.
  • The first valid attestation is recorded as the pin for that deployment.
  • A measurement mismatch on any later attestation surfaces in the dashboard and the receipt history. The runtime keeps serving; the operator decides whether to tear down or to accept the change.
  • Cryptographic verification of the vendor signing chain (AWS Nitro root, AMD VCEK, Intel PCS, Azure MAA JWKS) is an open item. Parsers expose the chain claims so the verifier can plug in without an API change.

CLI

Same flow from a terminal 
# list the targets the server supports today
kolm cloud targets

# issue a deploy script for a compiled artifact
kolm cloud deploy --target aws-nitro --artifact job_2026... --region us-east-1 --name acme-prod

# the CLI writes kolm-deploy-<id>.sh (mode 0700) and prints the enroll token
# review the script, then run it from your CI or your host
./kolm-deploy-<id>.sh

# on first boot the instance POSTs attestation back to kolm.ai
kolm cloud show <deployment_id>

# list every deployment, or tear one down
kolm cloud list
kolm cloud destroy <deployment_id>

Available target values: fly, aws-nitro, gcp-cvm, azure-cvm, docker. kolm cloud destroy removes the kolm.ai-side record; you still need to run your cloud teardown command to free the underlying resources.

Why BYOC

data residency

Inference traffic stays inside the cloud region you picked. The deploy script reads from your VPC; nothing routes back through kolm.ai at run time.

regulatory fit

HIPAA, GDPR, SR 11-7, EU AI Act, and similar regimes care about who can see the data and who can change the model. With BYOC and a TEE target, the runtime is auditable down to the PCR or SNP measurement.

vendor-independence

You own the artifact and you own the host. If kolm.ai disappeared tomorrow, the deployment keeps running and the receipt chain stays verifiable with the tenant receipt secret you already hold.

single artifact, many homes

The .kolm compiled for BYOC is the same file you would run locally or move into an airgapped network. One signed binary covers laptop, BYOC, and offline.

Trust model

    

    

      

        
01 · manifest

        
kolm.ai signs the deploy manifest (deploy_id + artifact_id + enroll_token + issued_at) with an HMAC anyone with the receipt secret can verify.

      

      

        
02 · deploy

        
You run the script. It pulls the artifact, builds a Docker image (or EIF for Nitro), and starts the runtime on your hardware.

      

      

        
03 · attest

        
On TEE targets the image POSTs the vendor-signed measurement (Nitro PCRs, GCP attestation token, Azure CVM SNP report). On non-TEE targets, an image SHA.

      

      

        
04 · payload-blind

        
TEE targets keep request payloads encrypted in memory; kolm.ai cannot see them. Non-TEE targets are private from us at the network level only.
