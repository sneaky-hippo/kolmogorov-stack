Every .kolm artifact is HMAC-signed over the components the current builder emits. Receipts can be verified from the artifact bytes. Base-model, LoRA, recall-index, and external-registry claims become production evidence only when the corresponding bridge ships.

01Architecture - data flow

The hosted preview accepts task specs, examples, and evals needed to compile. The current runtime surface is the Node CLI and MCP server. Customer-hosted compile and native offline runtimes are deployment targets that need contract review and benchmark evidence.

The compile pipeline runs in our cloud (or your VPC, on Enterprise). The runtime runs on your users' device. After compile, the only thing that crosses our boundary is the manifest hash - never the artifact, never the corpus, never the receipts.

In

Hosted preview submissions are service inputs. Retention, redaction, and regulated-data handling must be reviewed per deployment before sensitive data is in scope.

Encrypted at rest while in flight through the compile pipeline. Deleted post-compile. Eval set retained 30 days for K-score audit only, then purged.

Through

Provider keys should be pass-through and not persisted. Teacher outputs become compile evidence for the artifact; LoRA training is a bridge target until real adapters are emitted.

API key is pass-through; we never persist your key, prompts, or completions. The teacher's outputs become labels and dissolve into the LoRA.

Out

One signed file. You download it and can verify the receipt from the bytes you hold. Fully offline native model execution requires the runtime bridge and deployment tests.

One signed file. You download it. We delete the build inputs. Your users run it offline and the receipts they generate never leave their device.

02Three layers of provenance

Signed at compile.
Verified by receipt.

Each layer is independent. You can verify the artifact and receipt chain from the bytes you hold — offline, with a single command, no service required. External ledger anchoring ships as an Enterprise add-on.

Layer 01

HMAC-SHA-256 covers every emitted component: model pointer, recipes, eval set, manifest, and receipt body. Tampering with any byte breaks the chain.

kolm inspect prints the full chain. kolm verify re-runs the receipt against the artifact you hold — no key, no service.

$ kolm inspect support-triage.kolm
ok manifest.signature_alg hmac-sha256
ok receipt.signature_alg  hmac-sha256
ok anchors []
Layer 02

Registry record.

Public artifact metadata is published through the registry export endpoint and can be diffed against the artifact bytes you hold. Each entry carries the manifest hash and signature algorithm.

$ curl -fsSL https://kolm.ai/v1/registry/export
ok public recipe metadata export
Layer 03

External anchor extension.

The receipt schema carries an anchor field for transparency-log roots and third-party timestamps. Available as an Enterprise add-on; the field is empty by default on Free / Pro tiers.

$ kolm inspect support-triage.kolm
anchors  []
ok reserved extension field

05Disclosure

Scope: kolm.ai, the compile pipeline, the public registry, the runtime, the SDK, the CLI, RS-1 spec implementations.

Out of scope: social engineering, denial of service, spam, third-party services we don't control.
