kolm  /  frozen evaluation set

A frozen evaluation set is the only thing standing between a recipe and silent ship drift.

Every K-score on the page, every compile-time gate, every binder row "K=0.917 (passed)" rests on one assumption: the rows the score was computed against are rows the recipe never saw at training time. kolm makes that assumption a contract. The eval set is hash-pinned at compile time, the train and holdout partitions are disjoint under SHA-256, the K-score's accuracy axis is computed against the frozen holdout, and a contamination check forces ship-fail if anything overlaps. A pre-Wave-144 build derived eval cases from the recipe itself (input_hash == recipes_json_hash by construction); the seed gate makes that tautological path a hard reject.

The three frozen-eval invariants.

A holdout is "frozen" when all three invariants hold at the same time. Any one of them missing collapses the chain of reasoning that turns a published K-score into a buyer-defensible claim. The seed gate (binder check #7 in src/binder.js) re-asserts every one at verify time.

invariant 01 · disjoint

train ∩ holdout = ∅

Hash-set membership check via SHA-256 of the canonical input. The build pipeline buckets every row by sha256(canonicalInput(row.input) + ' ' + split_seed) % 1000; bucket < cutoff goes to holdout, the rest to train. Same seeds.jsonl plus same split_seed produces an identical split. A verifier rebuilds the split from the two recorded inputs and confirms train_hash ∩ holdout_hash = ∅ before any signature is checked.

YES · src/seeds.js splitSeeds + leakageReport
invariant 02 · pinned

holdout hash recorded at compile time

The holdout's canonical-JSON hash is written into manifest.seed_provenance.holdout_hash at compile time and bound through the receipt chain into artifact_hash. Check #7 ("Seed gate (train/holdout independence)") in src/binder.js rejects any post-build modification: the verifier loads seeds.jsonl, rebuilds the split with the recorded split_seed, recomputes the canonical hash, and confirms byte-identical match against the manifest claim. Any tamper breaks the artifact_hash and every downstream signature including the Rekor anchor.

YES · manifest.seed_provenance.holdout_hash + check #7
invariant 03 · sufficient

holdout size ≥ ceil(N_train / 8) and ≥ 50 examples

An eval set of three rows is technically disjoint and technically pinned, but the K-score it produces is noise. The recommended production minimum is the larger of 50 examples and ceil(N_train / 8) so the holdout grows with the corpus. Below the threshold the binder emits a warn pill labeled "sample check" and the K-score loses its production gate role until enough captured rows accumulate. production_ready=true with a small holdout is inconsistent and is downgraded to a hard fail at verify time.

YES · production threshold in src/spec-compile.js + check #7

The three invariants compose. Disjoint alone is meaningless if you can quietly substitute a different holdout after the score is published. Pinned alone is meaningless if train and holdout overlap. Sufficient alone is meaningless if either of the other two fails. The seed gate is a single check (#7) that enforces all three together so an auditor reading the binder sees one row, one verdict, one detail string covering the entire eval-independence claim.

What "frozen" means in the receipt chain.

Every effect a tenant relies on is captured by a manifest field and a verifier check that re-derives that field. The seed split contributes five fields to manifest.seed_provenance. Block hashes bind into artifact_hash_input.seed_provenance_hash; any tamper drifts artifact_hash.

split_seedstringcheck #7The seed string passed to the bucket function. Recorded so a third party with the seeds.jsonl can reproduce the train/holdout partition exactly. Default kolm-default-split-seed-v1; override with --split-seed.

All eight fields ship in manifest.seed_provenance and are bound into artifact_hash_input so the seed gate is part of the artifact identity, not a side-channel claim. The verifier emits one binder row covering all eight fields plus the disjointness check plus the size floor; a single fail anywhere in the chain blocks ship.

The contamination check, in pseudocode.

The disjointness invariant is a hash-set intersection. The build pipeline computes it once at compile time; the verifier recomputes it at ingest from the recorded seeds.jsonl and split_seed. Three normalization rules turn surface variation in seeds.jsonl into a stable canonical form before hashing.

# src/seeds.js - the disjointness contract, as enforced by # splitSeeds() + leakageReport() + the seed gate (check #7). function canonical_input(value): # Stable string form for hashing. Strings hash by their # string form. Objects hash by canonical JSON (sorted keys, # no insignificant whitespace) so {a:1,b:2} hashes the same # as {b:2,a:1}. Necessary because the split bucket is # sha256(canonicalInput + split_seed) and structured inputs # must be stable across machines. if value is string: return value if value is primitive: return str(value) return canonical_json(value) function split_rows(rows, split_seed, holdout_ratio=0.2): buckets = 1000 cutoff = floor(holdout_ratio * buckets) train, holdout = [], [] for row in rows: h = SHA-256(canonical_input(row.input) + ' ' + split_seed) bucket = int(h[:8], 16) % buckets if bucket < cutoff: holdout.append(row) else: train.append(row) return { train, holdout } function disjointness_check(train, holdout): # Hash-set membership. The intersection MUST be empty. train_input_hashes = { SHA-256(canonical_input(r.input)) for r in train } train_output_hashes = { SHA-256(canonical_input(r.expected)) for r in train } for r in holdout: if SHA-256(canonical_input(r.input)) in train_input_hashes: return CONTAMINATED if SHA-256(canonical_input(r.expected)) in train_output_hashes: return CONTAMINATED return CLEAN

The normalization rules canonical_input applies before hashing: stringify primitives, sort keys recursively on objects, no insignificant whitespace, UTF-8. Two semantically-identical structured inputs hash the same way regardless of key order; a row with input {a:1,b:2} in train and a row with input {b:2,a:1} in holdout is detected as overlap even though the surface JSON differs.

Documented invariant, enforcement in src/seeds.js (splitSeeds + leakageReport) and surfaced through the binder check in src/binder.js. The leakage report also runs a Jaccard-bigram similarity check at threshold 0.85 to catch near-duplicate text that survives exact-hash matching (lowercase + whitespace tokenization); near-duplicate pairs downgrade the seed gate verdict and surface in the binder detail string.

Grouped-key leakage is the third layer: if a row in train carries a tag like member_id:12345 and a row in holdout carries the same tag, the binder emits a warn even if the surface text differs. Same-entity leakage (the same patient, same claim, same customer appearing on both sides) is the most common silent contamination in healthcare and finance corpora; the grouping-key check catches it before ship.

The four ship-block scenarios.

A failed frozen-eval invariant exits the build with a non-zero status. There is no --force, no --skip-checks, no environment variable that silences the gate. The exit-code contract is the integration point with CI and with cron-driven re-distillation.

scenario a

train and holdout overlap

The leakage report finds input_overlap_count > 0 or output_overlap_count > 0. The binder detail string names the count and the suggested fix ("deduplicate seeds.jsonl or pick a different split_seed"). Exit non-zero, no signature emitted.

check #7: fail $? != 0

scenario b

holdout size below threshold

The holdout has fewer than ceil(N_train / 8) rows or fewer than the recommended 50 examples. If the recipe declares production_ready=true, the verdict is fail and ship is blocked. If production_ready is unset, the verdict is warn and the K-score is surfaced with a "sample check" label, not a production gate.

check #7: fail $? != 0

scenario c

holdout hash changed since last verify

The verifier rebuilds the split from the on-disk seeds.jsonl and the recorded split_seed, recomputes the canonical hash, and compares against manifest.seed_provenance.holdout_hash. Any byte difference breaks the artifact_hash and fails every downstream signature check including HMAC, Ed25519, and Rekor.

check #7: fail $? != 0

scenario d

K-score on frozen holdout below gate

The K-score's accuracy axis A is computed against the frozen holdout. If the composite K is below the recipe-declared min_k floor, the compile fails before signatures are emitted. The recipe author either improves the recipe, adds captured rows to seeds.jsonl, or lowers min_k (recorded as a manifest change a verifier can audit).

K-score gate: fail $? != 0

All four scenarios share the same exit-code contract: non-zero status, no signature emitted, the partial artifact never reaches the receipt chain. CI sees a build failure; cron-driven re-distillation sees a job failure and surfaces it through the same mail or webhook channel the kolm drift detect breach uses. The integration point is the exit code, not a parseable error format.

Override audit: there is no --force flag.

The frozen-eval invariants are designed so the only path forward after a failure is to fix the underlying cause or to roll the artifact version with a supersession block. Silent overrides do not exist by construction.

If a recipe author needs to ship despite a failed seed gate (for example, the holdout truly is small because the workload is a long-tail edge case, or the K-score floor needs to drop because traffic moved), the path is to compile a NEW manifest version with explicit acknowledgement: lower min_k, set production_ready=false with a documented reason, or rebuild against a wider seeds.jsonl. If the new artifact is intended to retire the previous one, attach a supersession block (shipped in wave 167, see RS-1 §7.15) with reason recipe_revision or tenant_request plus authorized_by and notes. The supersession is itself a receipt-chain entry, bound into artifact_hash_input.supersession_hash, walkable by any third-party verifier backward to genesis. Every override leaves a trail; no override is a config silencing.

This is the same architectural commitment as the rest of the kolm stack: every effect a tenant relies on is captured by a manifest field and a verifier check that re-derives that field. A signature you can mute with a flag is a signature that signs nothing.

The CLI flow, end to end.

Four commands compose the frozen-eval lifecycle. The split itself happens inside kolm compile when --seeds is passed, so the "kolm seeds split" step in the flow below is a conceptual stage name, not a standalone subcommand. The other three (kolm eval, kolm verify, kolm inspect) are real verbs documented in cli/kolm.js.

01 · split conceptual stage

kolm seeds split

Stage name for the deterministic 80/20 partition that runs inside kolm compile --seeds <file>.jsonl. Same seeds.jsonl plus same split_seed produces an identical split across machines. To inspect what the split would produce without compiling, run kolm seeds validate and read the row count.

$ kolm seeds validate seeds.jsonl
  rows:     1240 (canonical=1240, legacy=0)
  skipped:  0
  duplicates: 0

$ kolm compile spec.json --seeds seeds.jsonl \
    --holdout-ratio 0.2 --out my-recipe.kolm

  seeds -> split -> train (992) + holdout (248)
  split_seed:   kolm-default-split-seed-v1
  train_hash:   a1b2c3d4...
  holdout_hash: e5f60718...
  leakage:      0 input, 0 output, 0 near-dup, 0 grouped
  seed gate: pass
02 · eval against frozen holdout

kolm eval

Runs the compiled artifact against the embedded eval cases (the frozen holdout, by default). Pass --examples <file>.jsonl to evaluate against an external corpus instead (the A/B path). Pass --trace to surface every failing case.

03 · verify the chain

kolm verify

Runs the full binder check suite including check #7 ("Seed gate"). Pass --binder <out>.html to emit a printable compliance report. Exits non-zero if any check fails. The seed gate's detail string names the train/holdout counts, the comparator, the eval_source, and the leading 12 hex of the split_seed.

Extending the disjoint check to external holdouts.

The seed gate is the floor. The same canonical-hash disjointness mechanism extends upward to two more receipt-chain layers: external public benchmarks (Presidio, i2b2 2014, MIMIC-III, CMS / NCQA) and adversarial cross-family holdouts.

Wave 164 (N+3 + N+4) extended the receipt chain with manifest.external_holdout_provenance. An external holdout is a separate JSONL whose hash is anchored at compile time the same way the tenant seeds holdout is; the verifier (check #20) re-anchors the file at ingest and demands byte-identical match. The same SHA-256-of-canonical-input disjointness rule extends through the recipe: a train row whose input hashes the same as any external-holdout input is rejected. The full surface is documented in RS-1 §7.12.

Tenant shadow corpora (wave 165, N+5, manifest.tenant_shadow_corpus_provenance, check #21) layer on top of the same mechanism. The K-score's R axis (real-world independence) is computed against external_holdout; the F axis against subgroup tags in the holdout; the Z axis (drift) against a fresh holdout collected post-ship. All four axes share the same SHA-256 hash-pinning contract the tenant seeds holdout established.

The receipt chain, in order.

The seed split is the second link in the receipt chain (after spec). Every later link is conditional on it: train fed the recipe synthesis, recipes fed the evals, evals computed the K-score, the K-score gated ship. Any tamper at the seed split breaks every signature downstream including the Rekor anchor.

spec → seeds → split → train → recipes → evals → external_holdout → tenant_shadow → auditor_attestation → supersession → drift_report → export → signatures → rekor
  • seeds · canonical-JSON hash of all normalized rows. Anchored in manifest.seed_provenance.seeds_hash. The verifier reloads seeds.jsonl from disk, normalizes (canonical or legacy schema), and demands byte-identical match.
  • split · deterministic 80/20 partition by SHA-256(canonical_input + split_seed) % 1000. Same seeds.jsonl plus same split_seed produces an identical split. Verifier rebuilds the partition and confirms train_hash and holdout_hash match.
  • evals · K-score's accuracy axis A is computed by running the compiled artifact over the holdout inputs and comparing against holdout outputs using the recipe-declared comparator. A is by construction a measurement against UNSEEN rows.
  • artifact_hash · the final hash over the canonical artifact_hash_input object. seed_provenance_hash binds in; any tamper at the seed split drifts artifact_hash and breaks every downstream signature including the Rekor transparency log entry.

Where this fits in the wider stack.

k-score · explained

What the K-score actually measures

The K-score's accuracy axis A is computed against the frozen holdout the seed gate enforces. Five V1 axes (A / S / L / C / V) plus six V2 axes (R / F / E / Z / T) compose the published number. The K-score is meaningful only if the holdout is frozen.

 drift · lifecycle

Drift detection + supersession chain

The frozen holdout grounds the genesis K-score. The drift detection cron (wave 167, check #24) snapshots K-score axes against the same frozen baseline and emits one signal per axis. A breach triggers re-distillation with a supersession block.

 recipe classes · taxonomy

The four recipe classes

rule, compiled_rule, distilled_model, and template. The seed gate fires the same way for every class; the K-score axes that compose to the composite shift per class (T axis is meaningful only for distilled_model, S is trivially 1.0 for rule and compiled_rule).

 spec · canonical

RS-1 v2.1 §4 + §6 + §7.12

The normative spec sections. §4 defines the seeds.jsonl train/holdout split. §6 defines the K-score and how A is computed against the holdout. §7.12 extends the disjointness check to external + adversarial holdouts.
