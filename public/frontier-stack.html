kolm  /  frontier stack

Every technique kolm uses, at every step.

The list below is the working inventory of methods the compiler reaches for. Each technique has a one-sentence purpose and a link to the implementation or the paper. The ones tagged default run unless you override them; the rest are opt-in. Stack revs continuously; this page tracks it.

01 · DESCRIBE

Tell kolm what you want, in plain english.

The compile starts from a written spec. We translate it into a structured task graph, a tool schema if there are tools, and an output spec the K-score gate will hold the artifact to.

Spec parser default

Natural-language description → structured task graph with input shape, output shape, and acceptance rules.

/research/provenance-data-generation

Tool schema synthesis default

If the task uses tools we infer JSON-schema signatures from the description and validate at compile time and at every call.

/research/function-calling-training

Output spec extraction default

Structured outputs are pulled from the spec into a JSON-schema the gate uses for shape conformance scoring.

/research/output-specs

Guarded prompts opt-in

Insert constitutional clauses, refusal templates, or persona constraints before distillation. Stored on the recipe, not the prompt.

/research/guarded-prompts
02 · DISTILL

Borrow knowledge from a frontier teacher.

A teacher model writes the data the student will learn from. The cheapest and most predictable lever in the whole pipeline; the rest of the stack is largely consequence of doing this part well.

Teacher trace default

Teacher solves seed prompts with rationales. Both the answers and the reasoning are kept.

/research/teacher-trace

Synthetic expansion default

Seed prompts get rephrased, recombined, and stretched by the teacher into a balanced, deduplicated training corpus.

/research/synthetic-data

Response distillation default

Soft-label distillation against teacher logits where the teacher exposes them; falls back to hard-label SFT otherwise.

/research/response-distill

Frontier prompt cache default

Distillation calls re-use the teacher prompt cache. Reads cost about 10% of writes. Surfaces as cache_hit on every receipt.

/research/prompt-cache

Rationale distillation opt-in

Student is trained to reproduce the teacher reasoning as well as the answer. Larger compute, better OOD.

/research/rationale-distill

Tool-call traces opt-in

Teacher records full tool dialogues; student learns the agent loop, not just the final response.

/research/tool-traces
03 · ALIGN

Align the student to the spec, not just the corpus.

Supervised fine-tuning gives capability; preference alignment gives behavior. We pick the objective by task: pairwise preferences for classifiers, KTO for sparse-reward, GRPO for multi-step reasoning, online-DPO when judges are fast.

SFT default

Standard cross-entropy fine-tuning on the distilled corpus. Always the first pass.

/research/sft

DPO default

Direct preference optimization on pairwise teacher comparisons. Stable, cheap, well-understood.

/research/dpo

ORPO opt-in

Reference-free preference loss. Lower VRAM. Useful when reference model checkpoints would not fit.

/research/orpo

GRPO opt-in

Group-relative policy optimization for multi-step tasks. Used when the K-score includes a per-step component.

/research/grpo

KTO opt-in

Kahneman-Tversky objective. Works on unpaired "good/bad" labels when you cannot afford pairwise comparisons.

/research/kto

Online-DPO opt-in

Preferences are sampled from a live judge during training. Higher cost, sharper alignment, used on K-score-critical tasks.

/research/online-dpo

LoRA / QLoRA adapters opt-in

Parameter-efficient training when the base model is shared across tasks or when GPU minutes matter more than fidelity.

/research/adapter-merging
04 · EVALUATE

Hold the artifact to the spec.

Every compile ends with a gate. K-score is the headline number; underneath it is a mixture of judges, a process-reward model when sequences matter, and an eval-set drift check that catches when the holdout has aged out.

K-score default

Composite quality score in [0,1]. Mix of pairwise judge wins, output-spec conformance, and OOD generalization. Per-artifact, never global.

/k-score

Mixture of judges default

Three frontier judges score every eval sample. We use median to filter judge bias; tied votes go to a fourth.

/research/judges

Held-out task split default

A fraction of the seed prompts is held back from distillation and used only at gate time. Replenished at every compile.

/research/holdout

Eval-set drift default

If the holdout starts looking too much like the corpus, the gate refuses to grade and asks for fresh seeds.

/research/eval-drift

Process reward model opt-in

For multi-step reasoning we train a per-step reward and score sequences, not endpoints. Adds compile time, catches silent partial credit.

/research/prm

Adversarial probes opt-in

Red-team prompts for safety, jailbreaks, prompt injection, leakage. Pass-rate is a gate component when configured.

/research/probes
05 · SIGN

Pack and seal.

The artifact is one signed file. The signature anchors the recipe, the weights, the gate result, and an SBOM so the operator can prove what they are running and where every byte came from.

.kolm wire format default

Deterministic zip of recipe, weights or adapter, gate result, SBOM, and license. Fully described in the spec.

/spec/rs-1

4-ring HMAC receipt default

Pre-image → derived → execution → seal. Each ring depends on the previous. Verifier checks all four offline.

/research/receipt-chain

SBOM default

Source corpus hashes, model base, library versions, and license posture, embedded in the .kolm and downloadable.

/research/sbom

Tenant-rotated key default

HMAC secret is per tenant. Rotated on schedule with a 90-day verify-only window for old receipts.

/research/receipt-chains
06 · RUN

Inference, anywhere.

The artifact does not assume a runtime. Pick the one that fits the deployment. The bridge produces the same receipt regardless of the engine underneath.

vLLM default

The default server for CUDA-capable hosts. Paged-attention KV cache, continuous batching, sharded tensor parallel.

vllm-project

sglang opt-in

RadixAttention prefix cache and structured-output decoding. We reach for it when sequence outputs are dominant.

sgl-project

TensorRT-LLM opt-in

NVIDIA TRT-LLM kernels for high-throughput single-host serving. Used when latency budget is tightest.

NVIDIA/TensorRT-LLM

MLX opt-in

Apple Silicon serving. The artifact is the same; the runtime is local.

ml-explore/mlx

llama.cpp opt-in

GGUF on CPU or edge GPU. Slower; the choice when there is no other choice.

ggerganov/llama.cpp

Customer-hosted bridge default

The runtime that talks to your tenants. Produces the receipt, enforces the gate, holds your HMAC key.

/research/byoc-deploy-modes
