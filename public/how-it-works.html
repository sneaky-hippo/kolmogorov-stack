How it works

Compile, end-to-end.

The compiler is eight stages: gather, spec, synthesize, k-sample, verify, fit, observe, sign. The frontier model is the teacher. The artifact is the student. The receipt is the audit row. This page reads like a paper because it should - the only credible answer to "how does it work" is one that an engineer can implement.

Eight stages. Frontier as teacher. Artifact as student. Receipt as audit row.

01

Gather: collect the gold examples and the corpus.

The compile starts with what you bring. 30–200 input/output pairs (the gold set) and the directory of related documents (the corpus). The compiler holds 20% of the gold out for the eval set and never peeks at it during synthesis or training.

Inputs

  • examples.jsonl - one input/output pair per line
  • ./corpus/ - text, PDFs, images, audio, video. Mixed modalities are auto-detected (magika) and routed to the right embedder
  • policy.md (optional) - refusal patterns, formatting constraints, regulatory language
  • frontier API key - Anthropic, OpenAI, Google, or your own. Used as the teacher.

What the compiler does

  • parses, deduplicates, normalizes encoding
  • splits 80/20 train/eval with a deterministic seed (so re-compiles produce identical splits)
  • indexes the corpus into a tenant-scoped sqlite-vec namespace
  • writes the deterministic data_hash into the manifest - same inputs, same hash, every time
02

Spec: derive the verifier from the examples.

A task is only compilable if there's a deterministic check on the output. The compiler synthesizes that check from your examples: a JSON schema, a regex, a custom function - or a learned classifier when the output is free-form. This is what makes the K-score quantitative instead of vibes.

Verifier types

  • schema - JSON Schema synthesized from output shape (most common)
  • regex - for structured strings (patient IDs, claim numbers, redactions)
  • function - user-supplied verify.js or verify.py for arbitrary checks
  • classifier - a small learned head (logistic regression on frontier embeddings) when outputs are free-form - trained on the gold set
  • composite - AND/OR combinations of the above

Output

verifier.json - deterministic, hashed, embedded in the artifact.
score(input, output) -> [0,1]
the K-score is the mean over the held-out eval set.
03

Synthesize: draft the recipe pack from the examples.

Most tokens in a structured task are deterministic given the prefix: opening brackets, field names, fixed framings. The compiler observes those patterns across the gold set and emits a recipe pack - a deterministic draft table indexed by prefix shape and by the embedding of the recalled context. This is the speculative-decoding moat: at run time, the recipe pack drafts the boring tokens, the LoRA fills in the substantive ones.

What ships in the pack

  • prefix-shape entries - e.g., {"triage": always followed by one of "urgent"|"routine"|"escalate"
  • embedding-shape entries - e.g., when the recalled context resembles "ER intake form", the field order is fixed
  • fallback - if no recipe matches, the LoRA decodes from scratch (no quality cost)
04

k-sample: query the frontier as the teacher.

For each training example, the compiler asks the frontier model k times (default k=4) for the answer. The verifier scores all k outputs. The winner advances; the losers are discarded. This is verified inference: the labels we use to train the LoRA are not "what the frontier said" - they are "what the frontier said that the verifier accepts."

Why k-sampling instead of a single call

Single frontier calls are unreliable on the long tail. k=4 with deterministic verification gets you to ≥0.95 label correctness on most structured tasks. The cost is 4× the frontier API spend per training example - amortized to zero after compile, because every call against the artifact thereafter is local.

Recall injection

If a corpus namespace was provided, every k-sample request is grounded in the top-k recalled chunks. The frontier never sees the whole corpus; it sees the relevant chunks per example, retrieved by embedding similarity.

05

verify: the deterministic gate.

Every k-sampled output is run through the verifier. The score determines acceptance. If no sample passes, the example is dropped from training (we'd rather train on less than train on a wrong label). The fraction of examples that produced a passing sample is the label coverage; it goes into the manifest.

Failure handling

  • if <80% of examples produce passing samples, the compile pauses and asks for more or higher-quality gold
  • if the frontier consistently fails the verifier, you get a "the frontier cannot do this task at this gate" error - the compile fails fast, you re-spec the task
06

fit: train the LoRA.

The verified label set is the training data for a LoRA delta over the chosen base model. Training runs server-side (or on a customer-hosted bridge) on a single H100 / 5090; typical compile takes 8–25 minutes for a 3B-class base.

Hyperparameters & bases

basesQwen 2.5 / 3.x (1B, 3B, 7B, 14B), Llama-3 8B, Phi-3 3.8B, Hermes-3 8B, Gemma 2 9B. All open weights, all INT4-quantizable.
LoRA rank16 default, 32 for high-coverage tasks. Adapter ≤ 100MB at rank 16 on a 7B base.
quantizationINT4 GGUF for portability (CPU / phone), bf16 for GPU. Both produced in one compile.
training3 to 5 epochs, bf16 mixed-precision, gradient checkpointing on a single H100 or 5090.

What "Hermes-class on your task" actually means

We don't claim 3B-with-LoRA matches a 70B raw on arbitrary inputs. We claim - and measure - that on the task you compiled (your eval set, your input distribution), the resulting artifact achieves ≥K-score-floor accuracy. The eval set is held out from training; you can re-run it offline at any time.

07

observe: harden the recipe pack from the trained model.

After the LoRA is trained, the compiler runs it across the eval set with verbose logging on. Tokens that the LoRA emits with ≥99% confidence and that are predicted by the recipe pack are confirmed. Tokens predicted by the recipe pack but where the LoRA disagrees are removed. Tokens emitted with high confidence but not in the pack are candidates for inclusion in the next compile. The pack tightens with every compile.

This is the compounding loop

Every public compile feeds candidate patterns to the public registry. Every registry growth shrinks the marginal observation budget for the next compile. Over time, the pack covers a higher fraction of every task's structured-token surface, which means smaller models can draft faster, which means cheaper compiles, which means more compiles. The flywheel is real.

08

sign: bundle and seal.

The compiler bundles model + LoRA + recipes + verifier + recall index + manifest into a single signed zip. The signature is HMAC-SHA256 over the canonical hash chain of every component. The receipt secret is held by the registry; the public key is published. Anyone can verify a receipt offline against the published key.

What it means at the receiving end

  • any output the artifact produces ships with a per-call receipt over (input_hash, output_hash, manifest_sha)
  • any auditor with the published public key can re-verify any receipt offline, without contacting kolm
  • artifact tampering invalidates the signature; the runtime refuses to load it
  • frontier-model drift over time has no effect - the artifact is sealed; output is reproducible
09Properties

Why this shape produces what it does.

A compiled artifact is small (3B + 100MB LoRA + 10MB recipes ≈ 2–3GB), portable (runs on phone, laptop, edge box), reproducible (sealed manifest), auditable (HMAC receipt per output), and offline-first (no run-time network). The trade-offs are explicit: it does not generalize beyond the compiled task, and the K-score is the contract on its accuracy.

What it is not

  • not a quantized 70B running on a phone - it's a 3B-class artifact that behaves 70B-class on the compiled task
  • not memory or context-management - the corpus is grounding, not personality
  • not a frontier-model wrapper - the frontier is used at compile time only; runtime is local
