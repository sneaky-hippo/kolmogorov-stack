The open spec for AI behavior.

Recipe Spec (RS-1) is a portable, MIT-licensed standard for declaring AI behavior as deterministic, signed, executable contracts. Examples in. Behavior out. A cryptographic receipt on every run. Anyone can implement it. Anyone can verify it. No vendor lock-in.

The thesis in one paragraph

Today, every AI behavior — a classifier, a router, a filter — lives inside a stochastic black box. You can't pin it. You can't prove what ran. You can't verify it offline. Recipe Spec changes that. A recipe is a tiny piece of executable code with a published source hash, a JSON Schema for inputs and outputs, and a receipt that any third party can verify. It runs in microseconds, forever, on any hardware. It's the layer JSON Schema, OCI, and TCP/IP became — boring infrastructure under everything else.

Five primitives

1. Recipe

A signed function

An ID, a name, a version, a source hash, and the executable source. JS or WASM. Deterministic. Tiny.

2. Schema

JSON Schema in/out

Optional input and output schemas pin the contract. Outputs that don't validate are rejected before they leave the runtime.

3. Examples

The training corpus

Positives and negatives the recipe was synthesized against. Stored alongside source. Anyone can re-run them.

4. Receipt

Cryptographic proof

Every run produces an HMAC-signed JSON receipt: source_hash, input_hash, output_hash, runtime version, timestamp. Verifiable forever.

5. Conformance

Identical bytes everywhere

For a given (source_hash, input_hash), every conforming runtime MUST produce the same output_hash. The test suite enforces it.

6. Registry

A public catalog

Recipes ship with metadata (description, tags, evaluation scores) so they're discoverable. The registry is the npm/HuggingFace of behavior.

Recipe — the wire format

A recipe is a JSON document. This is the canonical shape:

{
  "id": "cpt_abc123",
  "name": "is_spam",
  "version": "1.0.0",
  "source_hash": "8f3d…16hex",
  "source": "function (x) { return /unsubscribe|free \\$\\$/i.test(x); }",
  "schema": {
    "input":  { "type": "string" },
    "output": { "type": "boolean" }
  },
  "examples": [
    { "input": "Hi, are we still on for Tuesday?", "expected": false },
    { "input": "FREE $$$ click here unsubscribe", "expected": true }
  ],
  "evaluation": {
    "quality_score":      0.96,
    "pass_rate_positive": 1.00,
    "reject_rate_negative": 1.00,
    "latency_p50_us":     35
  },
  "lineage": {
    "synthesized_from_n": 8,
    "attempts_n": 3,
    "model": "claude-opus-4-7"
  },
  "license": "MIT",
  "created_at": "2026-05-06T17:42:11Z"
}

Receipt — the proof format

Every POST /v1/run returns a receipt object alongside the output. Receipts are stable JSON, signed with HMAC-SHA256, and verifiable by anyone. The shape:

{
  "spec":            "rs-1",
  "source_hash":     "8f3d…16hex",
  "input_hash":      "sha256/64hex of canonical JSON of input",
  "output_hash":     "sha256/64hex of canonical JSON of output",
  "version_id":      "ver_xyz789",
  "runtime_version": "0.3.0",
  "issued_at":       "2026-05-06T17:42:11.443Z",
  "cache_hit":       false,
  "latency_us":      35,
  "hmac":            "f8a1…64hex (HMAC-SHA256 over the canonical-JSON payload)"
}

Re-verifiable via POST /v1/receipts/verify. Pass back the receipt — and optionally the original input/output — and the endpoint returns { valid: true|false, checks: { signature, input_match, output_match } }.

Canonical JSON

Hashes are only stable if everyone serializes the same way. RS-1 mandates canonical JSON:

  • Object keys sorted lexicographically at every depth.
  • Numbers, strings, booleans, null, arrays, objects only — no NaN, no Infinity, no undefined.
  • No trailing whitespace, no insignificant whitespace.
  • UTF-8 encoded.

Reference implementation: canonicalJson(v) in src/router.js (50 lines, public domain).

Why a spec, not just a product

Approach What you get What you lose
Closed product (Cohere, Helicone) One vendor's runtime Lock-in. Migration cost. No third-party verification.
Open spec (RS-1) Many runtimes. Public registry. Cryptographic receipts. Nothing — we still earn on hosting, attestation, fine-tunes.
Why this works Same way JSON Schema, OCI containers, sigstore, OAuth 2.0 work — the spec is free, the impls compete on quality. —

Conformance — claim RS-1

Any team can build a runtime that claims RS-1 conformance. The requirements:

  1. Implement POST /v1/synthesize, POST /v1/run, POST /v1/receipts/verify, GET /v1/spec.
  2. For any (source_hash, input_hash), produce the same output_hash as the reference implementation.
  3. Pass the public conformance suite (200+ recipes × 10+ inputs each, plus determinism + canonical-JSON edge cases).
  4. Issue receipts that re-verify under the public canonical-JSON algorithm.

Conformance gives you the right to ship the RS-1 compliant badge. It also means recipes published in our registry run unchanged on your runtime — and vice versa.
