The .kolm spec.

The contract that makes one .kolm byte-identical to another given the same inputs, signed end-to-end, and runnable on any device that can read the format. Open, MIT-licensed, small enough to read in one sitting.

Signature scheme

RS-1 v0.1 is intentionally narrow on the cryptography. One scheme ships today; one is on the v0.2 roadmap.

VersionSchemeCoverageVerification
v0.1 · currentHMAC-SHA256 over canonical JSONReceipt chain (4 rings) + manifest body signatureIssuer, or any party that holds the shared tenant receipt secret
v0.2 · roadmaped25519 over canonical JSONAdditional signature alongside HMAC, not a replacementAnyone with the publisher's ed25519 public key. No shared secret required.

Scroll horizontally →

The v0.1 chain is reproducible and tamper-evident under the issuer-or-shared-secret trust model that fits a single org or a contractually bound pair. ed25519 in v0.2 targets cross-org artifact distribution: a registry, a marketplace, or a partner network where the receiver should not have to hold the issuer's HMAC key to verify the signature.

Detached signatures, deterministic canonical-JSON encoding, and key rotation through the registry are open RFC items; track them under rs-2 on the roadmap above.

RFC process

RS-1 is the floor; everything beyond gets a numbered RFC.

  • Open a discussion on GitHub Discussions. One paragraph: what changes, what breaks.
  • Draft a PR against public/docs/ with the new spec as rs-N.md and a JSON Schema if applicable. Mark the prose as draft in the front matter.
  • Iterate until the breaking-change column is empty or unanimous. Two reviewers sign off.
  • Promote by removing the draft marker and bumping the registry signature key version. The reproducibility floor (§6 of RS-1) holds across the bump.

If you ship a runtime that implements an RFC before it lands, that is fine. Flag the artifact with "spec": "rs-N-rcM" so legacy runtimes refuse it cleanly.

Why a spec at all

A signed binary you cannot reproduce is trust theater. The .kolm format earns its weight when a holder of the inputs and the shared tenant receipt secret can re-derive the chain, validate the manifest, and re-run that artifact's K-score to the same numbers. Each .kolm carries its own eval set and its own K, so the recomputation is byte-deterministic for that file. That is what RS-1 buys today: reproducibility plus issuer-or-shared-secret verifiability. Independent public-key (Ed25519) verification across organizations is on the rs-2/rs-3 roadmap.

License

All specs in this repository are published under MIT. The intent is that any runtime (ours, a competitor's, an academic project) can implement the spec without asking permission. The receipt anchor (the registry public key) is the only externally-mutable input; everything else is self-describing.
