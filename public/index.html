how this is different

kolm vs. the things you already use.

vs Ollama

Ollama runs models. kolm compiles tasks.

Ollama gives you Llama-on-disk. kolm gives you a task-specific artifact: base model + LoRA + recipes + recall index + verifier — signed.
vs RAG

RAG is a runtime. kolm is a release.

RAG re-indexes at every query and stays online. kolm bakes the corpus and the verifier into the artifact, so the runtime is hermetic.
vs fine-tuning

Fine-tuning gives you weights. kolm gives you a release.

A LoRA without a verifier, recall, policy, and signature is not a release artifact — it's a checkpoint. kolm is the layer above.
vs frontier API

05primitives

One artifact. Four jobs.

The product should feel like one command, not a stack of ML chores. The internal pieces stay visible enough for engineering, security, and legal to trust the build.

$ kolm compile

Freeze the task.

Capture the task, examples, base model, policy, and corpus namespace in a signed manifest.

$ kolm compile --data

Attach context.

Package searchable context with the artifact so local runs can cite the approved source material.

$ kolm eval

Gate the release.

Re-run embedded tests, recompute K-score, and reject artifacts that fall under the threshold.

$ kolm run

Execute locally.

Run the artifact in the product boundary and emit a receipt that can be checked later.

08against the alternatives

Why a compiler, not a wrapper.

Most ways of shipping AI keep the cloud, the model, or the data outside your boundary. kolm collapses the three problems into one signed file you own. Here is what you trade vs each adjacent path.

PathWhat you keepWhat kolm changes
Cloud APIs
OpenAI, Anthropic, Bedrock		 Per-call inference. Network dependency. Per-token billing forever. Data leaves your machine on every call. No portable artifact. Pay the frontier teacher once at compile. Run inference locally at zero marginal cost. ~27× cheaper at 100k MAU. Data stays on the device. Output is signed.
Hosted fine-tune
Together, HF, Bedrock fine-tune		 A model you cannot move. Vendor-hosted weights. No quality gate. No portability across iOS, Android, edge, server. One signed file you own. K-score gate at build. Same artifact runs on phone, laptop, edge, VPC. No vendor lock.
On-prem inference
vLLM, Ollama, MLC, llama.cpp		 A general-purpose runtime. You build the eval pipeline, the LoRA pipeline, the recall index, the K-score, the receipt chain. Task-specific LoRA + recipe pack + multimodal recall + K-score + signed receipts, all in one artifact. Compiler does the assembly.
RAG / orchestration
LangChain, LlamaIndex		 Cloud calls every request. Glue you maintain. The intelligence stays in someone else's API. Compile the policy, the retrieval, and the model into one boundary-local artifact. Receipts span retrieval and inference. No glue at runtime.
DIY edge stack
CoreML, ExecuTorch, TFLite		 Three runtimes to ship and three quantization paths to maintain. No verification layer. No registry. One .kolm across iOS, Android, web, server. RS-1 spec is open. Verification + registry shipped.
