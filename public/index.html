the problem

Every requests.post("api.openai.com") in your code is a liability.

For cost. For privacy. For reproducibility. For compliance. For continuity. Frontier APIs are great teachers and terrible production dependencies. The 90%-deterministic part of your AI feature is paying frontier prices, leaving your network on every call, and breaking the moment the vendor ships a new model version.

cost
$0.04 per call for a job that's 90% deterministic. The same task compiled to a .kolm artifact runs at $0/token forever.
privacy
PHI, PII, PCI, attorney-client - all of it leaves your network on every API call. HIPAA, SR 11-7, EU AI Act, GDPR all have something to say about that.
reproducibility
Vendor ships a new model version, your prompts drift, your evals shift. "The AI said so" is not a record. No signed receipt. No reproducer.
continuity
Your AI feature lives at someone else's URL. Sunset = re-train. Air-gap impossible. Audit only as far as the vendor lets you see.

how this is different

kolm vs. the things you already use.

vs Ollama

Ollama runs models. kolm compiles tasks.

Ollama gives you Llama-on-disk. kolm gives you a task-specific artifact: base model + LoRA + recipes + recall index + verifier - signed.
vs RAG

RAG is a runtime. kolm is a release.

RAG re-indexes at every query and stays online. kolm bakes the corpus and the verifier into the artifact, so the runtime is hermetic.
vs fine-tuning

Fine-tuning gives you weights. kolm gives you a release.

A LoRA without a verifier, recall, policy, and signature is not a release artifact - it's a checkpoint. kolm is the layer above.
vs frontier API

Compile once. Run forever.

$0.04/call × 90%-deterministic tasks compounds forever. 15–30× year-1 savings on hot paths.
vs OpenPipe

Both capture. Only one gives you a file.

OpenPipe captures and hosts a fine-tune. kolm captures and compiles a portable signed artifact. Same loop, different deliverable.
vs Predibase

Same LoRA. Different deliverable.

Predibase trains and serves LoRAs in their cloud. kolm wraps the same LoRA into a signed file with verifier + recall + receipts.
vs LangSmith

Same trace. Different exit.

LangSmith captures traces for debug. kolm captures the same traces and compiles them into a smaller, signed model.

03compile

Cache the expensive intelligence once.

The compiler turns a task into a releaseable unit: examples become tests, private context becomes recall, policy becomes a runtime guard, and the approved behavior becomes the cache.

patient-intake.task approved task contract 
task: classify incoming patient messages
labels: urgent | billing | schedule | follow-up
examples: ./labels.jsonl
corpus: ./clinical-policy/
constraints:
  runtime_egress: 0
  k_score_min: 0.90
  receipt: required
$ kolm compile "classify patient messages" \
  --examples ./labels.jsonl \
  --data ./clinical-policy \
  --base-model qwen2.5-coder-7b-instruct-q4_0

ingest   1,284 policy chunks indexed
verify   412 examples accepted
package  manifest, verifier, recall, policy
sign     hmac-sha256 receipt chain

ok wrote patient-intake.kolm
ok K-score 0.92 / release gate 0.90
