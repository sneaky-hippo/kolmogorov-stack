Private git recipe

A compiled .kolm artifact is a file. Files want a home that knows about versions, hashes, distribution, and access control. A private git repo answers all four. With the W229 kolm sync verb, you push the artifact + the manifest + the receipt chain in one command, and the entire fleet can pull it with another.

Why pair them

An AI artifact has the same properties as source code. You want diffable history, you want signed commits, you want branch-per-tenant isolation, you want bisect for regressions, you want a real ACL story. Object stores like S3 give you durability but not history or diffability. Model registries give you a UI but lock you to one vendor. A private git remote (Gitea, Forgejo, GitLab self-hosted, GitHub Enterprise, plain ssh + bare repo) gives you everything that matters with tools you already trust.

The .kolm artifact format is binary, but the manifest is JSON and the receipt chain is a list of hashes. git diffs both cleanly. A code-review tool can show "tenant foo's recipe changed from K-score 0.87 to 0.91, here are the seeds that moved" in the same UI you already use for pull requests.

One-time setup

  1. Pick a git remote you already trust. Gitea (gitea.com) and Forgejo (forgejo.org) are excellent self-hosted choices; GitLab and GitHub Enterprise work; a plain bare repo on a Linux box works.
  2. Create an empty private repo named after the fleet: kolm-fleet is fine.
  3. Authorize the compile box's SSH key on the remote. Test with git ls-remote git@host:org/kolm-fleet.git.
  4. (Optional) Enable git LFS if your artifacts exceed 100 MB and your remote supports it; kolm sync writes to a per-artifact subdirectory so LFS rules can target **/artifact.kolm.

The daily flow

# 1. compile in tmux, push to the mirror when done
kolm compile spec.json --out artifact.kolm
kolm sync push artifact.kolm --git git@host:org/kolm-fleet.git \
  --subdir tenant-foo/$(date +%Y%m%d) \
  --message "recipe v3, K-score 0.91, seeds N=4200"

# 2. on a consumer box, pull the latest
kolm sync pull --git git@host:org/kolm-fleet.git
ls /tmp/kolm-sync-*/tenant-foo/
# 20260514  20260516  20260518

# 3. verify the receipt chain locally (no network call to a registry)
kolm verify /tmp/kolm-sync-*/tenant-foo/20260518/artifact.kolm

# 4. inspect what changed since the last revision
git -C /tmp/kolm-sync-* log --oneline tenant-foo/

What this gets you

  • Every revision of every artifact is a git commit. git bisect works for "when did the regression land?"
  • The receipt chain is in source control. Pull request review can show the diff between two manifests; the signing chain is visible in git log.
  • Branch per tenant isolates blast radius. A bad push to tenant-foo never touches tenant-bar. Merge to main only what passes a review gate.
  • Compatible with all the git tooling you already use: GitHub Actions or Gitea Actions for CI, code-review UI for human gates, CODEOWNERS for per-tenant approvers.
  • Disaster recovery is git's strength: clone the remote anywhere, you have the entire history.
kolm does not require a git remote. kolm compile produces the artifact locally; you can copy it with scp, S3, rsync, or a USB stick. The git recipe is for fleets where multiple machines and multiple humans need to coordinate on the same artifact set.

Layout convention

The W229 sync writes a fixed layout per push. Knowing it lets you build tools on top.

<subdir>/
  artifact.kolm        # the compiled binary (LFS-eligible path)
  manifest.json        # full manifest including signature trailer
  receipt-chain.json   # Merkle leaves in declared order
  synced_at.txt        # ISO timestamp of the push

You can grow the subdir tree however you like (tenant/date/, env/recipe-version/, flat/). The four file names are fixed so downstream consumers always know where to read.

Failure modes worth knowing

Repo size growth. Compiled artifacts can be hundreds of megabytes. Without LFS, the repo grows fast. Configure LFS for the artifact path on day one, or rotate the repo (rename old, start fresh) on a quarterly cadence.

Push contention. Two compile boxes pushing to the same subdir at the same time produce a merge conflict. Pick one of: (a) one canonical compile box per tenant, (b) date-stamped subdirs that never collide, (c) a queue tool in front of the remote.

Force pushes. Never force-push to the fleet repo. The receipt chain depends on commit history; rewriting it breaks every consumer's kolm verify.
