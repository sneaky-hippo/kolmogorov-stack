Yours, local, signed.

Anyone (human or AI agent) authors a signed .kolm in four commands. JSON spec in, signed binary out, runs locally with zero egress, every output ships with a byte-walkable receipt chain. No account needed for local builds.

quickstart authoring guide cookbook
01thirty seconds, four commands

Scaffold → compile → run → serve.

Local builds use a per-user receipt secret at ~/.kolm/config.json (mode 0600). Set RECIPE_RECEIPT_SECRET in your environment for a teammate's verifier to accept the same artifact.

$ npm i -g github:sneaky-hippo/kolmogorov-stack
$ kolm new my-redactor --from redactor
wrote ./my-redactor.spec.json    # a working spec, edit to taste

$ kolm compile --spec my-redactor.spec.json --out my-redactor.kolm
built my-redactor.kolm  4249 B  k-score 381.61  sha256 867dc0ce…

$ kolm run my-redactor.kolm '{"text":"call me at 555-123-4567"}'
{ "output": { "redacted": "call me at [PHONE]", "hits": [{ "name": "PHONE", "count": 1 }] } }

$ kolm serve --mcp --http --port 11455
ok serving ~/.kolm/artifacts as local tools
02graduate to a project

One kolm.yaml turns the script into a project.

Optional. Useful when you want one MCP namespace per repo, K-score gates that refuse to serve below a floor, paths globs that auto-attach artifacts when the agent is editing matching files, and a place for compile/run/bench hooks. The schema is published at kolm.ai/docs/kolm-yaml-v0.1.json.

$ kolm init
wrote kolm.yaml
mkdir .kolm/artifacts/  .kolm/skills/
updated .gitignore (+ .kolm/)

$ kolm compile --spec my-redactor.spec.json --out .kolm/artifacts/my-redactor.kolm
built .kolm/artifacts/my-redactor.kolm
skill .kolm/skills/my-redactor.md   # Claude Code / Cursor / Cline / Continue index this

Every kolm compile now emits a SKILL.md sidecar next to the artifact. The frontmatter follows Claude Code's format (name, description, allowed-tools, disable-model-invocation) so frontier-agent harnesses index it without manual wiring. The MCP tool name resolves to mcp__<project>__<artifact>.

# one command per harness. --apply writes; omit to dry-run.
$ kolm install claude-code --apply
wrote ~/.claude/settings.json

$ kolm install cursor --apply
wrote .cursor/mcp.json

$ kolm install continue --apply
wrote ~/.continue/config.yaml

$ kolm install cline --apply
wrote .clinerules/kolm.md

$ kolm doctor
ok node ≥18 · kolm.yaml · 3 artifacts · ~/.kolm/config.json · git

One command per harness. kolm doctor verifies the wiring; kolm logs --since 24h tails the run-history JSONL at ~/.kolm/logs/runs.jsonl. No daemons, no dashboards, no telemetry leaving the box.

03templates

Four shapes that cover most of the work.

Each template scaffolds a complete spec: recipes, evals, working pack, index. If none fit, start --from blank. The sandbox is JavaScript with a curated lib.

TemplateShape of the recipeRealistic example
redactor
--from redactor		 Replace regex matches with named tokens. Tenants pass extra_patterns/redact_words at run time, no re-signing. Strip phones, emails, MRNs, account numbers, employee ids, codenames.
extractor
--from extractor		 Pull named fields from free-text. Tenants pass extra_rules. Amount + date + counterparty from a payment; case_id + priority from a triage note.
classifier
--from classifier		 Score input against weighted keyword categories, return top label. Tenants pass extra_categories. Route tickets billing / bug / feature / rma. Tag sensors nominal / warning / critical.
blank
--from blank		 One stub recipe + one eval case. 40 lines of JSON. Anything the three above don't cover. Most one-off agent tools start here.
04spec is the program

Forty lines of JSON, one signed binary.

Deterministic over the spec: same spec, same content-hash (zip wrapper varies; signed manifest does not). Each recipe is function generate(input, lib) in a node:vm sandbox, 1 MiB input cap, 1 000 ms timeout.

# spec.json - this is the entire program
{
  "job_id": "job_clinic_redactor",
  "task": "Strip identifiers from clinical notes before they leave the device",
  "base_model": "none",
  "recipes": [{
    "id": "rcp_redact_v1",
    "name": "regex redactor",
    "source": "function generate(input, lib) {\n  var text = String(input.text || '');\n  var hits = [];\n  for (var p of (lib.params.extra_patterns || [])) {\n    var re = new RegExp(p.regex, 'g'); var n = 0;\n    text = text.replace(re, function(){ n++; return p.replacement || '[REDACTED]'; });\n    if (n) hits.push({ name: p.name, count: n });\n  }\n  return { redacted: text, hits: hits };\n}",
    "schema": { "input": { "text": "string" }, "output": { "redacted": "string", "hits": "array" } }
  }],
  "evals": {
    "spec": "rs-1-evals",
    "cases": [{ "id": "phone", "input": { "text": "call 555-123-4567" }, "expected": { "redacted": "call [PHONE]" },
      "params": { "extra_patterns": [{ "name": "PHONE", "regex": "\\d{3}-\\d{3}-\\d{4}", "replacement": "[PHONE]" }] } }]
  }
}

$ cat spec.json | kolm compile --spec - --out clinic-redactor.kolm
ok wrote clinic-redactor.kolm
$ kolm inspect clinic-redactor.kolm | head
{
  "tier": "recipe",
  "signature_valid": true,
  "recipes_n": 1,
  "evals_n": 1,
  "k_score": 412.7,
  "size_bytes": 4188
}

Full reference in docs/AUTHORING.md: spec field rules, sandbox surface (lib.patterns, lib.parseFloatSafe, lib.pack, lib.index, lib.params), pack/index containers, receipt chain, every compiler error code.

05AI-friendly authoring

Hand a model the spec; pipe the spec to kolm compile.

An agent that writes JSON ships a signed artifact. Validation is strict; every recipe type-checks, every eval case runs before sealing. Drop this prompt in, pipe out to kolm compile --spec -.

# AI authoring prompt (drop into any frontier model)

You are authoring a kolm spec - a JSON program that compiles to a signed
.kolm artifact and runs locally with zero egress.

Output ONLY a single JSON object. Required fields:

  job_id        "job_"   matches /^job_[a-z0-9_-]+$/i
  task          one-sentence human description
  recipes[]     each: { id: "rcp_", name, source }
                source is "function generate(input, lib) { ... return {...}; }"
                lib has: patterns, parseFloatSafe, pack, index, params
  evals.cases[] each: { id, input, expected, params? }

Constraints:
  - source must be valid JavaScript that compiles under Node's vm module
  - inputs are capped at 1 MiB, recipes at 1000 ms wall-time
  - never reach for fs, net, child_process - the sandbox blocks them
  - return JSON-serialisable objects only

Task: 

# Pipe the model's output straight to the compiler
$ claude --message "" \
    | jq -r '.content' \
    | kolm compile --spec - --out my-task.kolm
ok wrote my-task.kolm  k-score 387.4  signature valid

No account, no network. Artifact signed under your local secret. Teammate verification: set RECIPE_RECEIPT_SECRET to the same value, or pin a fleet secret in CI.

06sensitive verticals

What kolm does and does not promise.

We ship the runtime substrate: signed rules, zero-egress sandbox, receipt chain, audit log. We do not ship compliance attestation, recipe correctness, or input-handling guarantees outside the runtime.

What you getWhat you're still on the hook for
Zero runtime egress. Benchmark monitor patches fetch, http, https, net, tls, dns. Wire-touching artifacts fail benchmark. Recipe correctness. A regex missing a digit pattern still misses. Write evals for your edge cases.
Signed receipt chain. Five HMAC-SHA256 rings (task → seeds → recipes → evals → package). Byte-walkable; tampering detected at load. Compliance attestation. kolm does not hold cert; the artifact gives an auditor what they need. The audit boundary is yours.
Tenant runtime params. Buyers pass extra_patterns, extra_rules, extra_categories per call. Signature unchanged. What buyers do with output. A clinic logging PHI post-redaction is a clinic problem. Audit-sink is opt-in.
Audit-sink hook. Every runArtifact emits kolm-audit-1: recipe id, input sha256 prefix, latency, ran_at. SIEM-ready. Threat model outside sandbox. Recipe sandboxed; host isn't. Run kolm in your pipeline's trust boundary.

Three patterns to start from:

# clinic redactor - PHI redaction with tenant-specific patterns
$ kolm new clinic-redact --from redactor
# edit clinic-redact.spec.json: add MRN, DOB, NPI patterns; add evals; compile
$ kolm compile --spec clinic-redact.spec.json --out clinic-redact.kolm
$ kolm run clinic-redact.kolm '{"text":"pt MRN 0042-A seen 2026-04-01"}' \
    --params '{"extra_patterns":[{"name":"MRN","regex":"MRN \\d{4}-[A-Z]","replacement":"[MRN]"}]}'

# banking extractor - pull amount + date + counterparty out of payment text
$ kolm new wire-extract --from extractor
$ kolm compile --spec wire-extract.spec.json --out wire-extract.kolm
$ kolm run wire-extract.kolm '{"text":"$2,400 wire to ACME on 2026-04-15"}'

# support triage - route inbound tickets without an LLM
$ kolm new support-triage --from classifier
$ kolm compile --spec support-triage.spec.json --out support-triage.kolm
$ kolm run support-triage.kolm '{"text":"refund please, charge looks wrong"}'
{ "output": { "label": "billing", "score": 2 } }

Vertical walkthroughs (healthcare, finance, legal, edge, defense) with buyer gates (BAA, audit lineage, privilege, RAM/offline) in the cookbook.

07tenant params don't re-sign

One artifact, many tenants, no resigning.

One signed .kolm; buyers configure per call; signature unchanged; audit log records each tenant's params. A healthcare vendor's redactor: 200 clinics customize MRN formats without touching the binary.

# a single signed redactor, two tenants, two configurations, one audit trail

tenant-A$ kolm run ./redactor.kolm \
    '{"text":"pt MRN A-100042 reviewed"}' \
    --params '{"extra_patterns":[{"name":"MRN","regex":"MRN A-\\d{6}","replacement":"[MRN]"}]}'

tenant-B$ kolm run ./redactor.kolm \
    '{"text":"patient ID 99-7711 admitted"}' \
    --params '{"extra_patterns":[{"name":"PID","regex":"\\d{2}-\\d{4}","replacement":"[PID]"}]}'

# both tenants run the same artifact (same sha256, same signature).
# both audit logs distinguish which params were applied.
08buy instead of rent Sprint 3 roadmap

Compound your frontier-API spend into a local artifact.

Sprint 3 roadmap, not yet shipped. The capture → aggregate → distill loop is coming. Subscribe at /signup.

Spec authoring is the front door. The heavier door: every API call proxied through the capture endpoint writes a verified (input, output) pair into your namespace. Enough pairs distill into a recipe (or LoRA) running locally at zero per-call cost. Frontier API becomes a deposit account, not a forever-bill.

PhaseWhat will happenWhat you keep
Day 1
capture		 Point your client at the capture proxy. Forwards upstream with your key, records verified (input, output, latency) in observations. Same answer, same SLA; traffic now leaves a labelled trail you own.
Day 7-14
aggregate		 Observations cluster by intent. Bridges surface recommends synthesis past threshold; corpus exports as JSONL or parquet. A clean dataset of your own task to train any model.
Day 14-30
distill		 Auto-distill compiles the cluster to a recipe (or LoRA if size warrants). Output: a signed .kolm in ~/.kolm/artifacts. Offline specialist for the slice you run. Zero per-call cost.
Day 30+
compound		 Easy 60% to local, hard 40% to frontier (still capturing). Local-share grows; frontier bill shrinks. Latency, privacy, cost. Cancel kolm and the .kolm on disk still runs.

Illustrative: 50-engineer team, 80 000 frontier calls/mo, 30 days capture, 50 000 verified labels: a Phi-3-mini-LoRA hits ~78% of Opus quality at <5% latency and zero per-token cost. Capture/synthesis/auto-distill are Sprint 3; sections 01-06 ship today.
