03 ·Discovery

kolm serve --mcp. One command. One harness config.

Wrap any .kolm artifact in a Model Context Protocol server. Claude, Cursor, Continue, and Cline plug in via kolm install <harness> --apply; the harness config points at the local kolm serve --mcp stdio command. Each call returns the artifact output plus _kolm run metadata (recipe id, latency, K-score, audit fields). Automatic mDNS discovery, SSE transport, and .well-known/mcp.json publishing are target architecture.

# Default: line-delimited JSON-RPC over stdio (no port).
$ kolm serve --mcp

   - /span> kolm v0.4 ·serving artifacts from ~/.kolm/artifacts
   - /span> transport      : stdio json-rpc
   - /span> tools/list     : 5 artifacts discovered
   - /span> receipts       : signed artifact-level (per-call HMAC is target arch)

# Optional: localhost HTTP JSON-RPC on 127.0.0.1:8765
$ kolm serve --mcp --http

02 ·How it works

No daemons. No control plane. The harness spawns kolm serve --mcp over stdio; close the harness and the process exits.

01

Shipped today: kolm serve --mcp speaks line-delimited JSON-RPC over stdio (the MCP default transport). Pass --http to expose a localhost HTTP JSON-RPC endpoint on 127.0.0.1:8765 for clients that prefer HTTP. mDNS announce and .well-known/mcp.json publishing are target architecture.

02

Shipped today: kolm install <harness> --apply wires the harness config for claude-code, cursor, continue, and cline. Each harness points at a local kolm serve --mcp command; there is no auto-discovery yet.

03

Shipped today: the artifact signature is verified when an artifact is loaded; tools/call returns the recipe output plus _kolm run metadata (recipe id, latency, K-score, audit fields). Short-lived bearer-token exchange and per-call HMAC chaining are target architecture.

04

Shipped today: tools/list surfaces each artifact's recipes as MCP tools; schemas come straight from the artifact manifest, so the agent sees only the surface kolm sealed in. SSE transport is target architecture.

Works with any MCP client.

If your agent speaks MCP, kolm shows up. Three popular ones:

Claude Desktop

Drop the artifact path into claude_desktop_config.json. kolm-served tools appear in the next session.

"command": "kolm", "args": ["serve","--mcp","./support-triage.kolm"]

Cursor

MCP server settings → add Local. Cursor’s agent picks up tool schemas and types them into the prompt automatically.

cursor add-mcp support-triage

Continue.dev

Same shape: register a local MCP target, point at the artifact path. Continue auto-routes tool calls and surfaces receipts inline.

~/.continue/config.yaml

Receipts: shipped today, deeper tomorrow.

Shipped today: every .kolm artifact carries an HMAC-SHA256 signed manifest binding source hashes, input/output schemas, runtime version, and issue time; kolm inspect re-verifies the artifact bytes against the shared tenant receipt secret. MCP tools/call returns _kolm run metadata (recipe id, latency, K-score, audit fields). Target architecture: per-call HMAC chaining over input/output hashes, plus optional Sigstore / Rekor anchoring on Enterprise for cross-organization audit when an artifact is shared across machines.
