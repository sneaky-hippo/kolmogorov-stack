RS-1 v2.1: A Reproducible Signed-artifact format for ML inference.

Specification, threat model, four recipe classes, three-signature cascade, and the K-score reproducibility criterion. Verifiable revision matching shipping implementation.

kolm engineering . hello@kolm.ai

v2.1 . 2026-05-17 . supersedes v2.0 . citeable as kolm.ai/spec/rs-1

Abstract. Modern ML inference is auditable in name only. A frontier API call returns text and a request ID; everything else is gone. We propose RS-1, a reproducible signed artifact format for machine learning inference. An RS-1 artifact is a deterministic ZIP file (.kolm) containing a canonical-JSON manifest, a recipe bundle, an evaluation result, and class-dependent payload (deterministic rule code; native binary; or distilled neural weights). Each artifact declares an artifact_class from a fixed taxonomy of four (rule, synthesized_rule, compiled_rule, distilled_model) and the format verifier rejects artifacts whose contents do not match their declared class. The artifact is identified by a content-addressable identifier (CID). Inference receipts bind (input hash, output hash, manifest CID, timestamp) under a defense-in-depth signature cascade: HMAC-SHA256 (integrity), Ed25519 (asymmetric authenticity), and optional sigstore bundle with Rekor transparency-log anchoring. We define the K-score, a reproducibility criterion gating compile time and recorded in every receipt; eval independence is enforced by a deterministic train/holdout split of the tenant's seeds.jsonl with verifier-checked disjointness. The format is public; a reference verifier with zero third-party runtime dependencies is provided. RS-1 v2.0 is intended as the audit primitive for regulated AI deployment.
Provenance note (v2.1 reconciliation). RS-1 v2.0 (2026-05-17 a.m.) corrected the v1.0 overclaim of INT4/ONNX-for-every-artifact and introduced the four-class taxonomy and three-signature cascade, but two sections in §7 (PHI redactor §7.1, cross-vendor distillation §7.2) and the K-score R-axis still carried "wave forthcoming" markers. v2.1 (this revision, 2026-05-17 p.m.) closes the Q+11 reconciliation track from the Wave 144 plan: every numbered subsection in §7 is now grounded in a shipping wave; the K-score R axis is grounded in wave 164 (external + adversarial holdouts, §7.12); the Z drift axis is grounded in wave 167 (drift detection + supersession chain, §7.15); and §7.1 + §7.2 are demoted to cross-references that point at the implementation-confirmed subsections §7.6 (W157) and §7.7 (W158). 1. Motivation

Three open problems in deployed machine learning motivate this work.

Auditability. A regulator asking "what did the model see and what did it return on this date" cannot be answered by current frontier APIs. The token bill is reproducible; the inference is not.

Portability. A signed artifact is the canonical handle for ML deployment in adversarial environments: air-gap networks, regulated zones, edge devices, multi-tenant clouds. There is no industry-wide format for such an artifact.

Quality contract. "The model is good enough" is the most common production deployment criterion. It is not a contract. A blocking, formula-defined quality gate is.

2. The four recipe classes

RS-1 v2.0 plainly distinguishes four classes of recipe. Every recipe in an artifact carries an artifact_class field, and the artifact's artifact_class is the MAX-severity rollup over its recipes (rule:0 < synthesized_rule:1 < compiled_rule:2 < distilled_model:3). A breakdown of per-class counts is recorded in artifact_class_breakdown.

ClassWhat it isTeacherWeightsNative bin
ruleDeterministic finite-state transformation. No model, no teacher, no neural net.nonono
synthesized_ruleLLM emitted rule code at build time, then frozen. Records teacher_vendor + teacher_model.yesnono
compiled_ruleRule recipe compiled to native (C / Rust / WASM) via the P+1-P+3 backends.nonoyes
distilled_modelReal teacher-to-student distillation. Requires non-empty weights and teacher_vendor + teacher_model + distillation_method + lora_rank in manifest.training.yesyesno

The build-time validator (src/recipe-class.js validateRecipeClass) throws if a recipe is misdeclared. The verifier (src/binder.js check #3) rejects an artifact whose artifact_class claim does not match its contents before running any signature check. A distilled_model claim with model.gguf == sha256-of-empty is rejected; the operator sees "class doesn't match contents" not "signature failure."

3. The artifact layout

An RS-1 artifact is a deterministic ZIP file with the extension .kolm. The mandatory entries are:

artifact.kolm/
  manifest.json       # canonical-JSON manifest, root of trust
  spec.json           # declarative spec for the recipe(s)
  recipes.json        # the compiled recipe(s): source + metadata
  evals.json          # K-score breakdown + eval pack results
  seeds.jsonl         # tenant-captured IO corpus (used for train/holdout split)
  receipt.json        # build-time receipt chain (see section 5)

Class-dependent payload entries:

  # only for distilled_model
  model.gguf          # quantized weight tensors (GGUF) or empty for non-model classes
  lora.bin            # LoRA adapter bytes (empty for non-model classes)
  index.sqlite-vec    # optional vector index for retrieval recipes
  # only for compiled_rule
  native/             # compiled binary + source + Cargo.lock / build provenance
  # optional, all classes
  pretokenize/        # KOLMIDX2 + KOLMPCK2 binaries (wave 148)
  moe/                # MoE composition provenance (wave 147)
  lineage/            # ancestor artifact hashes

The ZIP is built with deterministic mtimes (epoch 0) and store/deflate compression so byte-identical inputs produce byte-identical artifacts. The CID is cidv1:sha256: followed by hex(SHA-256(canonical-JSON(manifest.artifact_hash_input))).

3.2 Canonical JSON

Canonical JSON is the byte-identical serialization used for hashing. Rules: sorted keys recursively, no insignificant whitespace, UTF-8, integers as integers. Reference implementations in JavaScript, Python, and Rust all produce byte-identical serializations; this is enforced by an interop test suite.

4. Eval independence: seeds.jsonl train/holdout split

The K-score's accuracy axis is measured against a holdout the recipe has never seen. RS-1 v2.0 defines the canonical mechanism: each tenant ships a seeds.jsonl file of captured IO (canonical shape {"input","output","tags"}; legacy shape {"prompt","completion"} is auto-normalized). The build pipeline splits 80/20 deterministically:

holdout = { row | sha256(row.input + split_seed) % 5 == 0 }
train   = seeds \\ holdout

The recipe is synthesized or distilled from the train partition only. K-score's A axis is computed as correct_on_holdout / |holdout| using the recipe-declared comparator (exact, json-subset, regex, semantic, or human-reviewed). The receipt chain extends to seeds_hash + split_seed + train_hash + holdout_hash; the verifier rebuilds the split from seeds.jsonl + split_seed and confirms train_hash ∩ holdout_hash = ∅. An artifact whose train and holdout intersect is rejected pre-signature.

Artifacts whose seeds were generated by kolm seeds bootstrap (synthetic starter rows) carry eval_source: "synthetic_starter" and product surfaces display "K-score measured on synthetic data — replace seeds with real captured IO before relying on this number." Tenant-captured seeds carry eval_source: "tenant_captured".

5. Receipt chain & signature cascade

Every build produces a receipt that hash-chains the canonical slots:

task -> seeds -> recipes -> evals -> package -> signature

For PHI workloads, the chain extends inside recipes to include redaction_map_hash + teacher_call_log_hash + reinjection_log_hash so an auditor can replay the redactor pipeline offline and confirm raw PHI never left the tenant boundary.

5.1 Three-signature cascade (waves 149-150)

RS-1 v2.0 signs in three layers, innermost to outermost:

  1. HMAC-SHA256. Computed over canonical-JSON(body) excluding all signature fields. Field: signature. Verifier uses constant-time comparison. Provides integrity against accidental corruption. Symmetric: anyone with the secret can forge.
  2. Ed25519. Computed over canonical-JSON(body with HMAC, without signature_ed25519 and signature_sigstore). Field: signature_ed25519. Asymmetric: only the holder of the private signing key can sign. Public keys are published at /v1/keys/public/<fingerprint> after a proof-of-control challenge.
  3. Sigstore bundle (optional). Cosign-compatible raw-key bundle with verificationMaterial.publicKey.{hint, rawBytes}, optional rekor_log_entry for transparency-log anchoring. Field: signature_sigstore. Verifies with cosign verify-blob --key <pem>. Dry-run by default (network optional); upgrades to Rekor-anchored when KOLM_SIGSTORE_REKOR_URL is set or via the post-build kolm sigstore-attest command.

Cascade encoding in signature_alg: 'sigstore+ed25519+hmac-sha256' (full), 'ed25519+hmac-sha256' (sigstore disabled), or 'hmac-sha256' (Ed25519 disabled). The verifier strips signature fields in reverse order before re-canonicalizing for each layer.

5.2 Runtime inference receipt

{
  "manifest_cid": "cidv1:sha256:1bcf2323...",
  "input_sha":    "sha256:f3a9...",

Runtime receipts use HMAC alone (the per-request hot path); the build-time chain carries Ed25519 and optional sigstore.

6. The K-score

The K-score is a reproducibility criterion gated at compile time. The V1 composite:

K = 0.40·A + 0.15·S + 0.15·L + 0.15·C + 0.15·V
  • A . Accuracy (0–1) — correct_on_holdout / |holdout| using the recipe-declared comparator. See section 4.
  • S . Sampling determinism (0–1) — 1 if seeded inference produces byte-identical output across n runs; geometric decay otherwise. Trivially 1.0 for rule and compiled_rule; meaningful for distilled_model.
  • L . Latency stability (0–1) — 1 if p99 latency ≤ 1.5·p50 over a 1k-sample bench.
  • C . Calibration (0–1) — 1 minus expected calibration error binned at 10 confidence buckets.
  • V . Verifier coverage (0–1) — fraction of receipt fields the reference verifier checks.

V2 scaffolding (in src/kscore.js) adds optional axes used by regulated verticals and by cross-vendor distillation:

  • R . Real-world independence — supplements A with an external public benchmark (Presidio, i2b2 2014, MIMIC-III for PHI; CMS / NCQA for claims). Sourcing and verifier check shipped in wave 164 (N+3 external benchmarks + N+4 adversarial cross-family holdouts); see §7.12.
  • F . Fairness — equal performance across declared subgroups in seeds.jsonl tags.
  • E . Eval-source label — categorical (synthetic_starter, tenant_captured, external_public, adversarial_pair, auditor_witnessed).
  • Z . Drift — running comparison against a fresh holdout collected post-ship. Snapshot/report/cron mechanism + verifier check shipped in wave 167 (M+3 drift detection + M+4 supersession chain); see §7.15.
  • T . Teacher-delta (distilled_model only) — T = teacher accuracy on same holdout; reported as A/T ratio. A/T=1.0 means student matches teacher; A/T=0.9 means student is at 90 percent of teacher on the same eval.

The compile-time K-score is recorded in manifest.json and in every inference receipt. A compile is gated by the recipe-declared min_k; the compile fails if K < min_k.

6.1 Calibrated decode

K=0.917 decodes as: "on traffic like the training set, the artifact echoes the teacher fairly closely, and is fast, deterministic, and cheap." It does not mean 91.7 percent correct on out-of-distribution input. The calibrated decode is part of the spec and is surfaced wherever a K-score is displayed.

7. Manifest extensions

7.1 PHI/PII redactor (Q+3a, wave 157)

Overview: when a recipe's redact_class is set (phi, phi-lds, pci, multi, or none), the build pipeline wraps every teacher API call in a redactor that maps the 18 HIPAA Safe Harbor identifiers (plus NPI / DEA / Medicaid extensions for healthcare provider identifiers) to placeholder tokens of the form [CLASS_INDEX]. The teacher operates only on placeholder text; the reinjector restores raw values in the response. The receipt chain extends to redaction_map_hash, teacher_call_log_hash, reinjection_log_hash. The redactor is a necessary but not sufficient PHI control; HIPAA Expert Determination per 45 CFR 164.514(b)(1) is the deeper control for high-risk workloads.

The verifier check (binder check #14, "PHI redactor receipt integrity"), manifest field shape, canary token-preservation test, and replay-offline auditor workflow are documented in §7.6. v2.1 keeps §7.1 as the conceptual overview and routes implementation detail to §7.6 so the section number stays stable across spec revisions while the implementation-confirmed claim lives next to its wave stamp.

The KOLMIDX2 binary is a 24-byte header followed by sorted 48-byte records (32 input_hash + 8 pack_offset u64 + 4 token_count u32 + 4 reserved); the KOLMPCK2 binary is a 24-byte header followed by a u32 LE token stream. 7.5 compiled_rule build provenance (wave 153, P+1)

For artifacts whose artifact_class rolls up to compiled_rule, the manifest's compiled_targets block records both the generated source (native.c, native.rs) and the optional native binaries (native.c.bin, native.rust.bin) the builder produced. The native section pins the toolchain so a verifier on the same compiler version can reproduce the bytes:

"compiled_targets": {

  "targets":           ["c","rust"],
  "native_spec":       "kolm-native-v1",
  "host_triple":       "x64-linux",
  "target_toolchain_pin": {
    "c":    { "compiler":"gcc",   "compiler_version":"gcc (Ubuntu 13.2.0) 13.2.0", "shim_source_hash":"<hex>", "source_date_epoch":0 },

    "rcp_echo": {
      "c":    { "filename":"native.c",   "source_hash":"<hex>", "bytes":824,

    }

}

The toolchain_version_hash is SHA-256(compiler + "\n" + compiler_version + "\n" + flags + "\n" + shim_source_hash). It lets a verifier confirm "this binary was produced by the same toolchain configuration the manifest claims" independently of whether the binary bytes are bit-identical. Cross-machine bit-identity is best-effort, not promised: reproducibility holds inside a pinned builder image per target triple. Verifier check #12 (src/binder.js, "Native binary integrity") re-hashes the bundled binary against bin_hash; a mismatch is a hard fail. Setting KOLM_COMPILE_NATIVE=1 (or passing compileNative: true to compileSpec) is opt-in. With no toolchain present, the artifact ships source-only and verification succeeds without the native check, preserving backward compatibility with JS-only artifacts.

7.5.1 hardening flag set + fine-grained toggles (wave 154, P+2)

The compiler invocation pins every nondeterminism source the toolchain exposes:

  • gcc / clang: -std=c99 -O2 -fno-stack-protector -fno-asynchronous-unwind-tables -fno-ident -ffile-prefix-map=BUILD=/build -Wl,--build-id=none plus SOURCE_DATE_EPOCH=0 in env. -fno-asynchronous-unwind-tables drops .eh_frame entries whose layout depends on compiler internals. -fno-ident drops the .comment section that gcc/clang stamp with the compiler version string. -Wl,--build-id=none strips the linker-injected unique build ID (best-effort, requires GNU ld / lld).
  • MSVC cl.exe: /nologo /O2 /Brepro /link /OPT:NOREF /OPT:NOICF /INCREMENTAL:NO /DEBUG:NONE. /Brepro requests compiler-side reproducible output. /OPT:NOREF and /OPT:NOICF disable link-time function dedup that reorders symbols nondeterministically. /INCREMENTAL:NO disables incremental linking; /DEBUG:NONE drops PDB references embedding absolute build paths.
  • rustc: --edition 2021 --crate-type bin -C opt-level=3 -C codegen-units=1 -C lto=fat -C strip=symbols -C panic=abort -C debuginfo=0 --remap-path-prefix=BUILD=/build plus SOURCE_DATE_EPOCH=0 in env. All flags are stable rustc options; -C strip=symbols stabilized in 1.59.

The compile pipeline also honors two fine-grained env-var toggles for hosts that have only one toolchain or builders that want to exercise one path at a time: KOLM_COMPILE_NATIVE_C_ONLY=1 disables Rust compilation even if rustc is on PATH; KOLM_COMPILE_NATIVE_RUST_ONLY=1 disables C compilation even if a C compiler is on PATH. Either flag implies opt-in (the corresponding compileNative: true opt is set automatically when these are read). The disabled kind shows up in compiled_targets.native_skipped.<kind> with the reason string naming the env var, so a verifier knows the skip was intentional rather than from a missing compiler.

7.5.2 WASM target (wave 155, P+3)

The third compile backend is wasm32-wasi, emitted via rustc --target=wasm32-wasi when the rustc target is installed (rustup target add wasm32-wasi), with a fallback to clang --target=wasm32-wasi when a wasi-sysroot is present. Rust is preferred when both are usable because the deterministic flag set is more comprehensive. WASM reuses an existing source kind: the spec already emits a .c and a .rs per recipe, and the WASM compile picks whichever matches the detected toolchain's source_kind. No new source generation step.

The flag posture mirrors §7.5.1 for the chosen source kind plus --target=wasm32-wasi. For rustc: --edition 2021 --crate-type bin --target=wasm32-wasi -C opt-level=3 -C codegen-units=1 -C lto=fat -C strip=symbols -C panic=abort -C debuginfo=0 --remap-path-prefix=BUILD=/build plus SOURCE_DATE_EPOCH=0 in env. For clang: --target=wasm32-wasi -std=c99 -O2 -fno-stack-protector -fno-asynchronous-unwind-tables -fno-ident -ffile-prefix-map=BUILD=/build -Wl,--strip-all.

compiled_targets.recipes.<rid>.wasm.bin = {
  source_kind: "rust" | "c",        // which source the wasm compile fed on
  compiler: "rustc" | "clang",
  compiler_version: "rustc 1.78.0 (9b00956e5 2024-04-29)",
  target_triple: "wasm32-wasi",
  flags: "--edition 2021 --crate-type bin --target=wasm32-wasi -C opt-level=3 ...",
  shim_source_hash: "<sha256 of the same shim used by the native path>",
  source_date_epoch: 0,
  toolchain_version_hash: "<sha256 of compiler+version+flags+shim_hash>",
  bin_filename: "native.wasm",       // or "native.<rid>.wasm" in multi-recipe artifacts
  bin_hash: "<sha256 of the .wasm bytes>",
  bytes: 12450,
}

compiled_targets.target_toolchain_pin.wasm = {
  source_kind: "rust" | "c",
  compiler, compiler_version, shim_source_hash,
  source_date_epoch: 0,
  target_triple: "wasm32-wasi",
}

WASM is its own opt-in target — it has separate toggles from native (a builder may want WASM without native, or native without WASM): KOLM_COMPILE_WASM=1 enables WASM alongside native; KOLM_COMPILE_WASM_ONLY=1 additionally nulls both C and Rust native paths so only the .wasm binary ships. The compileWasm: true / wasmOnly: true opts are the programmatic equivalents. Verifier check #12 (Wave 144) re-hashes the bundled .wasm bytes against bin_hash just like the native binaries — wasm tamper detection is part of the same chain.

7.5.3 Deterministic build verification (wave 156, P+4)

Verifier check #12 ("Native binary integrity") re-hashes the bundled binary against the manifest bin_hash claim. That catches naive tampering: an attacker who swaps native.c.bin for a different binary without updating the manifest is rejected. It does not catch the subtler case where the manifest bin_hash has been updated to match a forged binary — the claim still re-hashes to itself.

Check #13 ("Build reproducibility") closes the loop. When the verifier owns a local toolchain matching the bundle's target_toolchain_pin (same compiler, same compiler_version), it re-invokes the same compile path the original builder used — same source bytes from the zip, same flags from the manifest claim, same shim, same SOURCE_DATE_EPOCH=0 environment — and compares the resulting sha256 against the manifest bin_hash. A match is the strongest reproducibility evidence the receipt chain can produce: it proves the bundled binary is exactly what the pinned toolchain produces from the bundled source. A mismatch is a hard fail.

The check is opt-in because rebuild is expensive (seconds per recipe). Default behaviour: report warn — skipped (set KOLM_VERIFY_REBUILD=1 to enable) so the verifier stays fast and the cheap hash recomputation in check #12 still runs every time. Setting KOLM_VERIFY_REBUILD=1 enables the rebuild for binders who want the strongest check available offline.

Skip-graceful by design: a verifier without a matching local toolchain reports warn with a per-claim reason (no toolchain on host / compiler drift / version drift / source missing / compile threw). The check fires fail only when a rebuild actually completed and the resulting hash differed — the genuine reproducibility break the check is meant to catch. Version drift is exact-string match against the pin's compiler_version field; this is intentionally strict, because reproducibility across versions is not guaranteed and a permissive match would weaken the claim.

# Default — check #13 reports warn-skipped, check #12 still re-hashes
$ kolm verify artifact.kolm
  Build reproducibility            warn   1 native binary claim(s) eligible for
                                          deterministic rebuild; skipped
                                          (set KOLM_VERIFY_REBUILD=1 to enable)

# Opt-in — check #13 rebuilds and matches
$ KOLM_VERIFY_REBUILD=1 kolm verify artifact.kolm
  Build reproducibility            pass   1 of 1 binary(ies) rebuilt
                                          deterministically and matched
                                          manifest claim: rcp_echo/c=gcc@12480B

The receipt-chain implication: toolchain_version_hash (Wave 153) is no longer a passive claim. A buyer with the same toolchain can convert "this binary was built with rustc 1.78.0 + these flags" from a recorded provenance string into an automated re-derivation check. The pin moves from documentation to contract.

7.6 PHI redactor receipt integrity (wave 157, Q+3a)

Frontier-API teachers (Anthropic Opus, OpenAI GPT-5, Google Gemini, xAI Grok) cannot legally see raw PHI for a HIPAA-covered tenant — even with a BAA in place, the minimum-necessary standard pushes toward sending the least protected information required for the task. For most narrow distillation tasks that minimum is zero. The Wave Q+3a pattern: redact the 18 HIPAA Safe Harbor identifiers (plus the kolm extensions for NPI, DEA, and Medicaid provider IDs) before the teacher call; substitute reversible placeholder tokens of the form [PHI_<CLASS>_<INDEX>]; re-inject the original values into the teacher response on the way back. The teacher only ever sees redacted text. The raw values stay inside the tenant boundary.

RS-1 v2.0 makes this pattern auditable. The manifest.training block carries four fields when the recipe was distilled with a redactor in the loop:

  • redact_class — one of none / phi / pci / multi / auto. Records which redactor profile was applied.
  • redaction_map_hash — sha256 of the canonical-JSON serialization of the placeholder substitution table. Lets an auditor confirm the same mapping was applied across the whole training corpus.
  • teacher_call_log_hash — sha256 of teacher-call-log.jsonl: one line per teacher invocation with the redacted input, system prompt, and response.
  • reinjection_log_hash — sha256 of reinjection-log.jsonl: one line per call with per-row token counts and a preservation_ok flag (true when every token in the redacted input appeared verbatim in the teacher response, false when the teacher dropped or invented a placeholder).

Check #14 ("PHI redactor receipt integrity") is the verifier-side gate. When manifest.training.redact_class is anything other than none, all three log hashes are required. Missing any of them is an immediate fail — the recorded claim "we redacted" cannot stand without the offline-replay surface that lets a third party verify it. When the artifact additionally bundles the teacher-call-log.jsonl file (kolm artifacts may carry it; the manifest hash claims are sufficient on their own), check #14 also (a) re-hashes the bundled log against training.teacher_call_log_hash and fails on any mismatch; (b) re-runs the redactor over every redacted_input / redacted_system / redacted_response line in the log and fails on any new token detected — the redactor finding new identifiers in fields it had already masked means raw PHI was carried into the log, breaking the tenant-boundary guarantee.

The canary-token-preservation check is recorded but is not a verifier gate: the worker's reinjection-log carries the preservation_ok flag per row, and a recipe-level teacher_compliance_test can refuse to ship if the teacher consistently fails to echo placeholders unchanged. Some teachers normalize or paraphrase tokens; the worker falls back to a per-field call pattern (one teacher call per redacted field) when the canary fails so the teacher never sees an aggregate. Either way the receipt chain captures what was sent and what came back.

The Safe Harbor 18 + kolm extensions. The redactor's class set covers all 18 categories enumerated at 45 CFR 164.514(b)(2): names, geographic subdivisions, dates other than year, telephone, fax, email, SSN, MRN, health-plan beneficiary, account, certificate/license, vehicle, device, URL, IP, biometric, full-face photographs (flagged for vision recipes), and any other unique code. kolm adds three healthcare-provider identifiers regulators care about even though Safe Harbor does not: NPI, DEA, and state Medicaid provider IDs. These exist because real prior-auth and claims-adjudication workflows route on provider identity, and a redactor that masks the patient but leaves the prescriber unmasked still leaks identifying information.

Safe Harbor is necessary but not sufficient. HIPAA §164.514(b)(1) Expert Determination is the deeper de-identification control: it requires a qualified statistician to certify that the risk of re-identification by an "anticipated recipient" is "very small." The redactor handles the 18+3 identifier classes; it does not handle semantic re-identification ("the diabetic woman seen in room 4B on Tuesday") or quasi-identifier combinations (race + ZIP3 + diagnosis date) that uniquely identify a small population. Limited Data Sets per §164.514(e) have a separate carve-out for dates and geography; the redactor supports this via --redact=phi-lds. The spec section on T4 ("misdeclared artifact class") in the threat model below makes the same point in shorter form: the verifier cannot certify HIPAA compliance, only the integrity of the receipt chain that demonstrates the redactor was applied. Compliance review remains required for high-risk workloads.

The receipt-chain implication: the redactor is now a verifiable control, not a self-attested one. A regulated buyer can verify offline that every teacher call in the distillation corpus was redacted, that the placeholder mapping is consistent, that the bundled log re-hashes to the manifest claim, and that no raw identifiers leaked into the log. The artifact carries the proof. The buyer does not have to trust the builder.

7.7 Cross-vendor distillation provenance (wave 158, Q+3b)

Wave 144 Doc 7 §4 names the 2024-2026 industry pattern of distilling outputs from a frontier API (Anthropic Claude, OpenAI GPT, Google Gemini, xAI Grok) onto a small open-weight base model (Western: SmolLM2, Llama-3.2, Phi-3.5, Gemma-2, Mistral; Chinese-origin: Qwen2.5, DeepSeek-V3-distill, GLM-4, Yi-1.5). The user's framing was sharper: "how Chinese models distill American models we should have a distillation feature too." kolm offers it as a first-class CLI surface so a tenant who picks anthropic:claude-opus-4-7 → qwen2.5-0.5b ships an artifact whose receipt chain identifies exactly that teacher, exactly that student base, the HuggingFace commit revision, the distillation method, and the upstream license tag — without requiring an external lookup.

The kolm-distill-worker accepts five teacher vendors (anthropic, openai, google, xai, local) and thirteen catalog-known student bases. The single source of truth is workers/distill/catalog.mjs, which exports VENDORS + VENDOR_PROFILES + STUDENT_BASES + DISTILLATION_METHODS as JS data so third-party tooling can consume the same allow-list the worker enforces. The CLI surface kolm distill --local-worker --list-catalog dumps the catalog in a human-readable form. Catalog SKUs that have not been legally re-verified at catalog time carry a verify_before_ship: true flag, which the dump renders as a [VERIFY] marker — that flag is a build-time TODO for counsel, not a legal opinion.

RS-1 v2.0 extends manifest.training with seven cross-vendor provenance fields when the lineage source is distillation:

  • teacher_vendor — one of the five canonical vendors. Required.
  • teacher_model — the vendor's model SKU (e.g., claude-opus-4-7, gpt-5, gemini-2.5-pro, grok-3). Required.
  • teacher_version — the vendor's response version string when the API exposes one. Informational; pins the exact teacher snapshot a verifier would replay against.
  • student_base — the catalog slug (e.g., qwen2.5-0.5b) or an arbitrary HF repo when invoked with --allow-unknown-student-base. Required.
  • student_base_repo — the canonical HF repo path (e.g., Qwen/Qwen2.5-0.5B). Populated from the catalog when the slug is known.
  • student_base_origin — western or chinese. Lets a tenant filter for residency or licensing posture without parsing repo names.
  • student_base_license — the license tag (Apache-2.0, MIT, Llama-3.2-Community, Qwen-Research, Gemma-Terms-of-Use, GLM-4-License). The receipt chain self-describes the weights' license terms so a downstream consumer doesn't have to chase a HuggingFace lookup to know what they're shipping.
  • student_base_revision — the HF commit hash pin. Lets a verifier rebuild the exact corpus + weights that produced the LoRA.
  • distillation_method — one of lora, qlora, full-ft, prompt-distill. Required.

Check #15 ("Cross-vendor distillation provenance") is the verifier-side gate. It fires when manifest.lineage.source === 'distillation' and demands all four required fields above (teacher_vendor, teacher_model, student_base, distillation_method). Missing any of them is an immediate fail — the lineage block's "we distilled from X teacher" claim cannot stand without the manifest fields that name X. When all four are present, the check passes with a self-describing detail line that surfaces the full provenance so a reader can identify the teacher/student combination, the optional version pins, the license tag, and the training corpus hash without opening the lineage block:

Licensing posture. The OpenAI and Anthropic terms each restrict using model outputs to develop competing general-purpose models. The narrow-task carve-out — distilling a phone-number normalizer or denial-code classifier — is industry practice but legally unsettled; each tenant must do their own legal review. For tenants who don't want to bet on the carve-out, the always-safe path is --teacher=local:<model> against an open-weight teacher hosted on the tenant's own infrastructure (vLLM, TGI, llama.cpp server, ollama). The receipt chain records the choice either way; the license catalog flags the upstream terms so a compliance officer can see the posture at verify time without consulting an external matrix.

K-score implication (shipped in wave 160, Q+3c — see §7.8). When a cross-vendor distillation also runs --teacher-holdout, the receipt chain carries teacher_holdout_accuracy + teacher_holdout_count + teacher_holdout_log_hash. The K-score v2 envelope's T axis derives from these (T = student_holdout_accuracy / teacher_holdout_accuracy) and reports an A/T fidelity ratio. A/T = 1.0 means the student matched the teacher on the same holdout; A/T = 0.9 means the student delivered 90% of teacher accuracy at typically <5% of cost. The fidelity ratio makes the cost/quality tradeoff legible in a single number — and it depends on check #15 having confirmed the teacher identity, since "fidelity to teacher" is undefined without a named teacher.

7.8 K-score teacher-delta axis T and A/T reporting (wave 160, Q+3c)

The K-score v2 envelope ships the T axis as the distillation-honesty signal. Wave 144 Doc 7 §4.7 named it specifically: cost and offline are commodity, but a buyer cannot distinguish "5x cheaper at 95% of teacher accuracy" from "5x cheaper at 60% of teacher accuracy" unless the receipt chain records both numbers on the same holdout. Wave 145 wired the math (src/kscore.js exports computeKScoreV2 with T = student_holdout_accuracy / teacher_holdout_accuracy); Wave 158 wired the teacher-identity fields (§7.7 above); Wave 160 closes the loop with a verifier gate, an axis-rendering pass in the compliance binder, and the A/T reporting surface that makes the tradeoff legible without reading the manifest by hand.

The two inputs. The T axis is degenerate without both halves. The kolm-distill-worker writes both fields into manifest.training when invoked with --teacher-holdout=<path>.jsonl:

  • holdout_accuracy — the student model's accuracy on the holdout rows the recipe never saw at training time. Required for the R axis as well. Computed by running the compiled artifact over the holdout inputs and exact-matching against holdout outputs (or the recipe-declared comparator).
  • teacher_holdout_accuracy — the teacher's accuracy on the same holdout rows. Same comparator. The worker writes a per-row teacher-holdout-log.jsonl alongside the corpus and seals its sha256 in teacher_holdout_log_hash so a verifier can re-run the teacher offline and confirm the number.
  • teacher_fidelity_score — the T axis value, clamp01(student_holdout_accuracy / teacher_holdout_accuracy). This is what feeds the K-score V2 composite at weight 0.05 (or higher when other v2 axes are absent and weight redistributes).

Check #16 ("K-score teacher-delta (A/T)") is the verifier-side gate. It fires when manifest.lineage.source === 'distillation' and branches on the distillation method:

  • lora, qlora, full-ft — methods that produce real student weights are required to carry both holdout accuracies. Missing either is an immediate fail: an artifact that claims to ship distilled weights at 5x cost reduction but does not record what the teacher would have scored on the same holdout cannot honestly report fidelity. The fail detail names the missing field and points at the --teacher-holdout flag so the operator knows the exact remediation.
  • prompt-distill — teacher-as-judge codegen (no fine-tune; the recipe is a templated function selected for matching teacher outputs). The T axis is informational only here because there is no student model, only a function whose performance is the K-score's A axis. Missing fidelity inputs are a warn, not a fail; present inputs still render the A/T ratio for legibility.

    (student at 96.6% of teacher accuracy on the same holdout) ·
    T axis = 0.9662

# Fail — lora artifact, but holdout numbers missing
$ kolm verify suspect.kolm
  K-score teacher-delta (A/T)            fail
    distillation_method=lora requires teacher_holdout_accuracy +
    holdout_accuracy in k_score for the T axis. Re-run the distill
    worker with --teacher-holdout=path/to/teacher-holdout.jsonl so
    the receipt records the teacher's accuracy on the same holdout
    the student was scored against (Wave 144 Doc 7 §4.7).

# Warn — prompt-distill artifact, no student model, fidelity inputs absent
$ kolm verify summarizer.kolm

    distillation_method=prompt-distill (teacher-as-judge, no fine-tune);

Rendering in the compliance binder. The compliance binder's K-score section (renderKScore in src/binder.js) now switches on k_score.spec. V1 artifacts continue to render as five axes (A, S, L, C, V at weight 0.40/0.15/0.15/0.15/0.15). V2 artifacts render the five base axes plus whichever V2 axes the producer captured: R (robustness), T (teacher-fidelity), F (fairness), E (energy), Z (drift). Each axis row shows the supplied weight (after redistribution over present axes), the normalized score, a horizontal bar, and the raw evidence string — for T, the bar shows the A/T ratio directly and the evidence line reads student 0.9420 / teacher 0.9750 = 0.9662. The formula line above the axes switches to the V2 expression with a note that missing axes redistribute weight, so a reader can reproduce the composite by hand from the rendered numbers.

What this closes from the original critique. Pablo's Round 5 framing — "K = 0.917 decodes roughly to: on traffic like its training set, it echoes the big model fairly closely, and it's fast and deterministic — not 91.7% correct" — is now a property of the artifact instead of a disclaimer in marketing copy. A reader of the binder sees the A axis (raw accuracy on declared positives) next to the R axis (raw accuracy on unseen holdout) next to the T axis (student accuracy as a fraction of teacher accuracy on that holdout). The cost/quality tradeoff that the receipt chain is supposed to certify is the literal first thing on the page. The honest decode is the receipt, not a disclaimer.

7.9 Ed25519 as authoritative signer; HMAC as legacy integrity check (wave 161, Q+8)

Wave 144 §Q+8 named the existential problem in one line: HMAC-SHA256 is a symmetric MAC; anyone who can verify it can also forge it. A "signature" that requires the verifier to hold the same secret as the signer cannot prove provenance to a third party — it can only prove that two parties share a secret. For an artifact whose entire commercial value rests on a verifier-side compliance officer being able to certify provenance without trusting kolm, the HMAC must be relabeled and a public-key signature must become the contract.

Wave 149 made Ed25519 the default signer (the artifact builder loads or generates a per-machine signing key at ~/.kolm/signing-key.pem, or honors KOLM_ED25519_PRIVATE_KEY[_PATH] for CI). Wave 161 makes Ed25519 a contract. Three changes ship together:

  • manifest.policy.require_ed25519 — every new artifact carries an explicit policy field. Default true unless the build was opted out of Ed25519 (KOLM_ED25519_DISABLE=1) or out of policy enforcement (KOLM_POLICY_OPT_OUT=1). The field travels with the artifact bytes and is signed inside the same receipt body the Ed25519 block attests to, so a downstream verifier cannot strip the policy without invalidating the signature.
  • verifier check #17 "Signature policy (Ed25519)" — fires on every kolm verify. When manifest.policy.require_ed25519 === true OR the verifier sets KOLM_REQUIRE_ED25519=1, an HMAC-only receipt is a hard fail. The fail detail names the deciding side (artifact-side policy field vs. verifier-side env var), the current signature_alg, and the exact remediation flag. When the artifact carries Ed25519 and the prior check #5 verified the signature against the embedded public key, check #17 passes with the explicit framing that HMAC has been retained as a legacy integrity check, not as the primary signature.
  • HMAC relabeled in language, retained in bytes. The receipt body still carries the signature hex (HMAC-SHA256) AND the signature_ed25519 block; the binder check #6 ("Audit chain (HMAC receipt)") still verifies the chain step-by-step under the receipt secret. The relabeling is in the framing: pre-Wave-149 artifacts described HMAC as the third-party-verifiable signature. Post-Wave-161 the contract is that HMAC is an integrity check, a defense against in-flight tampering by a party who does not hold the shared secret, and the third-party-verifiable signature is Ed25519 (Wave 161, this section) or Sigstore (Wave 162, §7.10). Both layers remain because dropping HMAC would invalidate every pre-Wave-149 receipt; layering Ed25519 on top added the public-cryptography property without breaking historical bytes.
# Pass — artifact carries Ed25519 + policy.require_ed25519=true (default)
$ kolm verify health-insurance-prior-auth.kolm

    Ed25519 signature verified against embedded public key
    (fingerprint a3f7e2c1b908…); no shared secret needed
  Signature policy (Ed25519)               pass
    policy requires Ed25519 and the signature_ed25519 block verified
    against its embedded public key (HMAC retained as a legacy
    integrity check, not as the primary signature)

# Fail — HMAC-only artifact under a verifier that demands Ed25519
$ KOLM_REQUIRE_ED25519=1 kolm verify legacy.kolm

    no signature_ed25519 block — artifact built before Wave 149 or
    with KOLM_ED25519_DISABLE=1; HMAC integrity check below stands
    in until re-signed
  Signature policy (Ed25519)               fail
    env KOLM_REQUIRE_ED25519=1 but receipt carries only HMAC
    integrity (signature_alg=hmac-sha256). HMAC is a symmetric MAC
    and cannot prove provenance to a third party — re-sign with
    Ed25519 (unset KOLM_ED25519_DISABLE, or set
    KOLM_ED25519_PRIVATE_KEY_PATH=/path/to/key.pem) or relax the
    policy.

# Pass — same legacy artifact under a permissive verifier (no env opt-in)
$ kolm verify legacy.kolm

    policy does not require Ed25519; receipt carries HMAC integrity

Why an explicit policy field beats an implicit "Ed25519 if present" gate. Pre-Wave-161, the binder pushed a warn on artifacts missing the Ed25519 block, and the verdict still came back pass. A regulated tenant procuring a kolm artifact could not write "the artifact verifies under our compliance policy" without separately auditing whether the warn fired. With check #17 the artifact declares its policy stance and the verifier enforces it; the procurement contract becomes "kolm verify exits 0 against our standard environment." The escape hatch (KOLM_ED25519_DISABLE=1 at build time, KOLM_POLICY_OPT_OUT=1 for legacy issuance, no env at verify time for permissive verifiers) is preserved so historical artifacts continue to verify and so air-gapped tenants who can't manage keys aren't locked out — but the default for any new artifact is Ed25519-as-contract.

Receipt chain implication. The Wave 149 receipt chain order was: receipt body → HMAC over canonical(body) → body.signature; then Ed25519 over canonical(body INCLUDING HMAC) → body.signature_ed25519. Wave 161 does not change this ordering; it only adds manifest.policy.require_ed25519 to the manifest, which sits one level up in the artifact (the receipt's manifest_hash covers it). A downstream verifier that detects an HMAC-only receipt cannot silently substitute Ed25519, because doing so would require re-issuing the receipt under a different key — the policy field forces the substitution to be visible in the receipt body the next signer attests to.

7.10 Sigstore + Rekor transparency log (wave 162, Q+9)

Wave 144 §Q+9 closes the three-layer signature cascade: HMAC (the innermost integrity check), Ed25519 (the middle public-key signature), Sigstore (the outer transparency-log pin). Wave 150 shipped src/sigstore.js with buildSigstoreBundle, verifySigstoreBundle, and a dry-run mode that produces a structurally-valid bundle without contacting a public log. Wave 162 makes Sigstore pinning a contract: a new manifest.policy.require_rekor field, a new build-time helper attestArtifactWithRekor that pins an already-built .kolm to a Rekor instance, and a new verifier check #18 that fails closed when policy demands transparency-log pinning and the receipt only carries a dry-run bundle.

Three changes ship together:

  • manifest.policy.require_rekor — a new policy field next to require_ed25519. Defaults false because Rekor pinning requires network egress to a public log (the default Sigstore instance at https://rekor.sigstore.dev, or a tenant-private Rekor under KOLM_SIGSTORE_REKOR_URL). Builds that opt in set KOLM_REKOR_REQUIRE=1 at build time; KOLM_POLICY_OPT_OUT=1 still suppresses all policy fields. Like require_ed25519, the field travels inside the manifest and is covered by the receipt's manifest_hash, so a downstream verifier cannot strip it without invalidating the Ed25519 signature above it.
  • attestArtifactWithRekor(path) — a new helper in src/sigstore.js that reads a built .kolm, extracts receipt.json, verifies the existing dry-run sigstore bundle against the canonical receipt body (proof that the embedded signature is real, not a placeholder), submits the embedded public key + signature + digest to Rekor, and rewrites receipt.signature_sigstore.rekor_log_entry with the returned { uuid, logIndex, integratedTime, logID }. The artifact is re-emitted via archiver (NOT AdmZip.writeZip, which strips the CRC data descriptors that artifact-runner.js requires — see Wave 161 lesson). buildAndZip calls it automatically when policy.require_rekor=true or KOLM_SIGSTORE_REKOR_URL is set; standalone kolm sigstore-attest <artifact> is preserved for after-the-fact pinning of legacy artifacts.
  • verifier check #18 "Transparency policy (Rekor)" — reads two signals identically to check #17: artifact-side manifest.policy.require_rekor === true, verifier-side KOLM_REQUIRE_REKOR=1. Either being true makes Rekor mandatory. Then branches on whether the receipt carries a sigstore block, whether the bundle is dry-run, and whether the prior check #16 ("Receipt signature (Sigstore bundle)") passed. Pass/warn/fail with no skip (Wave 153 convention) — every artifact gets a row.
# Pass — artifact carries pinned Rekor entry + policy.require_rekor=true

# Fail — dry-run-only artifact under a verifier that demands Rekor
$ KOLM_REQUIRE_REKOR=1 kolm verify dryrun.kolm
  Receipt signature (Sigstore bundle)      pass
    sigstore bundle verified offline against embedded public key
    (dry-run; not anchored to a public transparency log)
  Transparency policy (Rekor)              fail
    env KOLM_REQUIRE_REKOR=1 but receipt carries a dry-run
    sigstore bundle (no rekor_log_entry). Re-attest with
    `kolm sigstore-attest <artifact>` against a reachable Rekor
    instance, or relax the policy.

# Pass — same dry-run artifact under a permissive verifier (no env opt-in)
$ kolm verify dryrun.kolm
  Transparency policy (Rekor)              pass
    policy does not require Rekor; sigstore bundle is dry-run only.
    Set manifest.policy.require_rekor=true at build time (with
    KOLM_REKOR_REQUIRE=1) or env KOLM_REQUIRE_REKOR=1 at verify
    time to upgrade the gate.

Strip-order safety. Pinning to Rekor mutates receipt.signature_sigstore.rekor_log_entry and receipt.signature_sigstore.dry_run after the HMAC and Ed25519 signatures are already affixed. This is safe because both signature verifiers strip the entire signature_sigstore block from the canonical payload before re-canonicalizing: buildBinder's Ed25519 check destructures { signature_ed25519, signature_sigstore, ...ed25519Payload }, and the HMAC check destructures { signature, signature_ed25519, signature_sigstore, ...rest }. Mutating signature_sigstore post-build invalidates neither signature. The sigstore bundle's own messageSignature stays bit-identical (only rekor_log_entry and dry_run are touched, never messageDigest or signature), so verifySigstoreBundle still passes against the same canonical payload it signed.

Why an explicit policy field beats an implicit "pin if reachable" gate. A regulated tenant whose compliance officer needs to assert "this artifact is pinned to a public transparency log" needs the assertion to be a property of the artifact, not of the build environment that produced it. Without require_rekor, a build could quietly fall back to dry-run mode when the Rekor instance was unreachable, and the downstream artifact would look indistinguishable from one pinned to a real log. With require_rekor=true, the build hard-fails when Rekor is unreachable, and the resulting artifact (if it exists) carries the policy stance the verifier enforces. The escape hatches (KOLM_REKOR_REQUIRE unset at build time, KOLM_POLICY_OPT_OUT=1, or KOLM_SIGSTORE_DISABLE=1 for sigstore-free artifacts) are preserved so air-gapped tenants and legacy artifacts continue to work — but the default for a tenant who has explicitly opted in is hard-fail on missing pin.

Receipt chain implication. The three-layer cascade is now complete. signature_alg on the receipt now ranges over hmac-sha256 (legacy), ed25519+hmac-sha256 (Wave 149+), and sigstore+ed25519+hmac-sha256 (Wave 162). The transparency-log layer answers a question the prior two layers could not: when was this artifact signed? Rekor's integratedTime is monotonic and append-only; an attacker who has stolen the Ed25519 private key cannot back-date a forged artifact to before the key was reported compromised, because the Rekor entry would post-date the revocation notice. The compliance contract becomes "kolm verify exits 0, and the integrated time on the Rekor entry pre-dates any known compromise of the signing key" — a property a third-party auditor can verify without trusting kolm and without re-running the build.

7.11 Export targets — GGUF / ONNX / CoreML / MLX / ExecuTorch / TensorRT / Safetensors (wave 163, P+6)

Wave 144 §P+6 closes the model-class compile path: a distilled_model artifact whose recipes[0].engine is one of the native runtimes (llama.cpp, onnxruntime, coreml, mlx, executorch, tensorrt) needs to bundle the actual model bytes inside the .kolm, hash them, and bind the hash into artifact_hash so a downstream verifier can prove that the artifact running on the device is the same artifact whose K-score the build pipeline recorded. The Wave 146 export-provenance bridge (src/export-provenance.js) already supplied the data path; Wave 163 closes the verifier loop, adds a one-command CLI shortcut, and raises every claim in §2 about distilled_model from "weights are addressable" to "weights are addressable AND verifiable AND bound into the artifact identity."

Three changes ship together:

  • CLI shortcut --export=<backend> --export-from <path>. kolm compile grows a flag pair that takes an already-produced native artifact (a single .gguf / .onnx / .safetensors / .pte file, or a .mlpackage / mlx_model/ / engine/ directory) and bundles it with full provenance into the .kolm. Mutually exclusive with --export-provenance: pick one. Honest scope: the shortcut does not run the Python exporter (no GGUF quantization, no ONNX conversion, no CoreML compilation). That work lives in apps/export/run.py precisely because each backend pulls a heavy ML toolchain (llama.cpp, onnxruntime, coremltools) that the JS CLI must not require to keep the rule-class path working. Run the Python exporter first; then use the shortcut to bind the result.
  • Verifier check #19 "Export targets (model files)". buildBinder grows a check that runs after the signature cascade. Three branches with pass/fail/pass-skip semantics, no opaque skip (Wave 153 convention): (a) manifest.export absent → pass with "no model files bundled (rule-class artifact)"; (b) manifest.export present and every declared target's bytes re-hash to the manifest's targets[].sha256 (file targets) or to the bridge's canonical directory hash (dir targets, computed as sha256 over sorted ${rel}\0${sha}\0${size} lines joined by newlines, matching export-provenance.js#hashDir) → pass with target count and total bytes; (c) any declared target is missing from the zip or its bytes drift from the manifest's sha256 → fail with a per-target diff. The check binds tightly to manifest.export.hash via validateExportBlock first, so a tampered manifest fails before any disk read.
  • Receipt chain implication. artifact_hash_input already carries export_hash (the export_block.hash from Wave 146), so any drift in the bundled bytes propagates into artifact_hash and through the three-layer signature cascade. Check #19 is the explicit verifier surface that makes the binding visible: a regulated tenant whose compliance review asks "prove the model.q4_k_m.gguf on this device is byte-identical to the one whose K-score we recorded at build time" can point at the check #19 row in the binder output, which states the recomputed sha256 next to the declared sha256.

7.12 External + adversarial holdouts (wave 164, N+3/N+4)

Wave 144 Doc 2 §7 laid out a five-tier eval-credibility ladder: N+1.5/Q+2 seeds.jsonl train/holdout split (shipped Wave 145, kills the tautological-eval failure mode within a single tenant's captured IO), N+3 external public benchmarks (a third party with no access to the build pipeline supplies the eval corpus), N+4 adversarial cross-family LLM-pair holdouts (a different model family paraphrases inputs the recipe was not trained on, so a recipe that overfit to one family's phrasing degrades visibly), N+5 tenant shadow corpus (a customer uploads their own holdout separately from seeds.jsonl; Wave 165), and N+7 third-party auditor attestation (Wave 166). Wave 164 ships N+3 and N+4 together because both are answered by the same primitive: a holdout corpus that lives outside the recipe's training data, identified by a catalog entry, scored by the recipe at build time, and bound into the artifact's hash chain so a downstream verifier can replay the scoring without trusting the build environment.

The shipped surface has four pieces:

  • Catalog as source-of-truth. holdouts/catalog.json enumerates every holdout the CLI knows about, one entry per corpus: { name, kind, license, source_url, accessed_at, file, suitable_for }, plus adversarial_generator_pair: { source_family, paraphrase_family } for kind="adversarial" entries. Three holdouts ship in v1.0: presidio-synthetic-v1 (external; 28 rows; CC0-1.0; patterned after Microsoft Presidio Research synthetic PHI), cms-claims-public-v1 (external; 26 rows; CC0-1.0; patterned after the CMS Medicare DE-SynPUF inpatient/outpatient/prescription/dme/other claims taxonomy), and cross-family-pair-v1 (adversarial; 26 rows; CC0-1.0; anthropic.claude source paraphrased by openai.gpt). Every entry carries an explicit license string and an accessed_at date so a compliance reviewer can audit when each corpus was last refreshed.
  • CLI flags --external-holdout <name> and --adversarial-holdout <name>. Both are repeatable and accept either form (--external-holdout name or --external-holdout=name). kolm compile resolves each name through the catalog, loads the on-disk JSONL, re-hashes its bytes against the catalog's expected file_sha256 (refusing to proceed on any drift), normalizes each row, executes the recipe against the holdout's inputs, scores via the recipe's declared comparator, and writes the result into manifest.external_holdout_provenance.
  • Manifest block external_holdout_provenance. Shape: { spec: "external-holdout-v1", comparator, holdouts: [{ name, kind, license, source_url, accessed_at, file_basename, file_sha256, normalized_hash, row_count, accuracy, passed_count, evaluated_count, suitable_for, adversarial_generator_pair }], hash }. The block-level hash is a canonical-JSON sha256 over the holdouts array plus the comparator, and is bound into artifact_hash_input.external_holdout_hash so any drift in the score, the file bytes, or the catalog metadata propagates through artifact_hash and the three-layer signature cascade (HMAC, Ed25519, sigstore).
  • Verifier check #20 "External / adversarial holdouts". Three branches, pass/warn/fail no opaque skip (Wave 153 convention): (a) manifest.external_holdout_provenance absent → pass with "no external or adversarial holdouts declared (seeds.jsonl gate is the floor)" plus an upgrade hint; (b) block present → validateExternalHoldoutBlock first (schema + hash round-trip), then for each holdout resolve the on-disk JSONL via the catalog, re-hash its bytes, and cross-check license and source_url against the live catalog entry (catches a tenant editing the catalog post-build to claim different provenance); (c) any schema drift, hash drift, file-missing, or catalog-vs-manifest mismatch → fail with a per-holdout reason string naming the exact field that drifted.

Honest scope. These are eval primitives, not benchmark corpora. A regulated tenant whose compliance review demands real-world distributions (HIPAA limited-data-set claims, signed-DUA Presidio access, MIMIC-III-style PHI) replaces the catalog entry's file with their own DUA-gated JSONL, updates file_sha256, and the verifier check anchors to their corpus. The catalog is the integration surface; the holdouts shipped in the repo are reference implementations a third party can read, audit, and replace.

K-score R axis implication. The external holdout populates the external R axis: how well the recipe generalizes to inputs drawn from a different distribution entirely. The adversarial holdout populates the adversarial R axis: how well the recipe survives cross-family paraphrase. Surfacing in the homepage and /r/<hash> receipt pages is a follow-up wave (O+ track); the data is in the manifest as of Wave 164.

Receipt chain implication. The receipt chain now reads spec → seeds → split → train → recipes → evals → external_holdout → export → signatures → rekor. Each link is hash-bound to the next; a tenant or auditor with the .kolm, the holdouts/catalog.json, and the on-disk holdout JSONL can replay the entire chain offline. The external-holdout layer specifically answers the question "did the recipe generalize beyond its own captured IO?", which the seeds.jsonl gate alone cannot answer because seeds.jsonl is by construction one tenant's data. Wave 164 makes "yes, on these CC0-licensed external corpora, with this accuracy, recomputable from these file bytes" a verifiable artifact property.

Backward compatibility. Every pre-Wave-164 artifact lacks manifest.external_holdout_provenance. Check #20's branch (a) handles this case by emitting pass with the "no external or adversarial holdouts declared" reason and an upgrade hint. No legacy .kolm regresses; the check only activates when the artifact opts in by declaring at least one holdout at compile time.

7.13 Tenant shadow corpus (wave 165, N+5)

The N+5 tier of the eval-credibility ladder closes the gap between "external holdouts the entire world can re-anchor" (Wave 164) and "tenant-private labeled data the tenant must keep on their own infrastructure" (this wave). A HIPAA-covered payer, a banking BAA holder, a 42 CFR Part 2 SUD records custodian, or any tenant whose procurement contract has a hard residency clause cannot ship their own labeled holdout corpus inside a portable artifact, because the artifact then carries the corpus everywhere it goes. They still need to prove the recipe was scored against their proprietary corpus, and prove that the manifest's recorded accuracy was computed over a hash-pinned snapshot of that corpus. Wave 165 ships the primitive: the corpus stays on the tenant's server storage; only the fingerprint {tenant_id, corpus_id, corpus_sha256, normalized_hash, accuracy, ...} rides in the manifest.

The shipped surface has five pieces:

  • Server-side storage at ${KOLM_DATA_DIR}/tenant_holdouts/<tenant_id>/<corpus_id>.jsonl. One file per corpus. Per-tenant subdir scopes naturally. Both tenant_id and corpus_id are validated to /^[a-z0-9][a-z0-9_-]{0,62}$/i to prevent path traversal. saveCorpus refuses to overwrite an existing corpus unless replace=true is passed, so a CI script that accidentally drifts the labeled set is caught instead of silently shadowing the prior version.
  • HTTP endpoints. POST /v1/eval/tenant_holdout accepts {corpus_id, rows: Array<{input,output} | {prompt,completion}>, replace?}; returns {tenant_id, corpus_id, corpus_sha256, normalized_hash, row_count, stored_at, bytes, residency_note}. GET /v1/eval/tenant_holdout lists all corpora for the authenticated tenant. GET /v1/eval/tenant_holdout/:corpus_id returns metadata for one corpus (never the rows). DELETE /v1/eval/tenant_holdout/:corpus_id removes it. All four are per-tenant scoped — the tenant_id is derived from req.tenant_record.id, never accepted as a body field, so cross-tenant access is impossible.
  • CLI flag --tenant-shadow-corpus <tenant_id>:<corpus_id>. Repeatable. kolm compile loads each named corpus from per-tenant disk via loadCorpus, normalizes its rows, executes the recipe against the corpus inputs, scores with the recipe's declared comparator, builds a per-corpus block via buildTenantShadowBlock, validates the block, and writes the array into manifest.tenant_shadow_corpus_provenance. The corpus bytes themselves are NEVER added to the .kolm zip.
  • Manifest block tenant_shadow_corpus_provenance (array). One block per shadow corpus. Shape: { spec: "tenant-shadow-corpus-v1", tenant_id, corpus_id, corpus_sha256, normalized_hash, row_count, accuracy, passed_count, evaluated_count, comparator, evaluated_at, residency_note, hash }. The per-block hash is a canonical-JSON sha256 over the other fields; the array-level binding into artifact_hash_input.tenant_shadow_corpus_hash is the canonical-JSON sha256 over the ordered [{tenant_id, corpus_id, hash}, ...] shape, so adding, dropping, swapping, or editing any corpus drifts artifact_hash and breaks every signature.
  • Verifier check #21 "Tenant shadow corpus". Three-way dual-mode pass plus two fail paths: (a) manifest.tenant_shadow_corpus_provenance absent → pass with "no shadow corpus declared" plus an upgrade hint; (b) block present + reAnchorTenantShadowBlock reports mode='reanchored' + corpus bytes match → pass with "re-anchored from tenant storage"; (c) block present + mode='unavailable' (verifier is external to tenant infrastructure) + schema/block-hash round-trip clean → pass with "external verifier; HIPAA data-never-leaves-tenant residency upheld; schema + block hash validated"; (d) block present + mode='reanchored' + bytes drift → FAIL (corpus edited post-build OR manifest tampered); (e) schema/hash drift → FAIL. The dual-mode behavior is what makes the check honest: an auditor with the .kolm but no tenant access can still verify the schema and the block hash, while a tenant-internal verifier can additionally re-anchor against the on-disk corpus.

HIPAA data-never-leaves-tenant by construction. The whole point of N+5 vs N+3/N+4 is that the corpus stays on the tenant's server. The .kolm artifact ships everywhere — pinned to a registry, mailed to a counterparty, shared with an external auditor — and at no point does it carry the corpus rows. An auditor receives the .kolm, validates the schema and the block hash, and sees "this artifact was evaluated against tenant acme's corpus claims-q2-2026 with accuracy 88.3%." They cannot read the corpus. They cannot reconstruct the corpus. They can only confirm that the manifest's fingerprint matches a corpus the manifest claims exists. To re-anchor the fingerprint against the actual bytes, they need access to the tenant's ${KOLM_DATA_DIR}/tenant_holdouts/<tenant_id>/ directory — which means either (a) they are the tenant, (b) they have a contractual right to the corpus and an inspection visit on tenant premises, or (c) the tenant cooperatively re-runs kolm verify on tenant infrastructure and shows the auditor the output.

Trust model honesty. An external verifier who cannot re-anchor the corpus bytes is trusting the tenant's build pipeline did honest work on a corpus matching the declared hash. This is a real trust assumption, and the spec is explicit about it. Mitigations: (1) the tenant can grant inspection access on demand; (2) Wave 166's N+7 third-party auditor attestation layer adds a co-signer who attests to having seen the corpus and the score on inspection day; (3) the Ed25519 + Rekor signature cascade still proves the manifest hasn't been retroactively tampered with — the only thing the corpus-side trust assumption gives the tenant is the ability to fudge the bytes at build time, and N+7 closes that.

K-score implication. Tenant shadow corpus accuracy populates the tenant-internal R axis: how well the recipe generalizes on the tenant's actual production-shaped data (which is what the tenant cares about for procurement) without exposing that data to the wider receipt-verification audience (which is what HIPAA/contracts demand). A composite of recipe_local_A × external_A × adversarial_A × tenant_shadow_A is the right thing to surface in the v2.x axis breakdown for tenants who upload shadow corpora. The data is in the manifest as of Wave 165; surfacing in homepage and /r/<hash> receipt pages is a follow-up O+ track wave.

Receipt chain implication. The receipt chain now reads spec → seeds → split → train → recipes → evals → external_holdout → tenant_shadow → export → signatures → rekor. Tenant shadow appends after external_holdout because tenant shadow accuracy is conditional on the same compiled recipe that external holdouts scored — it is one more eval signal layered on top, not a replacement.

Backward compatibility. Every pre-Wave-165 artifact lacks manifest.tenant_shadow_corpus_provenance. Check #21's branch (a) handles this case by emitting pass with the "no shadow corpus declared" reason and an upgrade hint. No legacy .kolm regresses; the check only activates when the artifact opts in by declaring at least one shadow corpus at compile time.

7.14 Third-party auditor attestation (wave 166, N+7)

The top tier of the eval-credibility ladder. Wave 161's Ed25519 builder signature (signature_ed25519) proves the artifact came off a build machine that holds the builder's private key. Waves 144 / 145 / 164 / 165 (seeds split, external holdouts, tenant shadow) bind eval-corpus provenance into artifact_hash so post-build tamper breaks every signature. None of those layers settle the question a regulated buyer always asks: "did anyone other than the vendor look at this?" N+7 closes that gap. The build pipeline accepts one or more Ed25519-signed attestation blocks, produced by parties whose key fingerprints are deliberately disjoint from the builder's, and embeds them in the manifest. The verifier re-runs every signature, re-checks every signed claim against the current artifact, and refuses any attestation whose key fingerprint matches the builder's, so a self-signed "attestation" can never masquerade as independent oversight.

The shipped surface has five pieces:

  • Module src/auditor-attestation.js. Exports AUDITOR_ATTESTATION_SPEC_VERSION = "kolm-auditor-attestation-v1", generateAuditorKeyPair(), loadAuditorKeyFromFile(path), loadAuditorKeyFromEnv() (reads KOLM_AUDITOR_ED25519_PRIVATE_KEY / KOLM_AUDITOR_ED25519_PRIVATE_KEY_PATH — a namespace deliberately separate from the builder's so a CI runner cannot accidentally sign both layers), buildAuditorAttestationBlock({signerKey, observation, identity}), validateAuditorAttestationBlock(block) (schema + Ed25519 signature + fingerprint re-derivation + block-hash recomputation), crossCheckAttestation(block, manifest) (matches signed claims against current manifest values), and the disk-I/O helpers loadAttestationFile(path) / writeAttestationFile(path, block).
  • CLI kolm auditor <keygen|sign|verify>. kolm auditor keygen --out auditor.key writes a fresh Ed25519 PEM keypair to disk (private key only — the public key is recomputed from the private on every load). kolm auditor sign <artifact.kolm> --key auditor.key --auditor-id <slug> [--accreditation "<text>"] [--scope "<text>"] [--notes "<text>"] [--out attestation.json] opens the .kolm, extracts the observation fields (artifact_hash from the receipt; cid, eval_set_hash, eval_score, artifact_class, k_score from the manifest; external_holdout_hash and tenant_shadow_corpus_hash reconstructed from artifact_hash_input), and signs them with the auditor's key. The output is a standalone JSON file the auditor hands back to the tenant. kolm auditor verify <artifact.kolm> <attestation.json> re-runs schema validation, signature verification, claim cross-check, and the auditor!=builder fingerprint check without needing access to any private key.
  • Compile-time wiring kolm compile --auditor-attestation <file> (repeatable). Each named JSON file is loaded by spec-compile.js, schema+signature-validated, embedded in manifest.auditor_attestation_provenance (array), and cross-checked against the freshly-built artifact's manifest. If the attestation's signed artifact_hash does not match the build's artifact_hash — the canonical "stale attestation" failure mode where someone tries to re-use last quarter's auditor sign-off on a rebuild that bumped any hash-bearing input — the build aborts with auditor attestation ... does not match this build and instructs the operator to re-run kolm auditor sign against the current build.
  • Manifest block auditor_attestation_provenance (array). One block per attesting party. Shape: { spec: "kolm-auditor-attestation-v1", alg: "ed25519", auditor_id, accreditation, scope, notes, artifact_hash, cid, eval_set_hash, eval_score, artifact_class, k_score, external_holdout_hash, tenant_shadow_corpus_hash, checks_passed, public_key, key_fingerprint, signed_at, signature, hash }. The Ed25519 signature covers canonicalJson(block - signature - hash); the post-signature hash field is sha256(canonicalJson(block including signature)) for the artifact_hash binding. The array-level binding into artifact_hash_input.auditor_attestation_hash is the canonical-JSON sha256 over the ordered [{auditor_id, key_fingerprint, hash}, ...] shape — so adding, dropping, swapping, or quietly editing any attestation drifts artifact_hash and breaks every downstream signature including Rekor.
  • Verifier check #22 "Third-party auditor attestation". Five branches: (a) manifest.auditor_attestation_provenance absent → pass with the upgrade hint naming the full kolm auditor keygen / sign / --auditor-attestation workflow; (b) present + every block's signature verifies + every signed claim matches the current manifest + every auditor key fingerprint differs from the builder's → pass with auditor identities, fingerprints, scopes, and re-ran check counts surfaced; (c) present + any block's Ed25519 signature fails verification (tampered signature, mutated payload, swapped public key) → FAIL; (d) present + signatures verify but a signed claim (artifact_hash, eval_score, k_score.composite, external_holdout_hash, tenant_shadow_corpus_hash) drifts from the current manifest → FAIL with the specific drift reason; (e) present + signatures verify but an auditor's key fingerprint matches the builder's → FAIL with "third-party attestation requires a distinct party." Tolerances: eval_score drift > 1e-6 fails; k_score.composite drift > 1e-4 fails; every hash is exact-match.

# Fail — auditor key fingerprint matches builder key fingerprint
$ kolm verify self-signed.kolm
  Third-party auditor attestation          fail
    manifest.auditor_attestation_provenance has 1 attestation failure(s):
      auditor_id='vendor-self-attest' uses the SAME Ed25519 key
      (5a3b7c9e1f04…) as the artifact's builder; third-party attestation
      requires a distinct party. Re-sign with an Ed25519 key not held by
      the build pipeline.

Independence guarantee, mechanically. The verifier reads the builder's fingerprint from receipt.signed_by (format ed25519:<hex32>) or receipt.signature_ed25519.key_fingerprint, and reads each auditor's fingerprint from the embedded attestation block. The two MUST differ. This is the structural property N+7 sells. An attestation block whose fingerprint equals the builder's adds zero independence — it is the vendor signing its own homework with a second pen. Check #22 (e) refuses to count such a block as a third-party attestation. The error message names both fingerprints so the operator can audit which key was reused and re-sign with a distinct one.

Multiple co-signers. The block is an array, not a single attestation. A regulated buyer's procurement workflow typically involves more than one independent reviewer (SOC 2 auditor + HIPAA compliance officer + insurance carrier risk assessor). Each runs kolm auditor sign with their own key; the tenant collects all attestations and passes --auditor-attestation multiple times. The manifest carries all of them. The verifier validates each independently and reports a per-auditor pass/fail. Removing any block post-build (e.g., quietly dropping the unfavorable reviewer) drifts artifact_hash_input.auditor_attestation_hash and breaks every downstream signature including Rekor.

Optional fields and partial attestation. Not every auditor verifies every layer. An attestation block whose external_holdout_hash is null asserts the auditor did not look at external holdouts (no claim, no check). An attestation whose k_score is null asserts the auditor did not score the artifact themselves (only confirmed eval metadata round-trip). The cross-check only fires on fields the auditor actually signed: block.field != null AND manifest.field != null AND they differ is the failure trigger; null on either side is a no-op. This lets a lightweight attestation ("we confirmed artifact_hash matches what the vendor showed us") coexist with a heavyweight one ("we re-ran the entire eval and signed off on every axis"). Both are valid; the checks_passed field documents the scope the auditor stands behind.

Stale-attestation defense. The post-build cross-check in spec-compile.js runs AFTER buildAndZip because artifact_hash is not knowable until the manifest is finalized. Any attestation whose signed artifact_hash does not equal the just-built artifact_hash fails the build immediately. This catches the canonical "tenant tries to attach last quarter's auditor sign-off to a rebuild" failure mode — re-running kolm compile with a different job_id, a different recipe, a different external holdout, or any other hash-bearing input change drifts artifact_hash, and the stale attestation is rejected before the zip is written. The verifier (check #22 branch d) re-runs the same cross-check at verify time so a third party gets the same guarantee even if the build-time check is somehow bypassed.

Auditor key custody. The auditor's private key never enters the kolm builder process. They generate it offline (kolm auditor keygen or any standard Ed25519 tooling — openssl, ssh-keygen with the right algorithm, an HSM, a cloud KMS that supports Ed25519), keep it in their own custody, and produce attestations on their own machine. The tenant only ever sees the public key (embedded in the attestation block) and the signature. There is no way for the tenant or the builder to forge an attestation, because they do not hold the auditor's private key. This is the same trust model that makes any third-party signature meaningful — the cryptography is doing the actual work, not policy.

Receipt chain implication. The receipt chain now reads spec → seeds → split → train → recipes → evals → external_holdout → tenant_shadow → auditor_attestation → export → signatures → rekor. Auditor attestation appends after tenant shadow because the attestation can sign claims about both the eval surface (external + tenant-shadow scores) and the builder's own Ed25519 signature is independent; the auditor signature layers on top, providing an independent witness to the artifact's verification outputs at sign time. A verifier with the .kolm can confirm: (1) the builder's signature is valid; (2) the Rekor entry pins the timestamp; (3) one or more independent third parties signed claims about the artifact's verification outputs that still match the current manifest; (4) those third parties are NOT the builder. That's the complete chain of custody for a regulated procurement.

K-score implication. Auditor attestation does not directly populate any axis of the K-score. It populates a separate provenance dimension — the count and identity of independent third parties who have signed claims about the artifact. Surfacing this in the homepage K-score card and /r/<hash> receipt pages as a discrete "attested by N independent parties" line (with click-through to the auditors' fingerprints and accreditations) is the natural follow-up O+ track wave.

Backward compatibility. Every pre-Wave-166 artifact lacks manifest.auditor_attestation_provenance. Check #22's branch (a) handles this case by emitting pass with the "no attestation declared" reason and the upgrade hint. No legacy .kolm regresses; the check only activates when the artifact opts in by attaching at least one attestation at compile time.

7.15 Drift detection + supersession chain (wave 167, M+3 + M+4)

The final two tiers of the lifecycle ladder Wave 144 Doc 3 §7 named. M+1 (stale notification) and M+2 (re-distillation cadence) shipped earlier. M+3 closes the question "how do we tell when a frozen artifact has drifted past tolerance against the current production distribution?" M+4 closes the question "when a tenant retires a drifted artifact, how does the successor cryptographically prove it knows about the predecessor and why it was retired?" Together they make the receipt chain re-runnable on a schedule and walkable in both directions: forward from genesis to current, backward from any successor to its predecessor's predecessor, etc., until the root.

The shipped surface has six pieces:

  • Module src/drift-supersession.js. Exports the four spec-version constants (SUPERSESSION_SPEC_VERSION = "supersession-v1", DRIFT_SNAPSHOT_SPEC_VERSION = "drift-snapshot-v1", DRIFT_REPORT_SPEC_VERSION = "drift-report-v1", DRIFT_CRON_SPEC_VERSION = "drift-cron-v1"), the SUPERSESSION_REASONS enum (drift_detected, scheduled_rebuild, security_patch, recipe_revision, policy_change, tenant_request), the DEFAULT_TOLERANCES table (eval_score warn 0.02 / fail 0.05; k_score_composite warn 0.03 / fail 0.08; k_score_axis warn 0.05 / fail 0.10; external_holdout_hash and tenant_shadow_corpus_hash exact-match; artifact_hash exact-information), and the builder/validator pairs buildSupersessionBlock / validateSupersessionBlock, buildDriftSnapshot / validateDriftSnapshot, buildDriftReport / validateDriftReport, buildDriftCronConfig / validateDriftCronConfig. Comparator detectDrift(baseline, current, tolerances) emits one signal per axis with status within (delta inside warn band), drift (warn band crossed but fail band not crossed), or breach (fail band crossed). Helper snapshotFromManifest(manifest, receipt) extracts a portable drift-snapshot-v1 from any built manifest. Disk helpers writeDriftReport(path, report) and loadDriftReport(path) round-trip JSON to/from disk with full re-validation on load. toCrontabLine(cfg) emits a 5-field cron line that wraps kolm drift detect with the same baseline + tolerances bound into the config block's hash.
  • CLI kolm drift <detect|cron|verify>. kolm drift detect <current.kolm> --baseline <baseline.kolm> [--out report.json] [--json] [--tolerances <file>] loads both artifacts, snapshots their manifests, runs the comparator, prints a human-readable signal table tagged [BREACH]/[drift]/blank, optionally writes the validated drift-report-v1 JSON to disk, and exits GATE_FAIL (exit 2) when the verdict is breach so a CI step fails loud. kolm drift cron --baseline <path> --current <path> --cadence "<5-field-cron>" [--out drift-cron.json] [--alert-webhook <url>] [--alert-on within,drift,breach] validates the cadence (5 fields, regex ^[*0-9,\-/]+$ per field), builds a hash-stamped drift-cron-v1 block, emits the crontab line a tenant drops directly into their scheduler, and prints both forms so the operator can audit what will run. kolm drift verify <report.json> re-validates a persisted report (schema, hash recomputation, both embedded snapshots) and exits GATE_FAIL on breach for use in compliance audit pipelines that take the report as input rather than re-running detection.
  • Compile-time wiring kolm compile --supersession-of <predecessor.kolm> --supersession-reason <reason> [--supersession-drift-report <report.json>] [--supersession-authorized-by <id>] [--supersession-notes "<text>"]. Loads the predecessor artifact via artifact-runner.loadArtifact, extracts receipt.artifact_hash and manifest.cid, builds a supersession-v1 block, and embeds it in manifest.supersession_provenance. When --supersession-reason=drift_detected the --supersession-drift-report flag is REQUIRED: the report is loaded, hash-validated, and the breached + drifted signals (filtered to omit within noise) are inlined into the block's drift_signals field alongside drift_report_hash so the supersession is self-contained — a verifier walking the chain doesn't need a sidecar JSON to understand why the predecessor was retired. The companion flag kolm compile --drift-report <report.json> embeds an arbitrary drift report into the new artifact's manifest.drift_report (useful for the cron pipeline that stamps the latest report onto every rebuild). Both supersession and drift-report block hashes bind into artifact_hash_input.supersession_hash and artifact_hash_input.drift_report_hash respectively.
  • Manifest block supersession_provenance. Shape: { spec: "supersession-v1", predecessor_artifact_hash, predecessor_cid?, reason, supersession_date, drift_signals?, drift_report_hash?, authorized_by?, notes?, hash }. The block names exactly ONE predecessor (chains form by walking predecessor.supersession_provenance.predecessor_artifact_hash recursively until the genesis artifact, which has no supersession_provenance field at all). Builder enforces: predecessor_artifact_hash hex64; reason ∈ SUPERSESSION_REASONS; supersession_date ISO 8601; authorized_by matches SAFE_ID; and the reason-specific guard that reason === 'drift_detected' requires either drift_signals or drift_report_hash so the retirement claim is auditable. Block hash is sha256(canonicalJson(block - hash)); binds into artifact_hash_input.supersession_hash; any tamper drifts artifact_hash and breaks every downstream signature including Rekor.
  • Manifest block drift_report. Shape: { spec: "drift-report-v1", computed_at, baseline_snapshot, current_snapshot, signals, verdict, breach_count, drift_count, tolerances?, notes?, hash }. Both embedded snapshots are themselves validated drift-snapshot-v1 objects with their own hashes, so the manifest carries a self-validating mini-receipt-chain. Verdict rollup: any breach signal → breach; any drift with no breach → drift; else within. Binds into artifact_hash_input.drift_report_hash.
  • Verifier checks #23 (Supersession chain) + #24 (Drift report). Check #23 has three branches: (a) manifest.supersession_provenance absent → pass + informational ("this artifact has no recorded predecessor") + upgrade hint quoting the full flag set; (b) present + validates → pass with predecessor hash prefix, reason, supersession date, optional cid + authorized_by, and an evidence summary (drift-signal breach/drift counts and/or drift_report_hash prefix); (c) present + schema/hash drift → fail with validator's message. Check #24 has four branches: (a) absent → pass + informational with kolm drift detect and kolm drift cron usage hints; (b) present + verdict within → pass with verdict detail; (c) present + verdict drift → pass with WARN — prefix and a "schedule re-distillation before next cadence" hint; (d) present + verdict breach → fail with the breached axes list and the remediation hint to ship a corrected successor via --supersession-of with --supersession-reason=drift_detected. Both checks emit on every artifact so the binder always shows whether they are activated, not silently absent.

Why two checks, not one. Supersession and drift report are independent receipts. A scheduled rebuild can carry a supersession block (reason='scheduled_rebuild', no drift evidence required) and no drift report at all. A drift-monitoring pipeline can stamp a drift report onto every cadence rebuild without changing the predecessor. The two-check shape lets a verifier surface each independently, so an auditor reading the binder PDF sees both rows whether the artifact is the first-in-chain (both informational), a routine rebuild with drift monitoring (drift report green, supersession green with scheduled_rebuild), or the drift-triggered successor that retires a breached predecessor (drift report on the predecessor was breach; new artifact carries supersession_provenance.reason='drift_detected' with the breached signals inlined).

Why cron config emission, not a daemon. kolm does not own a long-running scheduler process. Every regulated tenant already has one (cron, systemd timer, k8s CronJob, Airflow, GitHub Actions schedule). kolm drift cron builds a hash-stamped drift-cron-v1 config block and emits the crontab line; the tenant drops the line into their scheduler. The wrapped kolm drift detect exits non-zero on breach so cron mail delivery surfaces the alert to the operator without needing a webhook. The webhook + --alert-on filter exist for tenants who prefer push notification but they are layered on top of the same exit-code contract, not replacements for it.

Backward compatibility. Every pre-Wave-167 artifact lacks both manifest.supersession_provenance and manifest.drift_report. Checks #23 (a) and #24 (a) handle this case by emitting pass with the "no block" reason and the upgrade hint. No legacy .kolm regresses; both checks only activate when the artifact opts in at compile time. The lifecycle ladder is opt-in across the board, with each tier (stale → cadence → drift → supersession) adding new manifest blocks rather than mutating existing ones.

Receipt chain implication. The receipt chain now reads spec → seeds → split → train → recipes → evals → external_holdout → tenant_shadow → auditor_attestation → supersession → drift_report → export → signatures → rekor. Supersession and drift_report append after auditor attestation because the lifecycle layers are downstream concerns — an artifact can be attested by an auditor today and superseded by a drift-triggered rebuild tomorrow; both events bind into the receipt-chain hash and either ALL signatures verify against the manifest as written or some signature breaks. A third-party verifier walking the chain from current backwards to genesis can confirm at every step: (1) the predecessor existed and verified at the time it was named; (2) the supersession reason was legitimate and accompanied by required evidence (drift report inlined when reason='drift_detected'); (3) no quiet edit was made to the chain — the hash bindings break on any tamper. That's the complete lifecycle chain of custody.

K-score implication. Drift detection does not directly populate any K-score axis on the artifact being scored — instead, it populates the delta between two snapshots of the K-score over time. A drift report's signals include per-axis deltas (eval_score, k_score.composite, every named axis present in both snapshots) plus the exact-match check on the external holdout corpus identity and the tenant shadow corpus identity. Surfacing the latest drift verdict in the homepage K-score card (e.g., a "drift: within" / "drift: drift" / "drift: breach" pill next to the composite score) and on /r/<hash> receipt pages is the natural follow-up O+ track wave.

8. Threat model

RS-1 v2.0 defends against five attacker classes.

T1. Tampered artifact. Mitigated by manifest CID verification on load and per-entry SHA-256 verification. The artifact is hash-locked.

T2. Forged receipt. Mitigated by the three-signature cascade. HMAC catches integrity drift; Ed25519 catches third-party forgery (verifier checks against the public-key directory); sigstore with Rekor anchoring catches anyone who somehow stole the private key (transparency log proves time-of-signing).

T3. Drift attacker. A retrieval source whose contents change after the receipt is issued. Mitigated by content_sha in retrieval receipts: replay verifies the source content matched at retrieval time.

T4. Misdeclared artifact class. An artifact claiming distilled_model with empty model bytes (the e3b0c4...855 sha-256-of-empty constant) is rejected by the verifier before signature validation. A synthesized_rule claim with no teacher_vendor in manifest.training is rejected the same way. The class-mismatch error is distinct from a signature error so operators can route incident response correctly.

T5. Tautological eval. An artifact whose train_hash intersects its holdout_hash (the canonical pre-v2 failure mode where eval cases were derived from recipe code) is rejected by the verifier. The K-score's A axis is meaningless if train and holdout overlap; v2.0 makes this a hard gate.

Out of scope: side-channel timing on the inference engine; weight extraction attacks against the underlying model; rubber-hose attacks against the operator; semantic re-identification of redacted PHI via quasi-identifiers (HIPAA Expert Determination is the deeper control).

9. Reference implementations

Three reference verifiers exist, each pure-stdlib and free of network dependency for the verify path:

  • JavaScript / Node 20+ — src/binder.js, src/recipe-class.js, src/ed25519.js, src/sigstore.js. Uses node:crypto, node:fs/promises, node:zlib. The CLI kolm verify <artifact>.kolm runs the full chain.
  • Python 3.11+ — sdk/python/kolm/, mirror surface, uses stdlib hashlib, hmac, zipfile, cryptography (for Ed25519).
  • Rust (read-side) — sdk/rust/, the verifier bundled with the CLI binary release.

All three produce byte-identical canonical-JSON serializations, byte-identical CIDs, and byte-identical receipt verification results. This is enforced by an interop test suite.

10. Adoption path

RS-1 v2.0 is in active development. The format spec is public, the reference verifier is open source, and the K-score formula is fixed and citeable. Standards-body submission (IETF for CBOR receipt encoding, W3C Verifiable Credentials WG for the credential envelope) is a future consideration; it is not in progress as of this revision. Tenants adopting the format today get cryptographic verifiability and reproducible compile from real captured IO; they do not get RFC-track interop.

11. Related work

The artifact-with-receipt pattern draws on sigstore for code signing, SLSA for build provenance, and the SPIFFE identity model. The reproducibility criterion is closer to the MLPerf evaluation philosophy than to ad-hoc accuracy reporting. The receipt chain mirrors certificate transparency log semantics applied to inference. Distillation is implemented after Hinton et al. (2015) and Gu et al. (MiniLLM, 2023); HMAC over canonical body follows Bellare et al. (1996); Ed25519 over canonical body follows Bernstein et al. (2012); the sigstore Rekor anchoring follows Newman et al. (sigstore 2021). LoRA serving infrastructure is provided by Predibase LoRAX (Apache-2.0); kolm sits in the build-side category that produces signed, frozen, compiled artifacts.

12. Conclusion

An ML artifact you can hash, sign, and replay is the audit primitive the regulated AI economy is missing. RS-1 v2.0 is one proposal for that primitive. The spec is public. The verifier is open source. The K-score formula is fixed and citeable. The four-class taxonomy stops marketing copy from drifting away from implementation; the three-signature cascade gives defense-in-depth without breaking legacy verifiers; the seeds.jsonl train/holdout split kills the tautological-eval failure mode that motivated the v1.0 to v2.0 supersession. The cost of independent re-implementation in any language with stdlib hashing, ZIP handling, and Ed25519 is under two engineer-weeks.
