mobile target status

Mobile is a target, not a fake SDK page.

kolm's production surface today is the Node CLI, signed artifact runner, receipt verifier, registry APIs, and MCP server. iOS and Android adapters are planned runtime targets, but the public site should not pretend SwiftPM or Maven packages exist before they ship.

current runtime benchmark contract runtime comparison
01current

What works now.

The current runtime can inspect, verify, evaluate, run, and serve signed artifacts from the CLI. It is the honest base for design partners before native mobile wrappers exist.

Node CLI runtime

kolm run, kolm eval, and kolm inspect operate on local .kolm artifacts.

MCP serving

kolm serve --mcp exposes local artifacts as tools for agents and desktop workflows.

Receipts

Artifacts and runs are HMAC-bound today, with explicit readiness checks for required production secrets.

02roadmap

What has to ship before native mobile claims are credible.

Mobile support becomes a real claim only when these pieces exist, are versioned, and are benchmarked on named devices.

native loader

Swift and Kotlin packages.

Published packages that verify the manifest, signature, runtime policy, and receipt schema locally.

model bridge

Runtime backend per OS.

Documented Core ML, LiteRT, ONNX Runtime, or ExecuTorch target paths with fallback behavior.

battery budget

Hardware measurements.

Bundle size, warm load, p50/p95 latency, memory, battery, and thermal results on real phones.

app review

Store compliance evidence.

App Store and Play Store policy notes for downloaded artifacts, model updates, and local data access.
