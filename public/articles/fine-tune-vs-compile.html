Fine-tuning vs AI compilation: when each is the right tool

Fine-tuning a hosted model (OpenAI fine-tune, Predibase, Together) and compiling your own AI (kolm.ai) solve different problems. This article is a decision tree. The four axes that matter: portability, audit, latency budget, and lifecycle ownership.

What each tool actually does

A hosted fine-tuning service takes your training data, runs an updater against a base model on the provider's hardware, and returns a model id you call by network endpoint. The artifact never leaves their datacenter; you do not get weights. Inference billing continues per call until you stop using it. The provider can deprecate the base, deprecate the fine-tune format, or change the price.

An AI compiler takes your training data, runs a distillation against a base model on hardware you choose, and emits a file. The artifact is a signed, hashed, portable bundle of (weights + recipe + eval + receipt chain). Inference runs wherever you load the file. There is no per-call billing because there is no call to anyone else's endpoint. The provider relationship ends when the compile finishes.

The four-axis decision tree

AxisPick fine-tuning ifPick compilation if
PortabilityYou never want to move the model off the provider's network. The hosted endpoint is the deployment target.You need to run on customer hardware, in an airgap, on a phone, on a vehicle, on a forklift, or in a region the provider does not serve.
AuditYour audit posture accepts "the provider's SOC 2 report" as the chain of custody. Regulators do not ask for weights.Auditors must verify offline what was trained, on what corpus, with what eval, by whom. The receipt chain is part of your compliance program.
Latency budgetP99 of 800 ms is acceptable; the network hop is fine; cold starts on the provider's autoscaler are tolerable.You need sub-100 ms p99 in a closed loop, no network in the inference path, predictable memory footprint, no autoscaler.
Lifecycle ownershipYou are happy to track provider deprecation notices and re-run fine-tunes when the base changes under you.You want to pin the base, pin the recipe, pin the eval, and ship an artifact that runs identically until you choose to replace it.

Two clean cases

Hosted fine-tune is the right tool

Customer-support intent classifier behind a SaaS chat widget. The provider's BAA is sufficient. Latency budget is one second. The team is small and does not want to operate inference infrastructure. The model only ever runs through the provider's endpoint. Fine-tune via Predibase or OpenAI; ship the model id; move on.

AI compile is the right tool

Clinical-note summarizer that must run inside the hospital network, on PHI, with auditor sign-off. Latency budget is 200 ms. The artifact must be verifiable by the security team without trusting any third-party endpoint. The model needs to keep running for at least three years without a vendor deprecation forcing a re-fit. Compile via kolm compile --tier=3090; ship the .kolm; the receipt chain handles the auditor.

Mixed cases that look ambiguous but are not

You want fine-tuning but also want weights

This is a compile case dressed in fine-tune language. The moment "weights leave the provider" enters your requirements, the hosted services have lost the engagement. Compile.

You want a local artifact but do not want to think about hardware

kolm ships hardware-tier presets (--tier=3090 / 5090 / dgx-spark / m3-ultra-512); the tier picks base model + quant + KV-cache config so you do not pick those individually. See /runtimes. If even tier selection is too much, hosted is still right; do not fight your own preferences.

You want a local artifact for cost reasons

Run the math. For a workload below roughly one million requests/month, the per-call billing on a hosted fine-tune is cheaper than the amortized hardware + operator cost of a local artifact. Above that threshold the curves invert hard. Compile becomes obviously cheaper around five million requests/month for most workloads. If your traffic is unpredictable and may stay small, fine-tune; if your traffic is large and predictable, compile.

The kolm.ai opinion

Both tools are legitimate. The reason kolm.ai exists is that a growing share of workloads (regulated, latency-sensitive, multi-tenant ship-to-customer, defense, edge) have ended up at compile and the hosted ecosystem has been slow to acknowledge those workloads on their own terms. The pricing page calls out the kinds of customer the compile shape fits; /compare is the differentiation matrix that names every axis we win on and (importantly) the workloads where a hosted service is the right answer.

Honest scope. If your workload sits in the hosted-fits-fine column above and you call kolm anyway, you will spend more engineering time than you needed to. The compile shape carries operational ownership that hosted abstracts away. We will not pretend otherwise.

Common questions

Can I use both?

Yes. A common pattern: hosted fine-tune for an internal experiment, kolm compile for the production ship to customer hardware. The captured prompts from the hosted phase become seeds for the compile phase via kolm import openai-finetune ./training.jsonl.

What about retrieval-augmented generation?

RAG is orthogonal. It augments either tool with current documents at inference time. RAG plus fine-tune is a hosted retrieval architecture; RAG plus compile is a local retrieval architecture. The decision tree above is unchanged by whether RAG is in the loop.

Does the compile output run on CPU?

Yes for smaller artifacts (the redactor, the intent classifiers, the smaller code helpers) at acceptable latency. For larger artifacts (the 27B and 30B-A3B dense tiers) you want a recent NVIDIA / AMD / Apple Silicon GPU. See /models for the verified base catalog.
