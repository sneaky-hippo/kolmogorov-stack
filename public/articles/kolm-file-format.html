The .kolm file format, as it exists today.

A .kolm is a signed zip that co-versions the task manifest, recipe pack, eval metadata, receipt, an optional KOLMPACK behaviour pack (in the lora.bin slot), and an optional KOLMIDX lookup index (in the index.sqlite-vec slot). The current public artifact is recipe-mode. The reference fixture and the redactor / extractor / classifier examples ship populated packs and indexes that recipes call from. Native LoRA weights and sqlite-vec retrieval reuse the same slot names with richer encodings in later tiers.

Why one signed file.

An AI workflow has more than one behavior-affecting part: task spec, generated recipes, evals, verifier evidence, manifest hashes, and eventually model weights, adapters, and recall indexes. Keeping those pieces in one signed container makes drift visible.

The current implementation signs the emitted manifest, recipes, evals, model pointer, KOLMPACK behaviour pack, KOLMIDX lookup index, and receipt. Later runtime bridges substitute richer encodings into the same lora.bin and index.sqlite-vec slots without changing the high-level container contract or the receipt shape.

One file, one manifest, one receipt. Everything emitted by the current builder is fingerprinted, signed, and re-verifiable from the artifact bytes alone.

The current on-disk layout.

A .kolm is a deflate-compressed ZIP archive. The current builder emits pointer records, KOLMPACK / KOLMIDX containers, and JSON members. Empty entries appear when a recipe needs no pack or no index.

# unzip -l redactor.kolm
Length        Date    Time    Name
--------  ----------  -----   ----
   1,247  2026-05-09  09:14   manifest.json
     388  2026-05-09  09:14   model.gguf            # base-model pointer record
   1,612  2026-05-09  09:14   lora.bin              # KOLMPACK\x01 ·patterns + replacements
   2,094  2026-05-09  09:14   recipes.json          # recipe pack (sandboxed JS)
     287  2026-05-09  09:14   index.sqlite-vec      # KOLMIDX\x01 ·keyword to recipe map
     842  2026-05-09  09:14   evals.json            # compile evals (6 cases)
   1,024  2026-05-09  09:14   receipt.json          # HMAC receipt chain (5 steps)
      64  2026-05-09  09:14   signature.sig         # manifest HMAC

The current builder hashes each emitted component and binds those hashes into a receipt that can be re-checked from the artifact bytes.

The manifest.

The manifest is the artifact index. Its most important truth today is that runtime is still cloud, while local model execution is the bridge path.

{
  "spec": "kolm-1",
  "task": "summarize support tickets in our voice",
  "runtime": "cloud",
  "tier": "recipe",
  "recipes": { "n": 42, "registry_hash": "sha256:..." },
  "lora": null,
  "recall": null,
  "evals": { "n": 12, "hash": "sha256:..." },
  "hashes": {
    "model_pointer": "sha256:...",
    "recipes_json": "sha256:...",
    "lora_bin": "sha256:e3b0...",
    "index_bin": "sha256:e3b0...",
    "evals_json": "sha256:..."
  }
}

Payload components.

model.ggufpointer today

A JSON pointer record naming the base model and runtime stage. Real GGUF weights become a claim only when the bridge writes and verifies real model bytes.

lora.binKOLMPACK or empty

Behaviour pack slot. Today the recipe tier ships a length-prefixed UTF-8 KOLMPACK\x01 container (patterns, default rules, default categories, fallback labels) that recipes read as lib.pack. Empty when a recipe needs no pack. LoRA-delta tier reuses the slot name with a binary delta encoding.

recipes.jsoncurrent

The recipe pack emitted by the current compiler path.

index.sqlite-vecKOLMIDX or empty

Lookup index slot. Today the recipe tier ships a KOLMIDX\x01 container (keyword-to-recipe maps, by_recipe reverse index) that recipes read as lib.index. Empty when no index is needed. Retrieval tier replaces it with a real sqlite-vec database while keeping the slot name.

evals.jsoncurrent

The eval evidence used to recompute the release score and verify the artifact.

receipt.jsonHMAC-SHA256

The HMAC receipt over artifact hash, eval-set hash, score, judge id, and chain steps.

Receipt chain.

Each artifact carries a receipt chain over the fields emitted by the current builder: task, training stats, recipes, evals, and package hash. The anchors array is empty today and reserved for future external publication.

This chain is useful as retained evidence: if an artifact byte changes, recomputing the hashes breaks the receipt. Public or third-party anchoring is a future extension.

FAQ.

Is the format open?

The format is intended to be open, but the public spec/repository should be treated as pending until the canonical repo is published. The current CLI is the reference reader for the artifacts this service emits.

What about model weights?

Current public artifacts do not embed redistributable model weights. When the model bridge ships, each base model will carry its own license, and the manifest must identify the base family, hash, and redistribution terms.

Can I run a .kolm without the kolm CLI?

The current supported path is the kolm CLI. Third-party runtimes are a good future outcome, but they need the published spec, fixture artifacts, and conformance tests before they are a real integration surface.
