The .kolm artifact, field by field

A tour of the .kolm container format defined by RS-1. Magic bytes, manifest block, weights section, receipt chain, frozen-eval payload, recipe block, signature trailer. Every field exists to prove something to a verifier offline.

File layout in one picture

+------------------------+ offset 0
| MAGIC "KOLM" + version | 8 bytes
+------------------------+
| Manifest block (JSON)  | variable, length-prefixed
+------------------------+
| Weights section        | tensor-by-tensor, GGUF-compatible
+------------------------+
| Recipe block (TOML)    | the spec the compile ran
+------------------------+
| Frozen-eval payload    | questions + reference answers + score
+------------------------+
| Receipt chain (JSON)   | Merkle tree of inputs and steps
+------------------------+
| Signature trailer      | Ed25519 over the SHA-256 of all prior bytes
+------------------------+ EOF

Magic + version

First 4 bytes are 4B 4F 4C 4D (ASCII "KOLM"). Next 4 bytes are a little-endian uint32 spec version. RS-1 v2.1 (current) writes 0x00020100. A verifier that does not understand the spec version must refuse the artifact rather than silently fall through.

Manifest block

A length-prefixed JSON object. The manifest is the cheap front matter; the rest of the file is heavy. Indexers, web UIs, and CLI inspect read only the manifest and skip the rest.

{
  "spec_version": "RS-1/2.1",
  "artifact_id": "my-clinical-summarizer",
  "artifact_hash": "sha256:8e3f...",
  "base_model": "qwen3-27b-dense",
  "base_hash": "sha256:c12a...",
  "params_b": 27.4,
  "quant": "Q4_K_M",
  "context": 8192,
  "tier": "3090",
  "recipe_hash": "sha256:91b7...",
  "eval_score": 0.873,
  "eval_threshold": 0.85,
  "build_unix": 1747584321,
  "signed_by": "kolm-key-id:4f...e2",
  "supersedes": null,
  "redactor": "phi-redactor.kolm@sha256:71c4..."
}

Every hash in this block is a Merkle root over the corresponding section. Verification recomputes them from the file body and compares.

Weights section

Tensor blocks in a GGUF-compatible layout (kolm chose GGUF over safetensors because llama.cpp runtimes are the cheapest production deployment target, and the receipt chain works over either). Each tensor has a name, a dtype, a shape, and a payload. The section header is a flat index so the runtime can mmap weights without parsing the whole file.

Recipe block

TOML, by design. Recipes are human-edited. JSON would be machine-clean but human-hostile for nested config.

[spec]
class = "summarizer"
language = "en"
domain = "clinical-notes"

[base]
model = "qwen3-27b-dense"
quant = "Q4_K_M"

[seeds]
file = "seeds.jsonl"
sha256 = "..."

[capture]
namespace = "my-clinical-summarizer"
min_pairs = 1000
redactor = "phi-redactor.kolm"

[eval]
file = "frozen-eval.jsonl"
threshold = 0.85
extractive_baseline_max = 0.07

[ship]
tier = "3090"
license = "proprietary"
signing_key = "kolm-key-id:4f...e2"

Frozen-eval payload

The eval was frozen at recipe-spec time. The payload contains the question set, the reference answers, the scoring rubric (k-score by axis), and the score the artifact achieved at build. An auditor re-runs the eval against the shipped weights and verifies the score matches within tolerance.

{
  "eval_hash": "sha256:...",
  "items": [
    { "id": 1, "input": "...", "reference": "...", "axes": ["accuracy","brevity"] },
    ...
  ],
  "score_at_build": 0.873,
  "axis_scores": { "accuracy": 0.91, "brevity": 0.84 },
  "extractive_baseline": 0.04
}

Receipt chain

A Merkle tree. Leaves are the hashes of every input that informed the build; internal nodes are concatenated-then-hashed; the root sits in the manifest. The 14 canonical leaves (as of RS-1 v2.1):

  1. spec_hash
  2. seeds_hash
  3. split_hash (train and holdout disjointness proof)
  4. train_hash (post-redaction corpus)
  5. recipes_hash
  6. eval_hash
  7. external_holdout_hash
  8. tenant_shadow_hash
  9. auditor_attestation_hash
  10. supersession_hash (or zero-hash if this is a fresh build)
  11. drift_report_hash (set at first drift run, zero at build)
  12. export_hash
  13. signatures_hash
  14. rekor_inclusion_hash (transparency log entry)

Signature trailer

The trailer is exactly 96 bytes: a 32-byte Ed25519 public key id, a 64-byte Ed25519 signature. The signature is over the SHA-256 of every byte from the magic header through the receipt chain. A verifier fails if the trailer is the wrong length, the public key is not in its trust set, or the signature does not validate. There is no algorithm-agility byte; if the file does not start with KOLM and end with a 96-byte Ed25519 trailer, it is not a valid artifact for RS-1 v2.1.

What kolm verify actually does

  1. Read the manifest. Confirm the spec version is recognized.
  2. Read the signature trailer. Confirm the signing key is in the trust set.
  3. Recompute the SHA-256 over all prior bytes. Verify the Ed25519 signature.
  4. For each section, recompute the Merkle root and compare against the manifest hash.
  5. Replay the frozen eval against the loaded weights; confirm score matches within tolerance.
  6. If a supersession entry is present, walk back to the prior artifact and confirm the chain.
  7. If a Rekor inclusion hash is present, optionally check the transparency log.

Exit zero on full pass. Exit two with a structured reason on any mismatch.

Forward compatibility. A new RS-1 minor version may add fields to the manifest block; verifiers MUST ignore unrecognized fields. A new major version requires a new verifier. Spec-version bumps are not silent.

Common questions

Why not safetensors?

Safetensors is a fine weights container; it is not an artifact format. It carries no recipe, no eval, no receipt chain, no signature. The .kolm container wraps safetensors-style tensor storage with the audit envelope. A pure safetensors output is available via kolm extract --weights-only if you only want the weights.

What is the size overhead?

For a 27B Q4_K_M artifact (weights are ~14 GB) the manifest is 1.5 KB, the recipe is 800 bytes, the frozen-eval payload is 50 to 500 KB depending on item count, the receipt chain is 4 KB, the signature is 96 bytes. Overhead is in the low single-digit megabytes. Negligible.

Can I edit the manifest after build?

No. Any edit invalidates the signature and the file fails kolm verify. Reissue requires a new compile or a kolm sign --resign with the new signing key (which records the resign event in the receipt chain).
