04 - Your namespace, your data

Approved context. Compiled into the artifact.

Recall is the grounding layer: approved context travels with the artifact so runtime behavior can be reviewed where the data already lives. kolm compile --data ingests a corpus into the artifact; kolm inspect shows exactly what was bound; kolm recall queries the embedded index without leaving the boundary.

# compile a task with approved local context
$ kolm compile "answer questions about our infra" \
  --data ./my-data \
  --examples ./infra.examples.jsonl \
  --out ./artifacts

# inspect what the artifact declares
$ kolm inspect ./artifacts/infra.kolm

# standalone recall query against the embedded index
$ kolm query "how do we rotate the prod KMS key?" \
  --artifact ./artifacts/infra.kolm

01 - Ingest anything · target architecture

Multimodal sidecars.

Shipped in v0.1: local kolm rag builds a deterministic pure-JS BM25 index over text/code, hosted /v1/embed bridges to qmd when configured. Target architecture below · each non-text modality gets a Markdown sidecar so a unified retrieval layer can address pixels, frames, and waveforms; image captioning, transcript, and scene-detection adapters are roadmap.

.txt .md .ts .py

Text and code · shipped

Local BM25 today (deterministic, pure-JS). Vector embedding via bge-m3 is the target architecture; available through qmd bridge when configured server-side.

.png .jpg .heic

Audio · target architecture

Target: scene-detected, then per-shot caption + ASR + CLIP. Today: placeholder sidecars; scene-detection and per-shot adapters are roadmap.

02 - Hybrid retrieval · target architecture

BM25 plus vector. RRF, then rerank.

Shipped today: local pure-JS BM25 via kolm rag, plus a qmd bridge for hosted vector/hybrid when the server has qmd configured. Target architecture below · the unified pipeline that fuses lexical + vector via RRF and reorders the top with a cross-encoder is the v0.2 roadmap. Today, kolm.wrap grounds against whichever backend the namespace actually points at.

query → BM25 top-50 + vector top-50 → RRF fusion → cross-encoder rerank → top-k

Every retrieved chunk carries its source path, byte range, modality, and score. The compiler quotes the source verbatim and the receipt anchors each citation to a content hash. Hallucinated citations are detectable end-to-end.

03 - Three back-ends, one interface · target architecture

Same SDK. Same wire format.

Pick the back-end at the boundary; the rest of the stack does not notice. In v0.1, the on-device slot inside each .kolm is an optional KOLMIDX JSON lookup (filename index.sqlite-vec); a real on-device sqlite-vec index is a roadmap tier. Pinecone and Trieve adapters below are target architecture · today the hosted path is the qmd bridge when configured.

Managed · target

Pinecone

Target architecture for Cloud-compiled artifacts: per-tenant namespaces, regional residency, no infra. Adapter is roadmap.

KOLMOGOROV_INDEX_BACKEND=pinecone
Self-host · target

Trieve

Target architecture for self-hosted hybrid retrieval. Single Docker image. BYO embedding service or use the bundled bge-m3. Adapter is roadmap.

KOLMOGOROV_INDEX_BACKEND=trieve KOLM_TRIEVE_URL=https://trieve.your-vpc.local
On-device · target

sqlite-vec

Target architecture for the slot inside every .kolm at runtime. Today the slot carries an optional KOLMIDX JSON lookup container; a populated sqlite-vec index is roadmap.

KOLMOGOROV_INDEX_BACKEND=sqlite-vec

04 - Receipts on every chunk

Citations you can verify.
