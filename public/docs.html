Quickstart

Sign up, get a key, compile a redaction function from two examples, run it on real input. Six lines.

$ curl -sX POST https://kolm.ai/v1/signup \
    -H 'content-type: application/json' \
    -d '{"email":"you@example.com","plan":"free"}'
$ export KOLM_KEY=ks_...                            # from the response above
$ curl -sX POST https://kolm.ai/v1/synthesize \
    -H "authorization: Bearer $KOLM_KEY" -H 'content-type: application/json' \
    -d '{"name":"redact","positives":[{"input":"token=sk_live_xxx","expected":"[REDACTED]"}]}'
$ curl -sX POST https://kolm.ai/v1/run \
    -H "authorization: Bearer $KOLM_KEY" -H 'content-type: application/json' \
    -d '{"concept_id":"cpt_...","input":"my token=sk_live_abc123"}'

Synthesize

Turn a list of input/expected pairs into a compiled JavaScript function that classifies, redacts, or transforms. Returns a concept_id and a quality score.

POST/v1/synthesizebearer
curl -X POST https://kolm.ai/v1/synthesize \
  -H "authorization: Bearer $KOLM_KEY" \
  -H 'content-type: application/json' \
  -d '{
    "name": "redact-secrets",
    "positives": [
      { "input": "token=sk_live_xxx", "expected": "[REDACTED]" },
      { "input": "AWS_KEY=abcd",      "expected": "[REDACTED]" }
    ]
  }'
response
{
  "accepted": true,
  "source": "function generate(input, lib) { ... }",
  "quality_score": 1,
  "pass_rate_positive": 1,
  "reject_rate_negative": 1,
  "latency_p50_us": 22,
  "size_bytes": 412,
  "concept_id": "cpt_mp11ol06a3d4c1ae21ab3e70"
}

K-score gate: 0.85 by default. Sub-gate compiles return accepted: false with the failing axis named. Re-submit with more examples or adjusted spec.

Run

Execute a compiled concept against fresh input. Returns the output and a signed receipt that anyone with the same key can re-verify.

POST/v1/runbearer
curl -X POST https://kolm.ai/v1/run \
  -H "authorization: Bearer $KOLM_KEY" \
  -H 'content-type: application/json' \
  -d '{ "concept_id": "cpt_mp11ol06a...", "input": "my token=sk_live_abc123" }'

  "output": "token=[REDACTED]",

Recall

BM25 sidecar retrieval over content you have attached to the concept via kolm rag attach. Returns top-k chunks for the query.

POST/v1/recallbearer
curl -X POST https://kolm.ai/v1/recall \
  -H "authorization: Bearer $KOLM_KEY" \
  -H 'content-type: application/json' \
  -d '{ "query": "redact" }'
response · empty when no sidecar attached
{ "namespace": "default", "n": 0, "chunks": [] }

Assistant (natural language)

One endpoint that maps a sentence to a deterministic action: status, list builds, compile, install harness, change plan, doctor. No LLM call. Rule-based intent parser, free to run, airgap-safe.

POST/v1/assistantbearer
curl -X POST https://kolm.ai/v1/assistant \
  -H "authorization: Bearer $KOLM_KEY" \
  -H 'content-type: application/json' \
  -d '{ "prompt": "show my builds" }'
response
{
  "ok": true,
  "intent": "list",
  "narration": "No builds yet. Try \"compile a recipe that does <task>\" to create your first one.",
  "data": { "items": [] }
}

Recognised intents: status, usage, list, compile, run, tune, evolve, install, upgrade, doctor, help.

Receipts

Every /v1/run emits an HMAC-SHA256 chain: source_hash · input_hash · output_hash · version_id · issued_at. The HMAC is computed with your tenant's receipt secret. Anyone holding that secret can re-verify offline (Ed25519 public-key receipts are roadmap).

# offline verifier (Node)
import crypto from "crypto";
const r = receipt;       // the receipt object from /v1/run
const secret = process.env.KOLM_RECEIPT_SECRET;
const payload = [r.spec, r.source_hash, r.input_hash, r.output_hash,
                 r.version_id, r.runtime_version, r.issued_at].join("|");
const expected = crypto.createHmac("sha256", secret).update(payload).digest("hex");
console.log(expected === r.hmac ? "ok" : "tamper");

.kolm format

A signed zip with five files: spec.json (the recipe), policy.json (gates), recall/ (optional sidecar), verifier.js (cheap output check), receipt.json (the HMAC chain). Full schema at /spec.

K-score

One scalar per concept, gated at 0.85 to ship. Computed as 0.40·A + 0.15·(S + L + C + V) where A=accuracy, S=specificity, L=latency, C=cost, V=verifiability. Full breakdown at /k-score.

CLI

Eight verbs cover the loop. Same auth as the API; kolm login stores the key in ~/.kolm/config.json.

kolm init              # scaffold kolm.yaml + examples/
kolm compile           # synthesize → sign → write .kolm
kolm run file.kolm     # execute, emit receipt
kolm tune --airgap     # train a LoRA adapter from captured runs
kolm rag attach docs/  # build BM25 sidecar
kolm serve --mcp       # local MCP server for Claude Code / Cursor
kolm install claude-code   # wire up an agent harness
kolm doctor            # check env, keys, port, version

MCP serve

kolm serve --mcp exposes your local artifacts as MCP tools over stdio. Auto-discovered by Claude Code and Cursor after kolm install claude-code.

kolm serve --mcp --port 7777
kolm install claude-code --apply       # registers the MCP server in ~/.claude/mcp.json

Hooks

Six lifecycle hooks declared in kolm.yaml: PreCompile, PostCompile, PreRun, PostRun, PreBench, PostBench. Each runs a shell command with hook context on stdin.

# kolm.yaml
hooks:
  PreCompile: ./scripts/lint-examples.sh
  PostCompile: ./scripts/notify-slack.sh
  PreRun: ./scripts/redact-pii.sh
  PostRun: ./scripts/log-to-splunk.sh
