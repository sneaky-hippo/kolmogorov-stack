tutorial 1First .kolm in 5 minutes

Install the CLI, scaffold a project, compile one recipe, look at what landed on disk. No keys required for the local path; add one if you want to push to the cloud registry.

1Install the CLI
npm i -g github:sneaky-hippo/kolmogorov-stack
kolm version
2Scaffold a project
mkdir my-redactor && cd my-redactor
kolm init
# writes kolm.yaml + examples/ + .gitignore

The generated kolm.yaml declares your task, your examples directory, and the K-score floor (default 0.85). Open it. Edit the task line. Drop two or three (input, expected) pairs into examples/.

3Compile
kolm compile
# reads kolm.yaml, synthesizes a verifier, runs k-sample, signs, writes my-redactor.kolm

Output looks like this:

compile     my-redactor
synthesize  redact-secrets         pass_pos=1.000  reject_neg=1.000
K-score for my-redactor.kolm  0.917   floor=0.85  status=accepted
sign        hmac-sha256            8f3e2a...
write       my-redactor.kolm       size=14.2KB
4Inspect the artifact
kolm inspect my-redactor.kolm

You will see the manifest (spec/policy/verifier/receipt entries), this artifact's K-score breakdown by axis (per-artifact, not per-model), the signature, and the deterministic source hash. The .kolm file is a signed zip; you can unzip it and read every byte. Full schema at /spec.

What you just shipped: a single signed file that runs locally with kolm run my-redactor.kolm, emits a per-call receipt, and can be re-inspected forever without our servers.

tutorial 2Ship to production with CI/CD

Wire kolm compile into a GitHub Actions workflow. The job fails if the K-score floor drops below 0.85. The compiled artifact uploads as a workflow artifact you can download or push to your registry.

1Add your kolm key as a repo secret

In repo settings → Secrets and variables → Actions, add KOLM_KEY. Only needed if you push to the cloud registry; pure local compile + gate works without it.

2Create the workflow file

Drop the following at .github/workflows/kolm-gate.yml:

name: kolm-gate

on:
  pull_request:
    paths:
      - 'kolm.yaml'
      - 'examples/**'
      - 'recipes/**'
  push:
    branches: [main]

jobs:
  compile-and-gate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'

      - name: Install kolm CLI
        run: npm i -g github:sneaky-hippo/kolmogorov-stack

      - name: Compile and gate at K >= 0.85
        env:
          KOLM_KEY: ${{ secrets.KOLM_KEY }}
        run: |
          kolm compile --json > compile.json
          K=$(node -e "console.log(require('./compile.json').k_score)")
          echo "k-score: $K"
          node -e "process.exit(require('./compile.json').k_score >= 0.85 ? 0 : 1)"

      - name: Upload .kolm artifact
        if: success()
        uses: actions/upload-artifact@v4
        with:
          name: kolm-build
          path: '*.kolm'
          retention-days: 30

Push it. Open a PR that touches examples/. The job runs, prints the K-score, and either ships the green check or fails the merge.

3Optional: push the artifact to your registry
      - name: Publish to kolm registry
        if: github.ref == 'refs/heads/main'
        env:
          KOLM_KEY: ${{ secrets.KOLM_KEY }}
        run: kolm publish *.kolm
Why this gate is honest: the K-score is computed by a deterministic verifier, not a model. Same inputs always produce the same score. CI is a fair judge.

tutorial 3Run your model inside Cursor

The kolm CLI exposes your compiled .kolm artifacts as MCP tools over stdio. Cursor, Claude Code, Continue, and Cline all support MCP. One command wires it up.

1Pick your harness and install

For Cursor:

kolm install cursor --apply
# writes mcpServers.kolm into ~/.cursor/mcp.json

For Claude Code:

kolm install claude-code --apply
# writes mcpServers.kolm into ~/.claude/settings.json

Other supported harnesses: continue, cline. Drop --apply to dry-run and print the JSON block instead.

2Restart the editor

Cursor and Claude Code spawn MCP servers as subprocesses at startup. Quit and relaunch to pick up the new kolm entry. You should see kolm in the tool list when you open the command palette or agent panel.

3Verify the wiring
kolm doctor
# checks env, keys, port (if --http), MCP harness registration

You can also run the MCP server by hand on a port for inspection:

kolm serve --mcp --http
# stdio + optional HTTP JSON-RPC on 127.0.0.1:8765
curl -sX POST http://127.0.0.1:8765 \
  -H 'content-type: application/json' \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/list"}'
4Use it from the agent

Open Cursor's agent. Ask: "Use the kolm.run tool with my-redactor.kolm on this input: token=sk_live_test." The agent invokes the local kolm server, which executes the artifact and returns the redacted output plus a signed receipt. The agent never sees your data; it sees the result of running your signed recipe on your machine.

What you just built: a local tool the model can call. The recipe is deterministic. The signature proves the recipe didn't mutate. The receipt proves the call ran.

Receipts

Shipped today: each .kolm artifact carries a single HMAC-SHA256 signature over the archive contents (manifest + layer hashes); each /v1/run response returns a _kolm metadata block with source_hash · input_hash · output_hash · version_id · issued_at that the issuer or any holder of the shared tenant receipt secret can re-verify. Target architecture: per-call HMAC chaining across requests and Ed25519 public-key receipts are roadmap, not shipped.

# offline verifier (Node)
import crypto from "crypto";
const r = receipt;       // the receipt object from /v1/run
const secret = process.env.KOLM_RECEIPT_SECRET;
const payload = [r.spec, r.source_hash, r.input_hash, r.output_hash,
                 r.version_id, r.runtime_version, r.issued_at].join("|");
const expected = crypto.createHmac("sha256", secret).update(payload).digest("hex");
console.log(expected === r.hmac ? "ok" : "tamper");

.kolm format

A signed zip with five files: spec.json (the recipe), policy.json (gates), recall/ (optional sidecar), verifier.js (cheap output check), receipt.json (the HMAC chain). Full schema at /spec.

K-score

Per-artifact, not per-model. Every compiled .kolm gets its own K-score from its own eval set; the score gates that artifact, not the base model underneath. One scalar, gated at 0.85 to ship. Computed as 0.40·A + 0.15·(S + L + C + V) where A=accuracy, S=specificity, L=latency, C=cost, V=verifiability. Full breakdown at /k-score.

Remote access

The default deployment is local. kolm serve --local binds the runtime to your machine; your laptop, phone, or coding agent reach it over your own network. Your prompts and outputs never touch our infrastructure because there isn’t any to touch.

For teams that want hosted access without trusting a vendor, run the same .kolm inside a TEE (Trusted Execution Environment). Hardware enclaves (AWS Nitro, GCP Confidential, Azure SEV) seal the model, its weights, and the request data so neither kolm nor the cloud provider can read them. Relatively secure, relatively private. Not a cryptographic guarantee, but the strongest honest framing for cloud-hosted inference.

What kolm does not do: end-to-end encryption (traffic is local, so there’s nothing to encrypt in flight); managed-mesh networking (use your own Tailscale / VPN / private network); or a kolm-operated SaaS that sees your data.

Training presets

Nine named presets cover the training surface. Each one is a YAML block in kolm.yaml; the trainer reads them and dispatches to the right module under apps/trainer/. Most users never touch a hyperparameter.

# kolm.yaml
training:
  preset: lora-fast            # one of: sft, lora-fast, long-context, vlm,
                               #         merge-adapters, embed, fc-tools,
                               #         grpo-reasoning, instant
  base_model: Qwen/Qwen2.5-3B-Instruct
  k_threshold: 0.85

What each preset does:

  • sft. full supervised fine-tune. For when you own the data and want the strongest single-domain model. Slowest, highest VRAM.
  • lora-fast. PEFT LoRA, rank 16. Default for most concepts. 5x faster than SFT, 20x smaller adapter.
  • long-context. YaRN / Linear PI / NTK scaling. Push base context 4k → 32k+ with one knob.
  • vlm. vision-language LoRA, Qwen2.5-VL family. Vision tower frozen by default.
  • merge-adapters. linear / SLERP / DARE / TIES merges over two or more LoRA artifacts.
  • embed. embedding model training. InfoNCE + Matryoshka heads.
  • fc-tools. function-calling SFT, Hermes-FC schema + tolerant regex parser.
  • grpo-reasoning. GRPO online RL with verifiable rewards. DeepSeek R1 pattern.
  • instant. zero-data adapter. Synthesize examples via teacher LLM + verifier loop. Two minutes, no labels.

Reference articles for each module live under /research. The frontier-stack rail there cross-links to the implementation file and the paper.

Instant adapter

The fastest path from "I have a task" to "I have a signed .kolm." No labeled data, no GPU. The CLI proposes a verifier from your task description, asks a teacher model for candidate pairs, accepts only the ones the verifier passes, and ships the result.

kolm instant "redact patient names and phone numbers" \
  --n 64 --teacher gpt-4o-mini --k 0.90 --out phi.kolm

Status keys in the JSON output:

  • ready. verifier converged at or above your K threshold. You can ship.
  • needs_teacher. set KOLM_TEACHER_BASE, KOLM_TEACHER_KEY, KOLM_TEACHER_MODEL (OpenAI-compatible). Any vendor.
  • low_quality. reject rate too high. Tighten the task description or lower --n for a faster iteration loop.

The verifier is picked automatically by keyword (regex for redact / strip / mask, json_schema for extract / classify, answer_key for label / route). Pass --schema schema.json to override.

kolm improve

Continuous compounding. Run kolm improve on any existing .kolm: it captures the last N failed receipts from your runtime, distills them into new training pairs, recompiles, and only swaps the live artifact if the new K-score beats the old by at least --margin.

kolm improve phi-redactor.kolm --window 24h --margin 0.01 --auto-swap

The hot-swap is a symlink update plus SIGHUP to running servers. Zero downtime, fully reversible: the prior artifact stays on disk and kolm rollback restores it in one command.

SDKs. embed in 4 platforms

Each SDK is a thin wrapper around the runtime loader. load() returns a callable; predict() runs the artifact; receipts emit on every call. No vendor-specific glue; the artifact is the contract.

Python. pip install kolm

from kolm import load

model = load("phi-redactor.kolm", verify="on")
out = model.predict("Patient John Doe, MRN 8847-21, called from 555-201-8842.")
print(out.text)        # "Patient [NAME], MRN [MRN], called from [PHONE]."
print(out.receipt.cid) # "cidv1:sha256:..."

Backend autopick order: gguf (llama-cpp-python) → onnx (onnxruntime) → remote (KOLM_RUNTIME_URL) → transformers + peft. verify="strict" raises on signature or CID mismatch; verify="off" skips for trusted local artifacts. Default is "on". check signature, warn-and-continue on missing.

Swift. SwiftPM / iOS & macOS

import Kolm

let model = try await Kolm.load(named: "phi-redactor")
let out = try await model.predict("Patient John Doe...")
print(out.text)

Bundles look for .kolm in the app bundle. Backend pick: CoreML (.mlpackage) → MLX (.mlx) → GGUF (.gguf). All three are unpacked from the signed zip at first load and cached under Library/Caches/kolm/<cid>/. KolmConfiguration.shared.hmacSecret binds your verification key.

Kotlin. Android / Gradle

import ai.kolm.Kolm

val model = Kolm.load(context, "phi-redactor.kolm")
val out = model.predict("Patient John Doe...")
Log.d("kolm", out.text)

The Kotlin SDK ships as an AAR with no runtime deps; bring your own llama.cpp / ExecuTorch / ONNX runtime AAR. The loader prints crisp errors when a backend is missing: kolm: no runtime found. add net.java.jllama:jllama or org.pytorch:executorch-android to dependencies.

React Native. react-native-kolm

import Kolm from 'react-native-kolm';

await Kolm.setConfig({ verify: 'on', hmacSecret: process.env.KOLM_SECRET });
const model = await Kolm.load(require('./phi-redactor.kolm'));
const out = await model.predict('Patient John Doe...');
console.log(out.text);

HTTPS sources cache via react-native-fs under CachesDirectoryPath. iOS dispatches to the Swift backend; Android to the Kotlin backend. One API surface, two native paths.

Compliance packs

A compliance pack is a versioned bundle of (recipe.json + evidence.md + dpia.md + mrm.md + controls.md). It encodes the regulatory shape of a task. what redactions are mandatory, which fields must validate, what auditors will ask. so your engineering team builds against the same target your legal team signed off on.

Three packs ship today. Each lives under data/compliance-packs/<pack>/.

HIPAA. 2-hour onboarding path

Pack: hipaa-phi-redactor. K-threshold 0.92. Preset lora-fast. Base Qwen/Qwen2.5-3B-Instruct. Verifier kind="regex" with must_not_contain SSN / email / phone post-redaction.

  1. Minute 0–5: kolm init --pack hipaa-phi-redactor. Scaffolds kolm.yaml, the synthetic seed corpus, the verifier, and the audit-log shape.
  2. Minute 5–30: drop your own labeled examples into examples/ (or skip and use seeds). 18 HIPAA Safe Harbor categories pre-mapped.
  3. Minute 30–90: kolm compile. Pattern path runs in seconds; LoRA path takes ~45 min on a single A10. K-gate at 0.92.
  4. Minute 90–110: kolm sign --credential. Emits the kolm-credential/0.1 provenance credential and signs the artifact.
  5. Minute 110–120: kolm inspect dist/phi-redactor.kolm --verify on. Receipt body verifies, CID round-trips, credential signature checks. Hand to security review.
What the pack does NOT do: it is not Expert Determination certification. The pack ships the Safe Harbor identifier path. Expert Determination requires a qualified statistician’s sign-off. see evidence.md for scope language you can paste into your DPA.
