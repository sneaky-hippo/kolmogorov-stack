docs

Build against the current product.

Everything you need to compile a task into a portable, signed file you can run anywhere. These docs describe what the repo ships today: the Node CLI, the hosted compile API, signed recipe-mode .kolm artifacts, HMAC receipt metadata, and local MCP serving from ~/.kolm/artifacts. If a flag, endpoint, or field is not listed here, treat it as out of scope.

04API reference

Hosted API surface.

The CLI uses bearer auth. Browser sessions can use the httpOnly kolm_session cookie set by sign-in routes.

POST /v1/signupbody { email, name? }; returns tenant metadata and a one-time API key
POST /v1/signinbody { api_key }; validates key and sets kolm_session
POST /v1/session/loginsame API-key session flow for app clients
GET /v1/accountreturns tenant id, name, plan, quota, usage, and remaining compile budget
POST /v1/account/rotate-keyreturns a new API key; new keys are stored as hashes at rest
POST /v1/compilebody { task, examples?, corpus_namespace?, base_model? }; returns a compile job id
GET /v1/compile/:idreturns status, progress, K-score, and artifact download URL after completion
GET /v1/compile/:id/.kolmdownloads the compiled artifact for the authenticated tenant
GET /v1/registry/exportexports public recipe metadata and source for current registry entries
GET /sdk-current.jsoncurrent browser SDK SHA, SRI hash, URL, and byte size

Compile a task.

$ curl -sS -X POST https://kolm.ai/v1/compile \
       -H "Authorization: Bearer ks_live_..." \

05artifact format

Open schemas for the artifact contract.

The current .kolm file is a zip artifact with manifest, model pointer metadata, recipe data, placeholder LoRA/index members where appropriate, eval data, and HMAC receipt metadata. Native LoRA training, populated vector indexes, and external receipt anchoring remain product-hardening work, not current defaults.

06receipts and verification

Verify any artifact, end to end.

Every .kolm file ships with a receipt.json that hashes each archive member with SHA-256 and signs the canonical chain with HMAC-SHA256. kolm inspect walks the chain locally; nothing leaves your machine.

# receipt.json (excerpt)
{
  "spec": "receipt-v0.1",
  "alg": "HMAC-SHA256",
  "signed_at": "2026-05-08T18:42:14Z",
  "signer": "kolm-cloud@kolm.ai",
  "chain": [
    { "member": "manifest.json",  "sha256": "8b2e…01af", "size_bytes": 1822 },
    { "member": "recipes.json",   "sha256": "61c4…dd09", "size_bytes": 4120 },
    { "member": "evals.jsonl",    "sha256": "9a7f…42b1", "size_bytes": 9211 },
    { "member": "model.ptr",      "sha256": "21d8…77ee", "size_bytes": 96   },
    { "member": "lora.bin",       "sha256": "00…00",     "optional": true   },
    { "member": "index.sqlite-vec","sha256": "00…00",    "optional": true   }
  ],
  "signature": "f0a3…b4c8"
}
verify locallykolm inspect ./artifact.kolm recomputes every member SHA-256 and the HMAC over the chain. Fails closed if anything drifts.
re-fetch the registryrecipe_registry.url in the manifest is the exact endpoint to re-pull the registry snapshot the artifact was compiled against.
optional memberslora.bin and index.sqlite-vec are listed with optional: true in the chain. Their absence does not break the receipt; their presence is hashed.
offline-safeVerification needs only the artifact bytes and the receipt signing key. No network call is required to confirm integrity.
07errors

What the API actually returns.

Errors share one shape: { "error": "<machine_code>", "detail"?: "<human prose>" }. detail is omitted in production unless KOLM_DEBUG=1.

400 invalid_inputrequest body failed validation. detail names the offending field when debug is on.
401 missing_authno Authorization: Bearer ks_... header and no kolm_session cookie.
401 invalid_keykey shape is right but does not match a tenant.
403 quota_exhaustedtenant compile budget is spent for the period. Check GET /v1/account.
404 not_foundused for unknown jobs and unknown artifacts; never leaks tenant existence.
409 job_conflicta compile with the same idempotency key is already running.
429 rate_limitedsoft per-IP and per-tenant limits. Backoff hint in Retry-After.
500 server_errorunhandled path on the cloud. detail is omitted in prod by design.
503 not_configureda required upstream secret is missing. Affects /v1/compile if the cloud-side model key is unset.
08faq

Plain answers.

Why a single file?A .kolm is a deterministic ZIP. One file you can email, archive, sign for, or pin in CI. The compiler is the cloud; the artifact is yours.
Does my data leave my machine?During compile, your examples and corpus are sent to the cloud to drive distillation. After compile, the artifact runs locally with no network call. kolm benchmark blocks runtime network calls by default.
What if my API key leaks?Rotate it: POST /v1/account/rotate-key. Old key invalidates immediately. Keys are stored as hashes at rest.
Can I run an artifact offline?Yes. kolm run, kolm inspect, kolm eval, and kolm serve --mcp all work with no network. Recipe drafting and the K-score are computed from the artifact, not pulled from the cloud.
What is "recipe mode"?Today's compile path produces an artifact whose runtime is the recipe pack (deterministic-token drafts) plus a base-model pointer. Native LoRA training and populated vector indexes are product-hardening work, not the current default.
Is there a stable on-disk format?Yes. The artifact contract is RS-1 and the manifest/receipt schemas are versioned independently. Breaking changes mint rs-2; rs-1 readers will continue to work against existing artifacts.
How do I serve a compiled artifact to Claude Desktop or Cursor?kolm serve --mcp --http --port 11455. The endpoint speaks Model Context Protocol over HTTP; point your MCP client at http://localhost:11455.
What is the K-score?A composite of accuracy, coverage, size, and p50 latency. The artifact only ships if every gate passes. See /k-score for the gate definitions.
