kolm  /  compare  /  kolm vs AWS Bedrock model distillation

Bedrock distillation makes a Bedrock-only model. kolm makes a file.

AWS Bedrock model distillation lets you take a powerful teacher model in the Bedrock catalog (Claude, Llama, Nova, Titan) and distill its behavior into a smaller, cheaper Bedrock-served student. The result is a Custom Model that runs inside Bedrock. kolm distills against the same teacher families and emits a portable signed compiled artifact that runs anywhere. The architectural commitment is different and the AWS lock-in deserves to be named.

Wave 274 · reviewed 2026-05-18 · AWS-resident custom model vs portable signed file

What Bedrock model distillation ships in 2026.

  • Distillation job in the AWS console: pick a Bedrock teacher (Claude, Nova, Llama, Titan), pick a Bedrock-supported student family, upload training prompts (or use latency-optimized inference logs as the synthetic dataset).
  • Custom Model on Bedrock. The distilled student becomes a Bedrock Custom Model with a provisioned-throughput SKU. You call it via the Bedrock runtime API.
  • S3-native data pipeline. Training data, validation data, and outputs flow through S3 buckets in your AWS account.
  • IAM + KMS + CloudTrail integration. Access controls, encryption keys, and audit logs match the AWS posture your security team already approved.
  • Provisioned Throughput billing for inference. You reserve compute units per hour and call the model through the Bedrock runtime.

What Bedrock distillation does not ship: a file you can download and run outside AWS, a portable signed manifest with a third-party-verifiable receipt chain, a compile-time eval gate that blocks the build, or a deployment shape that runs air-gapped or on edge hardware.

The AWS lock-in note.

Lock-in caveat
The output of a Bedrock distillation job is a Bedrock Custom Model. The weights are not exportable. The model is callable only through the Bedrock runtime API. The CloudTrail and S3 audit trail lives in your AWS account; it does not move with the model if you ever migrate off AWS. If your buyer requires a cryptographically verifiable, vendor-portable artifact, Bedrock distillation alone does not deliver it.

This is not a criticism of Bedrock. It is the deliberate shape of a hyperscaler integration: deep integration with IAM, KMS, CloudTrail, S3, and Provisioned Throughput is the entire value proposition, and that integration is what makes the output non-portable. Kolm makes the opposite trade: the .kolm file is portable, signed, and verifiable, and you bring your own runtime infrastructure.

The cleanest combined deployment is to use kolm on the build side (signed artifact, receipt chain, eval gate, PHI redactor) and import the LoRA weights into Bedrock via Custom Model Import if your serving target is Bedrock. The signed .kolm file is the cryptographic contract that travels with the artifact; the Bedrock import handles serve-side AWS integration.

kolm-built adapter imported into Bedrock.

# 1. Build the signed adapter with kolm (teacher can be Bedrock Claude)
$ kolm distill claims-router \
    --teacher=aws-bedrock:anthropic.claude-opus-4-7 \
    --student-base=llama-3.1-8b \
    --redact=phi --eval-gate=0.85

  PASS  K-score 0.911 (A=0.93 T=0.90 F=0.95 R=0.88)
  PASS  PHI redactor: 18 Safe Harbor + 3 kolm extensions, no leakage
  PASS  Ed25519 signature, Rekor transparency log entry
  WROTE claims-router.kolm

# 2. Extract the LoRA-merged weights for Bedrock import
$ kolm extract claims-router.kolm --target=merged-weights --out=s3://my-bucket/claims-router/

# 3. Import as a Bedrock Custom Model
$ aws bedrock create-model-import-job \
    --job-name claims-router-import \
    --model-name claims-router \
    --imported-model-name claims-router \
    --model-data-source s3DataSource={s3Uri=s3://my-bucket/claims-router/}

# 4. Verify the served model still matches what kolm signed
$ kolm verify claims-router.kolm
  PASS  24/24 checks · Ed25519 + Sigstore validated · receipt chain intact

When AWS Bedrock distillation is the right answer.

Your stack is entirely on AWS already

Procurement signed the EDP; data lives in S3; IAM, KMS, and CloudTrail are the security primitives your buyer accepts. Staying inside Bedrock is the path of least resistance and the audit story is already built.

AWS-native compliance posture is enough

Your auditor accepts FedRAMP High, SOC 2, HIPAA-via-BAA, and CloudTrail as sufficient evidence. You do not need a cryptographically signed build artifact external to AWS.

Bedrock teacher catalog matches your need

Your distillation teacher is already in Bedrock (Claude, Nova, Llama, Titan) and your student family is on the Bedrock-approved list. No cross-vendor distillation is required.

Provisioned Throughput is the serving model you want

You want hourly committed capacity for a custom model with no separate serving infrastructure. The Bedrock runtime API and IAM-scoped access fits the buyer's procurement language.

When kolm is the right answer.

You need portable weights you can mail to a phone, run air-gapped, or serve on hardware AWS does not own. Your auditor specifically asked for a third-party-verifiable receipt chain bound to the build artifact, not just CloudTrail logs. You need cross-vendor distillation (Anthropic teacher with a Qwen student, or OpenAI teacher with a Llama student) outside the Bedrock catalog. You want flat compile cost rather than Provisioned Throughput forever. The .kolm file is the contract; you can still import it into Bedrock when the serving target is AWS.
