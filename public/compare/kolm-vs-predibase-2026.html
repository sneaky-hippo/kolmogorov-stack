Predibase governs the runtime. kolm proves the build.

Rubrik (NYSE: RBRK) acquired Predibase in June 2025 and the combined product pivoted hard toward "agentic AI governance" - monitoring agent actions, rewinding bad state, applying Rubrik's enterprise data-security posture to LLM workloads. That is a real category, and it sits adjacent to kolm. kolm lives on the build side and emits a portable, signed compiled artifact. The two work together; they do not overlap.

What Predibase + Rubrik ships in 2026.

  • Hosted LoRAX serving on Predibase cloud or a customer VPC under Enterprise contract. Turbo LoRA inference path, multi-LoRA batching, thousands of adapters per GPU.
  • Hosted fine-tuning on managed serverless infrastructure with per-second GPU billing. Developer and Enterprise SKUs.
  • Agentic AI governance (Rubrik 2026 push). Agent action monitoring, time-travel rewind of agent state, Rubrik data-security posture applied to LLM workloads, integration with the Rubrik Security Cloud control plane.
  • Rubrik enterprise sales motion. NYSE-listed parent, Series-of-record procurement story, MSAs in days, SOC 2 Type 2 + HIPAA-eligible.

What Predibase + Rubrik does not ship: a portable signed compiled file you mail to a phone, an air-gapped deployment that does not call back to Predibase or Rubrik, a third-party-verifiable receipt chain proving what teacher distilled the model and what data the train set saw, or a cryptographically signed eval gate bound into the build artifact.

The Rubrik acquisition validated the category but did not close the gap.

June 2025: Rubrik acquired Predibase. The combined product pivoted toward agentic AI governance - monitoring agent actions, rewinding state, applying Rubrik's data-security posture to LLM workloads. Rubrik gained a fine-tuning + serving footprint; Predibase gained an enterprise security distribution motion.

This proved the market: enterprise security plus LoRA serving is a real spend category. It did not close the gap that motivates kolm. Rubrik + Predibase governs the runtime. kolm proves the build. An agent action monitor cannot tell you what teacher distilled the model, what data the train set held, whether the eval gate fired, whether the bytes on disk match the bytes signed at compile time, or that no raw PHI leaked into adapter weights. Those are kolm's surface.

The two products work together. Build a signed adapter with kolm. Deploy it through Predibase. Let Rubrik govern the runtime. The signed .kolm file is the cryptographic contract between the build team and the runtime governance plane.

The integration in one CLI session.

# 1. Build the signed adapter with kolm
$ kolm distill claim-summarizer \
    --teacher=anthropic:claude-opus-4-7 \
    --student-base=qwen2.5-7b \
    --redact=phi --eval-gate=0.85

  PASS  K-score 0.917 (A=0.94 T=0.91 F=0.97 R=0.88)
  PASS  PHI redactor: 18 Safe Harbor + 3 kolm extensions, no leakage
  PASS  Ed25519 signature, Rekor transparency log entry
  WROTE claim-summarizer.kolm

# 2. Extract the LoRA for Predibase deploy
$ kolm extract claim-summarizer.kolm --target=lora-weights --out=./adapter

# 3. Push to Predibase the way you already do
$ pb adapter upload ./adapter --base=qwen2.5-7b --name=claim-summarizer

# 4. Verify the served adapter still matches what kolm signed
$ kolm verify claim-summarizer.kolm
  PASS  24/24 checks · Ed25519 + Sigstore validated · receipt chain intact

When Predibase + Rubrik is the right answer.

Multi-tenant LoRA serving at scale

You serve thousands of distinct adapters across many customers. LoRAX + Turbo LoRA on Predibase cloud is genuinely strong on this workload, and someone else owning GPU operations is the point.

Agent governance is your primary spend

Your buyer needs action monitoring, rewind, and a Rubrik security narrative. That is what the post-acquisition product was built for and what the sales motion sells against.

Single enterprise MSA matters

Procurement wants one MSA covering fine-tuning, serving, and runtime governance from a NYSE-listed vendor. Rubrik delivers that posture cleanly.

Runtime-only compliance answer

Your auditor accepts a console audit log plus runtime governance reports as the answer. You do not need a cryptographically signed build artifact.

When kolm is the right answer.

Your team can consume a signed file and run it themselves. Your workload is edge, air-gapped, or on-device. Your auditor needs cryptographic provenance on the build, not just runtime telemetry. Your compliance team specifically asked for the receipt chain. The PHI redactor (W157), cross-vendor distillation (W158), and supersession chain (W167) become load-bearing the moment your buyer asks "what trained this model and how do I prove it did not see raw PHI."
