Edge · one file · ARM/x86/RISC-V

One artifact for every edge box.

Edge AI usually means three runtimes (CoreML, ExecuTorch, TFLite), three quantization passes, and a six-month integration before the first device ships. kolm collapses the stack: compile once on your gold examples, ship the same signed file to ARM, x86, or RISC-V hardware. Runtime is offline, deterministic, and verifiable. The .kolm doesn't care whether you're on a NVIDIA Jetson, a Raspberry Pi, or a fleet of Linux x86 kiosks.

02compile

Compile for a target class.

$ kolm compile fault-triage \
    --base qwen2.5-3b-instruct \
    --quantize int4 \
    --target-ram 4G \
    --recipe-pack-depth 64 \
    --examples ./plant-logs.jsonl

04provisioning

Fits your existing OTA pipeline.

A .kolm is a regular zip with a sha256. Push it through whatever you already use (Mender, Balena, Azure IoT Hub, AWS IoT, your own apt repo). The runtime caches by hash and verifies the signature on every cold load.

05guarantees

What we say. What we don't.

same artifact across architecturesYes. The compiler emits target-class metadata; runtime picks the right path. No retargeting required.
deterministic at run timeYes for recipe-mode tasks. Adapter-mode tasks include a deterministic seed in the manifest; same input + same artifact = same output.
tiny-target SKUs (<1GB RAM)Recipe-only mode on the Pro tier. Default compile targets 2GB RAM minimum; sub-1GB SKUs use the recipe-only flag and skip the adapter layer.
real-time guaranteesNo. p50 latency is reported; p99 / hard real-time is your integrator's responsibility, not the artifact's.
06why edge AI is different

Three problems cloud inference cannot solve at the edge.

A disconnected device is not a connected device with worse latency. It is a different deployment shape, with constraints that the typical cloud-first AI stack cannot satisfy.

The link dropsManufacturing floors lose backhaul. Maritime vessels are out of range for days. Remote oil and gas sites run on intermittent satellite. A retail kiosk in a basement loses WiFi. Any AI feature that depends on a cloud round-trip becomes a P0 outage the moment the link drops. A compiled artifact has no link to lose.
Fleet heterogeneityA single deployment fleet often spans Jetson Orin boxes on robotics, Apple Silicon in field tablets, x86 industrial PCs on the line, and Raspberry Pi class hardware in low-cost sensors. Maintaining four runtimes with four quantization passes and four release pipelines is the cost most teams underestimate. One .kolm across all targets reduces that surface.
Power and thermal budgetA fanless industrial PC at 12W TDP cannot host an 8B-parameter model running at full precision. A robot in continuous duty cannot tolerate thermal throttling that drops inference rate mid-cycle. The compile target chooses the quantization and recipe-pack depth to fit the budget at compile time, not at runtime.
07how kolm helps

Three properties that come with the artifact.

Each is a property of the compile output, not a flag your field-service team configures.

Zero network at run timeThe artifact loads from local storage and writes to local storage. No telemetry beacon, no model-pull, no remote config. The device's network interface can be down or absent at run time and the .kolm still produces output. Receipts mirror upstream when connectivity returns, on whatever cadence the fleet operator chooses.
Backend-aware executionThe runtime picks the right compute path on the host. NVIDIA Jetson uses the local-cuda backend; Apple Silicon dev boxes use local-mps; industrial x86 falls back to local-cpu. The artifact carries the metadata that lets the runtime pick without rebuilding. Provenance (backend, device, latency) lands in the receipt for the fleet operator to audit.
Signed by hash, pinned at deploymentThe .kolm sha256 is part of the OTA manifest. A field device only accepts artifacts whose anchor matches what the fleet operator approved. There is no path to silently swap behavior on devices in production. Rollback is a one-line command against the local cache.
08a factory walkthrough

Vibration triage on the plant floor, end to end.

Concrete example. A manufacturing operator wants to flag bearing-fault patterns in vibration sensor data. The line runs on industrial x86 PCs at each station. Backhaul to corporate is intermittent. Here is the deployment.

01 gatherReliability team exports 60 days of vibration-sensor traces from the historian, paired with the maintenance log entries that followed each event. Output is a jsonl file of 8,400 input-output examples (sensor window, recommended next action). The file lives on the corporate compile box; the plant floor is not in the loop yet.
02 compileEngineer runs kolm compile fault-triage --base qwen2.5-3b-instruct --quantize int4 --target-ram 4G --examples ./plant-logs.jsonl. Compile picks the local-cuda backend on the corporate compile box. K-score lands at 0.79, above the ship floor. Output is a 1.84 GB signed .kolm with an HMAC chain anchored to the corporate fleet registry.
03 distributeOTA pipeline pushes the artifact to the plant. Existing Mender or Balena flow handles the transport; the runtime cares only about sha256 match. Devices verify the signature on cold load. A failed verify halts load and pages the fleet operator. Delta updates ship as a small recipe-pack patch on subsequent policy changes.
04 runEach station's industrial PC pulls vibration windows from the local sensor stream and runs kolm run fault-triage.kolm. p50 latency is 3.8ms on the Intel N100 SKU. Output is the recommended action plus a receipt. When the link to corporate is up, receipts mirror to the fleet's S3 bucket on a 5-minute cadence. When the link is down, receipts queue locally and ship later. No window is dropped.
09common questions

What fleet operators ask first.

Direct answers. Edge teams have heard a lot of vendor promises that did not survive contact with the plant.

Will the same .kolm really run on Jetson and on Raspberry Pi?Yes for compatible parameter sizes. The compile target sets a minimum RAM and quantization. A 3B int4 artifact targeted at 4GB RAM runs on Jetson Orin Nano (local-cuda), on Apple Silicon dev boxes (local-mps), on Intel N100 industrial PCs (local-cpu), and on Raspberry Pi 5 at lower throughput. The same sha256 file. Below 1GB RAM the recipe-only mode is required and the adapter layer is skipped at compile time.
Can the device run completely offline forever?Yes. The runtime never initiates an outbound connection. Receipts are local files. If the fleet operator wants telemetry, they sync the receipt directory upstream when the network is available. If they want no telemetry, they delete the receipt directory on a schedule. The artifact does not know or care.
What happens when a device's storage fills up with receipts?The runtime emits receipts to a configured directory. Retention is the fleet operator's policy. Most deployments rotate the local cache on a size or age policy and mirror upstream first. The runtime does not delete on its own.
How do we update behavior without re-pushing 2GB?Recipe-pack updates ship as deltas against the base model pointer. A policy change (new fault classes, new sensor schemas) is typically a few megabytes. The full .kolm only re-ships when the base model itself changes.
What about real-time deadlines?We report p50 latency. Hard real-time guarantees are the integrator's responsibility. The runtime does not preempt the host OS scheduler, does not own interrupt priority, and does not claim deterministic worst-case execution time. If the deployment needs guaranteed deadlines, the integrator wraps the artifact in their existing RTOS pattern.
Does the artifact phone home for licensing or telemetry?No. The artifact is a regular zip file with a sha256. The runtime is MIT-licensed. There is no per-device licensing check at run time, no telemetry beacon, no kill switch. The fleet operator's existing OTA system is the only update channel.
