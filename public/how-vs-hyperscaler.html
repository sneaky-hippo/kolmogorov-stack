kolm  /  compare  /  how kolm relates to hyperscaler fine-tune

The model you OWN as a file vs a model that lives in a vendor API.

OpenAI fine-tuning, Anthropic custom models, Google Vertex AI tuning, AWS Bedrock customization — you upload data, you get back a model that exists only inside the vendor's API. The tuned weights stay in their infrastructure. Air-gap, residency, and exit cost are not features they don't ship; they're structural impossibilities for closed-weight fine-tuning. kolm produces a file.

Wave 169 · reviewed 2026-05-17 · structural, not feature-gap

What hyperscaler fine-tuning ships.

  • OpenAI fine-tuning — supervised fine-tuning + DPO + RFT on GPT-4o-mini, GPT-4.1, o-series base models. Tuned model accessible via the standard chat-completions API with a custom model ID.
  • Anthropic custom models — managed customization on Claude models, accessed via standard Messages API with a custom model identifier (Enterprise tier).
  • Google Vertex AI tuning — supervised fine-tuning on Gemini bases; tuned model deployed as a Vertex AI endpoint.
  • AWS Bedrock customization — supervised fine-tuning on Bedrock-hosted models (Claude, Llama, Titan, Cohere variants); tuned model accessed via Bedrock invoke-model.
  • Ease-of-start — managed GPU + managed dataset upload + managed eval UX. Zero ML infrastructure to operate.

The structural impossibilities.

This is not a feature gap. No amount of vendor product velocity closes these. Closed-weight fine-tuning, by construction, cannot produce a portable signed file you ship to a phone, an air-gapped VPC, or an offshore subsidiary. The weights never leave the vendor's infrastructure. That's the trade you make when you choose the managed path.

The cross-vendor distillation pattern uses hyperscalers as teachers.

This is the Wave 158 capability that the user named explicitly: "How do Chinese models distill American models." The pattern is to use a hyperscaler API (OpenAI, Anthropic, Google, xAI) as the teacher, capture the prompt/completion pairs into seeds.jsonl, then LoRA-fine-tune a small open-weight base. The fine-tuned student approximates the teacher on the task distribution defined by the prompts.

kolm makes this a first-class CLI flag pair:

Licensing note: OpenAI / Anthropic / Google terms restrict using API outputs to develop "competing" general-purpose models. A narrow-task kolm artifact (phone normalizer, denial-code classifier) is not a competing general model; the legal status of the narrow-task carve-out is unsettled. The always-safe option is --teacher=local:<open-weight-model>, run on your own GPU. See RS-1 §7.7 for licensing posture.

When hyperscaler fine-tune is the right answer.

Your only deploy target is a hosted API. You don't need a portable artifact. Your data isn't regulated, or your regulator accepts the vendor's BAA + audit trail. Procurement wants a Series-mega vendor with enterprise contracts. Engineering wants the shortest path from "upload data" to "tuned model." Use the hyperscaler tuning UX.

When kolm is the right answer.

Air-gap deploy is a requirement, not a nice-to-have. Residency policy says the model can't sit in the vendor's cloud. Your auditor needs cryptographic provenance on the build. You want to use a hyperscaler as the teacher but ship a model you can run on a phone. The .kolm file structurally solves what closed-weight tuning structurally can't.
