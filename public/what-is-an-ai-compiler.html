What is an AI compiler?

An AI compiler is the missing build step between a frontier model API and a local artifact that runs on your hardware. It takes a task specification, a corpus of captured prompts and responses, and a teacher model, and emits a single signed file you can ship anywhere. Compile your first one in five minutes.

Definition

An AI compiler is a tool that turns three inputs (a task specification, a corpus of captured prompts and responses, and a teacher model) into one output: a single portable artifact containing compiled model weights, the prompt-response evaluator, the receipt chain that proves how it was built, and the recipe needed to rebuild it byte-for-byte.

kolm.ai is an AI compiler. The artifact format is .kolm. The CLI verb is kolm compile. The fact that the artifact is signed and self-contained is what makes the category different from anything that came before it.

A traditional compiler turns source code into a binary. An AI compiler turns a task and a corpus into a model. In both cases the build step is the abstraction that lets the rest of the stack treat the output as a normal file.

The three-step compile flow

Every kolm compile run follows the same three steps. The first time you run it is the only time you have to think about the substrate; afterward it disappears the way gcc does.

  1. Capture. Run your application against any frontier API behind kolm capture (the proxy). Every prompt and response is stored in a per-namespace corpus with a durability receipt header (x-kolm-capture-durable) so you can prove the row landed.
  2. Distill. Once a namespace crosses 100 captured pairs you can promote them into a recipe. The recipe specifies the task, the input/output shape, the holdout split, and the eval. Distillation runs against your captured pairs as the training set and against a frozen-eval holdout as the gate. The teacher is whichever frontier model you captured against; the student is whatever base you compile to (see the frontier-model catalog).
  3. Compile. The compiler emits one file: your-task.kolm. The file contains the weights, the eval, the receipt chain, the spec, and the seeds. The compiler also emits a verifier so anyone can re-run the build and confirm the byte-for-byte match.

Three commands. kolm capture, kolm distill, kolm compile. The quickstart walks the full loop in five minutes.

What a .kolm artifact contains

A .kolm is a single signed archive with a strict internal layout. The signature binds every section to the build that produced it; if any byte changes, the signature breaks and kolm verify refuses to load it.

  • Weights. The compiled student model in the format the runtime expects (typically GGUF or ONNX, depending on tier).
  • Spec. The task specification the recipe was compiled against. Includes the input schema, the output schema, the latency budget, the safety policy, and the eval threshold.
  • Seeds. The seed set used to bootstrap distillation (when the captured corpus needed augmentation).
  • Eval. The frozen-eval set the artifact passed. Anyone with the .kolm can re-run the eval and see the same score.
  • Receipt chain. A Merkle chain over the spec, seeds, captured corpus hash, teacher model id, holdout hash, build timestamp, and signing key. Walking the chain backward produces a complete provenance trail to the genesis commit.
  • Recipe. The exact kolm compile invocation needed to rebuild this artifact byte-for-byte. Reproducible builds are the default, not a feature.

See the .kolm format spec for the exact field layout and RS-1, the receipt-chain specification, for the Merkle structure.

AI compiler vs fine-tuning service

A fine-tuning service uploads your data to a hosted training environment, runs gradient updates against a base checkpoint, and returns an API endpoint. The weights stay on the provider's GPUs; the model name is the new artifact; the inference path is a hosted call.

An AI compiler runs the equivalent build on infrastructure you control and gives you the weights as a file. You can move that file to a 3090, a DGX Spark, a Mac M3 Ultra, an air-gapped server, a customer's hardware, or every device in a Tailscale mesh. The provider has nothing to revoke.

Fine-tuning service

Service. Endpoint. Vendor owns weights. Re-bills per token.

AI compiler

Build step. File. You own weights. Cost is the watts.

See "Fine-tuning vs AI compilation: when each is the right tool" for the full decision tree.

AI compiler vs hosted endpoint

A hosted endpoint (OpenAI, Anthropic, Google, Cohere) is a remote service. Every request leaves your network, costs per token, requires connectivity, and the provider can change the model under you without warning. Hosted endpoints are the right answer when you are still discovering the task; they are the wrong answer once the task is stable enough to compile.

The honest framing: hosted endpoints are the teacher in the compile flow. You use them, capture their outputs, distill the captures into a recipe, and compile the recipe into a .kolm. The hosted call cost goes to zero as the captured corpus stabilizes and the local artifact takes over. /captures is the live dashboard where you watch this happen.

AI compiler vs LoRA serving infra

Tools like LoRAX, vLLM with LoRA, and Predibase serve LoRA adapters at multi-tenant scale. They are complementary to an AI compiler, not competitive: kolm builds the adapter (or the full-rank fine-tune, or the merged-weights compilation), and a serving layer like LoRAX hosts it. The honest production stack is "kolm compiles + LoRAX serves" if you need multi-tenant adapter serving; the artifact is portable enough to also run on a single device behind kolm run.

See /how-vs-lorax for the integration pattern and /compare for the canonical 10-axis matrix.

Receipts, durability, frozen-eval, drift

Four guarantees fall out of the artifact being signed and self-contained. None of them are optional; the verifier rejects any artifact missing them.

  • Receipt chain. Every captured pair, every distillation step, every eval result, every signature is hashed into a Merkle chain. kolm verify walks the chain backward to genesis. A regulated buyer can audit the build without trusting the builder.
  • Durability receipt. The capture proxy returns x-kolm-capture-durable: true only after the row has actually landed in the store. If the header is missing, the row did not persist and the client must retry. No silent swallows.
  • Frozen eval. The eval set is frozen at recipe-spec time. The artifact cannot ship if the score falls below the threshold on that frozen set. The set is hashed into the receipt chain so a future auditor can verify the gate fired on the same questions.
  • Drift + supersession. Artifacts do not stay correct forever. kolm drift detect compares a live artifact's eval score against the frozen baseline on a schedule; a breach emits a supersession entry naming the successor. The lifecycle is documented at /drift.

Why now, why a category

Three trends converged in 2025 and 2026 that made the AI compiler a useful unit of abstraction.

  1. Open-weight frontier models caught up. Step-3.5 Flash REAP-121B at Q6 fits in a DGX Spark. DeepSeek v4 PRO 1.6T at Q2 runs on a Mac M3 Ultra 512GB. Qwen 3.6 27B Dense at Q4 fits on a single RTX 3090. The base model is no longer the bottleneck. See the catalog.
  2. Capture-first beat prompt-first. Teams discovered that captured production traffic is a better fine-tuning corpus than handwritten examples. The proxy pattern (kolm capture) is now the obvious way to gather training data from real use.
  3. Regulation arrived. EU AI Act, HIPAA, SOC 2 Type II, and FedRAMP all require a provenance answer that hosted endpoints structurally cannot provide. A signed artifact with a receipt chain is the only honest answer.

The category did not exist in 2024 because none of those three were true at the same time. In 2026 all three are. The name "AI compiler" claims the build-step abstraction that the trio enables.

Your first compile

The shortest path from zero to a .kolm:

curl -fsSL https://kolm.ai/install.sh | sh
kolm init my-task
kolm capture --teacher=openai --to=my-task
# (run your application; let captures accumulate)
kolm distill --from-captures my-task --min-pairs 100
kolm compile my-task --tier=3090
kolm verify my-task.kolm
kolm run my-task.kolm "the prompt you would have sent to the teacher"

Seven commands. The first two are setup. The middle three are the capture, distill, compile loop. The last two prove the artifact is correct and run it locally. The quickstart has a video.

Honest scope. "Five minutes" is wall-clock time on the seven commands once the capture step has run. The capture step needs real traffic; it takes as long as it takes to accumulate a useful corpus. We recommend 100 pairs as the floor for a viable distill and 1,000 for a confident one. /captures shows you the live counter.

Common questions

Is an AI compiler the same as model distillation?

Distillation is one of the steps inside the compile flow, not the whole flow. An AI compiler also handles capture, eval freezing, signing, receipt-chain construction, drift detection, supersession, and verification. A standalone distillation library (DistilBERT-era) does only the gradient updates.

Do I need GPUs to use an AI compiler?

For inference: no, a 3090 or M3 Mac is enough for most artifacts; a CPU build is enough for small classifiers. For compilation: yes for the distillation step, which is the only GPU-heavy phase. See /runtimes for the source-build path against your exact hardware target.

What runs the .kolm artifact?

The kolm runtime, which is a thin wrapper over llama.cpp or the ONNX runtime depending on the model architecture. The runtime is small (single binary, no Python) and runs offline. See /runtimes.

Can I use kolm without sharing my data with kolm.ai?

Yes. The default install is air-gapped. kolm capture runs as a local proxy; the captured corpus stays on the file system you point it at. The only thing that leaves your network is the inference call to the teacher model (which goes to OpenAI or Anthropic or whoever you chose), and even that can be replaced with a local teacher via --teacher=local:llama-3-70b.
